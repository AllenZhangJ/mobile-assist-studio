# iOS Assist Studio V4.0 Open Source Integration Plan

## 0. TL;DR

- V4.0 必须站在成熟项目肩膀上，避免重复造轮子。
- 吸纳策略不是统一复制源码，而是按依赖、适配、移植、复制和参考五档处理。
- Airtest、Pyxelator、Appium Inspector 和 appium-mcp 是 V4.0 核心吸纳对象。
- AppiumAir 只作为外围历史参考，不进入 V4.0 核心融合对象。
- 所有复制或 vendored 代码必须进入 `third_party/`，并登记到 `THIRD_PARTY_NOTICES.md`。
- 所有许可证必须以对应仓库当前 LICENSE 文件为准，落地前复核。

## 1. Integration Philosophy

V4.0 的融合原则：

```text
依赖优先 > 适配优先 > 移植优先 > 小模块复制 > 产品参考 > 自研
```

含义：

- 能用包管理器稳定依赖，就不要复制源码。
- 能通过 adapter 隔离，就不要把第三方模型散落到 Runtime。
- 能移植协议和数据结构，就不要拉入不合适的运行时。
- 只有小而稳定、许可证清晰、维护成本可控的模块才允许复制。
- 大型 UI 或 IDE 项目优先学习产品能力，不复制界面代码。

## 2. Integration Matrix

| 项目 | 主要价值 | 吸纳方式 | 进入模块 | 复制策略 |
|---|---|---|---|---|
| Airtest | 图像识别、断言、create/run/report、报告 | Python Sidecar 依赖 | Vision / Recorder / Evidence | 不整体复制 |
| Pyxelator | 截图模板定位、置信度、按图点击 API | Python 依赖或小模块复制 | Vision / TargetResolver | 可复制核心小模块 |
| Appium Inspector | Session、截图、Source、元素树、命令面板 | 体验和协议移植 | Inspector / Driver | 不复制 Electron UI |
| appium-mcp | MCP 工具、AI 调用 Appium、视觉恢复 | 协议和工具模型移植 | AI / MCP | 不复制 Node 运行时 |

## 3. Airtest Integration

### 3.1 What To Absorb

- 图像识别能力。
- 模板化视觉断言。
- `create -> run -> report` 工作流思路。
- 本地报告结构。
- 截图和失败现场组织方式。
- AirtestIDE 的录制和报告心智。

### 3.2 How To Integrate

Airtest 作为 Python Sidecar provider：

```text
Dart Runtime
  -> Vision Core
  -> AirtestProvider
  -> Python Sidecar
  -> Airtest library
```

AirtestProvider 只做：

- 接收截图和模板。
- 返回匹配结果。
- 返回断言结果。
- 返回证据摘要。

AirtestProvider 不做：

- 直接连接设备。
- 直接点击。
- 直接运行 `.air` 作为项目真源。
- 替代 Project DSL。

### 3.3 Acceptance

- 无 Airtest 环境时，坐标和 Appium 基础流程仍可用。
- Airtest provider 可通过 fixture screenshot 测试。
- Airtest 失败必须返回结构化诊断。
- Airtest 报告思路只能写入本地 Evidence，不上传云端。

### 3.4 Current Runtime Slice

- 当前已通过 Python Sidecar 接入 Airtest 后端选择，优先尝试 `airtest.aircv.find_template`。
- 本机未安装 Airtest、API 不匹配或识别失败时，provider 只返回结构化不可用或未命中结果，由 TargetResolver 继续调度其它 provider。
- 当前 Airtest provider 只做视觉目标解析和断言表达，不运行 `.air` 文件、不连接设备、不下发点击。
- Airtest 报告心智已先沉淀为 Runtime 本地报告模型，并已接入 Monitor / Execute 共用详情抽屉的本地报告面板、脱敏 JSON 复制和 Runtime 文件保存；报告文件写入当前运行目录内的 `exports/`，随 evidence 保留策略一起生命周期管理，不成为新的 workflow 真源。

## 4. Pyxelator Integration

### 4.1 What To Absorb

- 以截图模板定位元素。
- confidence 阈值。
- locate / find 思路。
- retry / debug 思路。
- Appium 场景下的视觉定位经验。

### 4.2 How To Integrate

首选方式：作为 Python 依赖。

备选方式：复制核心模板匹配小模块到 `third_party/pyxelator_core/`，并保留来源、许可证和修改记录。

PyxelatorProvider 输出：

- found。
- center coordinate。
- region。
- confidence。
- debug artifact。
- user reason。

PyxelatorProvider 不输出：

- 设备点击命令。
- workflow 修改。
- 未脱敏路径。

### 4.3 Copy Criteria

只有满足全部条件，才允许复制 Pyxelator 局部代码：

- 许可证允许。
- 模块独立。
- 复制范围小。
- 测试可覆盖。
- 不引入大量间接依赖。
- `THIRD_PARTY_NOTICES.md` 已登记。
- 修改记录清晰。

### 4.4 Current Runtime Slice

- 当前优先按 Python 依赖适配 Pyxelator，并通过短生命周期 sidecar 尝试常见模板定位 API。
- 未安装 Pyxelator、API 不匹配或模板无法命中时，provider 返回结构化不可用或未命中结果，不阻断后续 Airtest / builtin provider。
- 当前项目没有复制 Pyxelator 源码，因此暂不需要更新第三方源码登记；未来若复制小模块，必须先进 `third_party/` 并更新 `THIRD_PARTY_NOTICES.md`。
- Pyxelator provider 只输出坐标、区域、置信度和证据引用，不直接执行点击、不修改 workflow。

### 4.5 OCR Text Provider Boundary

- OCR text provider 属于 Python Sidecar 视觉增强，不属于第二执行体系。
- 当前 OCR 优先尝试本机 `pytesseract`，缺少包、缺少系统 OCR 能力或 API 不匹配时只返回结构化不可用。
- Text target 始终优先走 Appium Source 精确匹配；只有 Source 未命中且视觉增强开启时，才继续尝试 OCR。
- OCR 命中也只返回坐标、区域、置信度和证据引用，是否点击仍由 Runtime 的 Tap Target / Wait For Target / Visual Branch 决策。
- OCR 低置信、未命中或不可用时必须暂停或保留原未命中，不得盲点。

## 5. Appium Inspector Integration

### 5.1 What To Absorb

- Appium server 配置体验。
- Capability 编辑体验。
- Session 创建和错误表达。
- 截图与 page source 联动。
- 元素树和属性面板。
- 命令执行面板。
- Appium 问题归因方式。

### 5.2 How To Integrate

Appium Inspector 不作为 UI 依赖。

V4.0 在 Flutter 中重做：

- Inspector 页面。
- Session Drawer。
- Capability Panel。
- Element Tree。
- Element Detail。
- Command Panel。
- Screenshot Overlay。

允许移植：

- 命令分类。
- capability 字段分组。
- session 错误分类思路。
- inspector 状态模型。

禁止复制：

- Electron Shell。
- React 页面。
- 云平台集成 UI。
- 与本项目风格冲突的多平台服务入口。

## 6. appium-mcp Integration

### 6.1 What To Absorb

- MCP 工具抽象。
- AI Agent 调用 Appium 的边界。
- 工具权限分级。
- 视觉恢复思路。
- W3C actions 的工具化表达。
- Xcode / simulator 命令能力归类思路。

### 6.2 How To Integrate

不引入 Node MCP 服务。

V4.0 采用 Dart 侧 MCP-compatible Tool Model：

```text
AI / MCP Core
  -> Tool Registry
  -> Permission Gate
  -> Runtime Command Proposal
  -> User Confirmation
  -> Runtime Execution
```

工具只能生成建议或草稿，危险动作必须确认。

### 6.3 Explicit Node Ban

禁止：

- `mcp-appium-visual` 作为常驻 Node 服务进入主链路。
- Flutter 调 Node MCP API。
- Node 读取项目证据或截图。
- Node 直接调用 Appium 控制设备。

允许：

- 参考工具 schema。
- 移植安全工具分层。
- 在文档中引用能力来源。

## 7. AppiumAir Status

AppiumAir 不进入 V4.0 核心融合对象。

可参考：

- 设备自动发现思路。
- Appium 服务管理思路。
- 截图、视频、错误信息和报告的组织方式。

不吸纳：

- Android-only 运行时封装。
- 多 Appium 服务并发管理。
- 老旧 UI 或报告实现。

## 8. Third Party Governance

新增根文件：

- `THIRD_PARTY_NOTICES.md`

如果复制源码，必须登记：

- 项目名称。
- 上游链接。
- 许可证。
- 复制范围。
- 是否修改。
- 修改摘要。
- 本项目用途。
- 复核日期。

如果仅依赖包，必须登记到对应 package 文档或依赖清单，不需要复制源码到 `third_party/`。

## 9. Privacy Requirements

第三方能力不得保存：

- 完整设备标识。
- 完整 session。
- 本机绝对路径。
- 账号。
- 证书主体。
- 未经用户主动展开的隐私截图。

第三方日志进入本项目前必须脱敏。

## 10. Integration Acceptance

每接入一个外部项目能力，必须满足：

- 有明确 adapter。
- 有 fake 或 fixture 测试。
- 有本地失败诊断。
- 有脱敏策略。
- 有许可证记录。
- 不绕过 Runtime。
- 不恢复 Node。
- 不破坏 iOS 和 Android 对等闭环。
