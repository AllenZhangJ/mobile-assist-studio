# iOS Assist Studio V2.0 Development Plan

## 0. TL;DR

- V2.0 开发以 Flutter Desktop Mac App 为唯一主产品入口。
- V2.0 开发最多拆成三个阶段，三个阶段完成后必须形成完整 Enterprise Local Workstation。
- 终极目标必须包含完整 Workflow 画布，不能只停留在配置页或只读拓扑。
- 每个阶段都必须可运行、可验证、可回退。

## 1. Planning Principles

V2.0 开发遵循：

- Flutter Desktop 是唯一主产品入口。
- Dart Runtime 是 V2.0 业务运行时。
- Appium / WDA 是唯一主驱动。
- Node / Web 是 Legacy / Debug / Migration reference。
- Single Device First。
- 执行始终串行。
- 停止始终安全停止。
- Workflow DSL 是唯一工作流真源。
- 视觉能力可选，不成为核心运行硬依赖。
- 本地数据不上传云端。
- Primary User 是不会 Appium / WDA 的运营或 QA。
- 交互优先级是可理解性、灵活性、工程能力。

- Flutter App 可手动采集当前设备截图并在 Device 页面预览。
- L2 Device 预览在 connected 且 idle 时可点击和拖动当前截图，UI 只发送归一化屏幕意图，Runtime 通过 Appium `window/rect` 换算为 WDA viewport tap / swipe，并在动作后释放 pointer actions；运行中、未连接或无截图时必须锁定。
- Runtime 设备预览命令必须按截图、点击、手势和输入拆分，并复用 `runtime_device_preview_helpers.dart` 的 connected + idle + session 校验、坐标校验、时长校验、pinch 坐标构造和 actions 释放规则；Tap、Double Tap、Long Press、Swipe 和 Pinch 不得各自复制前置校验。预览 Tap 失败时仍必须释放 pointer actions，并用 Runtime 测试覆盖。
- L2 Device Preview 的交互状态模型、预览状态、坐标换算、预览舞台、头部组合、受控主页键、输入、缩放、点击/滑动覆盖层和动作发送应保持分片；交互状态模型只负责截图尺寸、手势痕迹、发送中标记、缩放和启用判断，预览状态层只负责截图解析和布局装配，动作发送按设备命令、点击、滑动、键盘滑动和动作 helper 继续拆分，坐标换算分片只负责显示区域与归一化坐标，预览舞台只负责手机外框、手势绑定和覆盖层组合，展示控件不得直接调用设备动作。
- L2 Device Center 可基于 Runtime snapshot 展示设备摘要、Local Readiness Guide、设备控制区和设备预览。
- Local Readiness Guide 覆盖 Appium Service、USB Device、Developer Trust、WDA Session、Safe Capture 和 Workflow DSL，并以用户可理解的下一步动作替代原始技术细节。
- L2 Device Center 可通过 Dart Runtime 手动刷新 Local Stack Check，覆盖 Appium CLI、Xcode CLI、iOS Device Tools、本机隧道和 WDA Prerequisites 的本机依赖诊断。
- 本机隧道缺失作为一键连接的内部处理项；Local Stack Check 只检查不启动隧道，“连接设备”可先自动对齐当前唯一 USB 手机，再在用户输入 Mac 密码后由 Runtime 受控启动 Appium XCUITest tunnel-creation，并等待目标手机出现在 tunnel registry 后再创建会话；如果点击前快照误判本机隧道就绪，但 Runtime 刷新后发现仍需要密码，Flutter 连接入口必须在同一次点击内补弹 Mac 密码框并继续连接，不要求用户再点一次；若已有 tunnel-creation 进程但 registry 为空，则先进入“等待手机允许”中间态，不继续创建 `/session`；若等待超时且该隧道不是当前 Runtime 持有的受控进程，Runtime 可在同一次密码授权内清理旧 tunnel-creation 进程并启动新的受控隧道；若只发现无线可用手机而没有 USB 手机，Runtime 必须停止一键连接并提示用户使用数据线连接，不得继续使用旧配置硬连；所有创建手机会话的入口都必须在发送 `/session` 前再次确认当前唯一 USB 手机并同步 session 配置，不能让直接重连绕过一键连接前置检查，也不能在缺少有效设备绑定时继续请求 Appium，更不能把脱敏或示例设备占位符发给 Appium；若 registry 或 session 显示 Appium 看不到目标手机，Runtime 可在唯一 USB 手机明确时自动重绑并重跑连接链路一次；若绑定手机已确认但驱动仍不可见该设备，Runtime 可重置当前端口旧驱动并重试一次：受控驱动正常停止，外部旧驱动只按当前 host/port 精准清理 Appium 主服务，不清理 tunnel-creation；最终失败时界面统一提示“驱动未识别手机”和“保持解锁，点连接设备。仍失败就重插线。”；若受控驱动创建手机会话时出现 `socket hang up`、`could not proxy command` 或 `port 8100` 等 WDA 代理瞬时断开，Runtime 可自动停止受控驱动、重新准备并重试一次；Xcode 构建、证书信任和 USB 问题不得自动重启。无法唯一确认 USB 手机时，界面统一提示“未找到 USB 手机”和“用数据线连接一台手机并解锁，再点连接设备”，不得展示绑定、UDID、脱敏占位符或 session 细节。Device Controls 必须把“连接设备”作为大号主操作；查环境、指引、停止驱动、重绑、断开和截图是辅助动作，连接、断开、驱动启动停止或运行处理中必须禁用可能打断状态机的辅助动作。Device Controls 仍提供“重绑”入口：Runtime 只绑定当前唯一 USB 手机，过滤 localNetwork 设备，并在写回本地项目配置后同步当前 session / tunnel 配置。pymobiledevice3 隧道仅作为辅助诊断通道。
- Local Setup Guide 可在本机隧道提醒或错误时提供“复制命令”动作作为高级备用；该动作只写入剪贴板，不执行命令、不打开终端、不请求权限。
- Local Setup Guide 代码必须保持外壳与内容分片：`device_setup_guide.dart` 只承载抽屉壳、快照读取和分区装配；`device_setup_guide_sections.dart` 只承载准备项卡片、复制命令、依赖内联摘要和 V2.0 边界提示。新增指引项先判断属于抽屉编排还是内容卡片，不得重新堆回单个大文件。
- Local Stack Check 不调用 Legacy Node，不列出设备清单，不展示完整路径、完整设备标识或原始命令输出。
- L2 Device Center 主界面只展示摘要信息，不展示完整设备标识、完整 session、WDA endpoint、source XML 或原始 WebDriver payload；连接失败时用共享连接诊断卡展示“原因”和“下一步”，并可复制脱敏诊断摘要；底层详情只进入状态抽屉、控制台或复制文本的脱敏区域；就绪检查里的“手机会话”项必须复用同一个连接诊断，不能展示与摘要诊断卡不一致的泛化指引。
- Flutter App 可执行一轮基础 Tap / Wait 模板。
- Flutter App 可执行 N 轮串行任务。
- L5 Execute 可展示 Run Configuration、Preflight、Command Center、Execution Summary、Runtime State、Last Run Evidence 和精简 Execution Timeline，并通过 Runtime snapshot 驱动执行状态；Execute 必须复用 Device 的一键连接主操作和同一 Runtime 连接链路，未连接时可在运行页直接连接，连接处理中运行与连接入口都必须锁定。
- L5 Execute UI 必须保持低耦合拆分：页面入口只组合布局，运行配置只承载单次/循环/持续模式和流程摘要，控件分片只承载有限轮次步进器，运行前检查只承载 Preflight，命令分片只承载启动、连接、停止和暂停收口命令面板，运行确认分片只承载确认弹窗和确认后 Runtime 提交，主按钮分片只承载高优先级按钮样式与异步入口，摘要分片只承载区域装配，焦点分片只承载进度与执行焦点，运行事实分片只承载 Runtime State，最近证据分片只承载 Last Run Evidence 入口，事件线分片只承载精简 Execution Timeline；Device 和 Execute 的连接主按钮必须共用 shared 分片，不在页面内复制连接文案、密码弹窗或禁用规则。
- Flutter App 可安全停止。
- 每个 Tap 后必须调用 release actions，Tap 失败时也要尽力释放。
- 连接状态和运行状态分离展示。
- 没有真实设备时 fake Appium tests 通过。

### 3.4 Acceptance

- 用户可从旧 sequence 导入基础工作流模板。

- 无效 workflow 不会进入运行。

- Workflow 页可展示画布 / 源码 / 检查的桥接形态；用户可见标签必须使用这三个短中文，代码内部可继续使用 visual / source / validate。
- 源码视图可编辑 Project DSL JSON，并在保存前完成解析与 validator 检查。
- 源码视图必须展示源码诊断，将解析或 validator 错误定位到节点、字段或源码级位置；点击诊断只定位编辑器选区，不写入 Project DSL、不触发运行；当诊断同时包含节点和字段时，必须优先定位该节点 JSON 对象内的字段，不能跳到其它节点的同名字段。
- 检查视图必须使用同一诊断模型展示离线 validator 结果；点击可映射节点的诊断必须切回画布并选中节点，无法映射节点的诊断必须切回源码并定位字段或源码级位置。
- 源码视图是高级编辑入口，可以保留节点 ID 和字段名用于源码定位；检查视图与 Node Inspector 是普通用户入口，必须把同一诊断转成节点名称、中文字段和短中文原因，不直接展示裸节点 ID、缺失子流程 ID 或底层字段。
- 源码诊断和检查视图必须合并 DSL 结构校验与本地项目级 Sub Workflow 引用校验；直接缺失子流程引用、嵌套缺失子流程引用、直接自引用和间接循环引用必须在编辑期可见，并能定位到对应 Sub Workflow 节点和 `workflowId` 字段；普通 `next` 自引用和 Catch `onError` 自引用必须显示短中文原因，并分别定位到 `next` 或 `onError` 字段；入口非 Start、Start 多主线和 End 继续连接也必须显示短中文原因，并定位到入口或后续字段。
- Workflow Source helper 必须保持低耦合拆分：`workflow_source_helpers.dart` 只保留路由说明，`workflow_source_models.dart` 只承载草稿和诊断模型，`workflow_source_parser.dart` 只承载 JSON 序列化、解析和项目级 validator 调用，`workflow_source_diagnostic_extractors.dart` 只承载错误到节点、字段和备用定位文本的提取，Source 短文案、用户文案和编辑器选区定位继续进入各自分片。
- Visual Canvas 节点卡片和 Node Inspector 必须只读展示节点级 validator 问题；该提示只来源于同一个 Project DSL validator，不写入 DSL、不触发运行、不形成第二套校验状态。
- Top Status、Dashboard、Device Readiness、Execute、Settings 和 Bottom Console 的 Workflow 状态必须复用同一个项目级校验 helper；缺失子流程或自引用时，全局入口不得显示“可运行”，Dashboard 进入 Execute 的快捷入口必须禁用，状态详情必须给出短中文原因。
- 跨页面 readiness helper 必须保持低耦合拆分：依赖检查时间、短状态、色调和图标归属 `dependency_status_helpers.dart`；驱动服务准备项归属 `readiness_driver_entries.dart`；USB 手机和开发者信任准备项归属 `readiness_device_entries.dart`；WDA 手机会话和安全截图准备项归属 `readiness_session_entries.dart`；流程文件准备项归属 `readiness_workflow_entries.dart`；准备度图标归属 `readiness_icon_helpers.dart`；`readiness_helpers.dart` 和 `readiness_entry_helpers.dart` 只保留路由说明。Device、Execute、Recorder 和 Top Status 不得各自维护重复准备度 switch。
- 无效 Source 草稿不得替换当前 Runtime workflow。
- Source 保存必须写入 Runtime snapshot 和本地 Workflow Store。
- 运行中 Source 保存必须锁定。
