# iOS Assist Studio V3.0 PRD: Cross-Platform Mobile Workstation

## 0. TL;DR

- V3.0 的正式定位是：单人单设备的跨平台本地移动自动化工作站。
- V3.0 从 iOS-only 走向 iOS / Android，但仍坚持本地桌面、单人、单台设备、串行执行和安全停止。
- V3.0 要吸收 TestHub 类竞品已验证的痛点解法：设备资源化、目标库、远控与自动化共存、图像/OCR 辅助定位、执行状态与测试结果分离。
- V3.0 不做 SaaS、多用户、权限、审计、设备池、云端协作、多设备并发或完整测试管理平台。
- V3.0 的用户价值是：不会移动自动化框架的 QA / 运营，也能在本机完成连接设备、创建目标、编排流程、执行、介入和复盘。

## 1. Product Definition

`iOS Assist Studio V3.0` 是面向 iOS 和 Android 的本地移动自动化工作站。

它不是把 Appium、ADB、WDA、scrcpy、OCR 或 CV 暴露给用户，而是把这些底层能力组织成一个可理解的工作区：

- 连接一台当前设备
- 远控或预览设备现场
- 从现场创建目标
- 用 Target + Action 编排流程
- 串行运行工作流
- 低置信或异常时暂停介入
- 查看本地证据、日志、性能和结果归因

V3.0 的核心产品句：

```text
把一台手机上的人工操作流程，变成本机可维护、可执行、可诊断的跨平台自动化工作流。
```

## 2. Positioning Boundary

V3.0 是：

- 本地 Mac 桌面工作站
- 单人单设备移动自动化工具
- iOS / Android 跨平台工作流编排器
- 目标驱动的移动自动化工具
- 面向非 Appium / ADB 专家的可视化操作环境
- 面向自动化工程师的可扩展 Runtime 底座

V3.0 不是：

- SaaS 平台
- 多租户平台
- 权限系统
- 审计系统
- 云端协作工具
- 设备农场
- 群控平台
- 多设备并发执行器
- 完整测试管理平台
- 任意脚本执行平台

## 3. Why V3.0

V2.0 已经解决了 iOS 本地工作站的核心闭环：

- Flutter Desktop 主入口
- Dart Runtime
- Appium / WDA 主驱动
- Project DSL
- Workflow Canvas
- Recorder
- Execute
- Monitor
- 本地 evidence

但 V2.0 仍缺少竞品已经证明重要的能力：

- Android 支持
- 设备资源锁产品化
- 目标库
- 图像识别和 OCR 辅助定位
- 远控、录制和执行的统一资源模型
- App 生命周期
- 平台日志和性能采集
- 任务状态和用例结果的明确分离

V3.0 的目标不是守着现有资产不变，而是在保留 V2 安全底座的前提下，拥抱这些真实痛点。

## 4. Target Users

### 4.1 Primary User

Primary User 是不会 Appium、WDA、ADB 或 Airtest 的运营、QA 或业务测试人员。

他们需要：

- 插上一台手机就知道能不能用
- 不写代码也能创建目标
- 不理解坐标也能录制动作
- 不理解驱动差异也能运行流程
- 出错时知道是设备、驱动、目标、流程还是断言问题

### 4.2 Secondary User

Secondary User 是自动化工程师。

他们需要：

- 稳定的平台抽象
- 可扩展的 iOS / Android adapter
- 明确的 DSL 真源
- 可测试的 Target Resolver
- 可追踪的运行事件和证据
- 可验证的安全边界

### 4.3 Interaction Priority

V3.0 交互优先级：

1. 可理解性
2. 安全性
3. 稳定性
4. 灵活性
5. 工程能力

任何需要用户理解 UDID、WDA endpoint、ADB serial、session、source XML、raw payload 的信息，都默认进入 Drawer、Inspector、Advanced 或 Bottom Console，并且必须脱敏。

## 5. Product Goals

V3.0 需要达成：

- 让用户在一个桌面 App 中连接 iOS 或 Android 当前设备。
- 让用户通过远控/预览创建可复用目标。
- 让用户用 Target + Action 编排业务流程。
- 让 Runtime 根据当前平台选择正确驱动和能力。
- 让失败从单一错误升级为可诊断的问题归因。
- 让本地报告区分任务状态、用例结果和问题类型。
- 让视觉、OCR 和坐标定位成为辅助能力，而不是不受控的第二套执行体系。

## 6. Success Criteria

V3.0 体验成功的判断标准：

- 用户能连接一台 iOS 或 Android 设备，且同一时刻只允许一台当前设备。
- 远控、录制、目标创建和执行共用同一当前设备资源锁。
- 用户能创建图片、坐标、区域、文本或 selector 目标。
- Workflow 节点可以引用目标，而不是只写死坐标。
- iOS 既有 Appium / WDA 流程不退化。
- Android 至少完成截图、点击、滑动、输入、启动 App 和基础日志闭环。
- 低置信视觉/OCR 判断默认进入 paused。
- Run Detail 能区分基础设施问题、驱动问题、目标问题、流程问题和断言问题。
- 所有数据默认本地保存，不上传云端。

## 7. Functional Scope

### 7.1 L1 Dashboard

目标：打开工作站后第一眼知道当前平台、设备、流程和最近运行状态。

新增能力：

- 当前平台摘要：iOS / Android / 未连接
- 当前设备资源状态：空闲、远控中、录制中、运行中、暂停中、错误
- Target Library 摘要
- 最近运行的问题类型摘要

边界：

- Dashboard 不承载目标编辑、驱动配置或日志详情。

### 7.2 L2 Device

目标：跨平台设备工作台。

能力：

- 连接当前 iOS 或 Android 设备
- 展示平台、系统版本、连接方式、驱动状态和当前 App
- 展示远控/预览
- 支持点击、滑动、输入、主页键或平台等价动作
- 支持从当前画面创建目标
- 展示设备资源锁状态

边界：

- 只允许一台当前设备。
- 不做设备列表调度。
- 不做多设备并发。
- 不做远程设备池。

### 7.3 L3 Recorder

目标：把人工操作捕获为 Target + Action。

能力：

- 录制 Tap、Wait、Swipe、Input、Launch App、Stop App
- 录制时可选择创建或复用目标
- 动作时间轴默认显示业务名称和目标名称
- 坐标、selector、图片裁剪和 OCR 细节进入 Drawer
- Promote 写入 Project DSL

边界：

- 录制动作默认不直接执行设备动作，除非用户在 Device 远控区明确操作。
- 录制截图不长期暴露本机路径或完整设备标识。

### 7.4 L4 Target Library

目标：管理可复用的移动 UI 目标。

V3.0 可将 Target Library 作为 Workflow 内部一级面板，或作为 L4 Workflow 下的固定侧栏；是否升为独立导航由 IA 文档决定。

目标类型：

- Coordinate Target
- Image Target
- Region Target
- Text / OCR Target
- Accessibility / Selector Target
- App Target

能力：

- 从设备截图创建目标
- 裁剪图片目标
- 标注区域目标
- 配置 OCR 文本匹配
- 配置平台 selector
- 设置置信度阈值、等待策略和兜底动作
- 目标可被 Workflow 节点引用

边界：

- Target Library 不保存账号、完整设备标识或生产环境私密数据。
- 目标解析失败不允许盲点，默认挂起或失败收口。

### 7.5 L5 Workflow

目标：跨平台目标驱动的流程编排。

新增节点方向：

- Launch App
- Stop App
- Tap Target
- Wait For Target
- Assert Target
- Assert Text
- Collect Logs
- Collect Performance
- Manual Checkpoint

保留：

- Tap
- Wait
- Swipe
- Input
- Snapshot
- Condition
- Visual Branch
- Catch
- Loop
- Sub Workflow

边界：

- Project DSL 仍是唯一真源。
- DSL 不开放任意 JavaScript、Python 或 shell。
- 平台差异必须通过 Runtime capability report 和 adapter 处理。

### 7.6 L6 Execute

目标：放心开始、看懂进度、随时安全停止和人工介入。

新增能力：

- 运行前检查当前平台 capability
- 检查 Target Library 引用是否完整
- 检查当前设备资源锁是否空闲
- 运行中展示当前目标、当前动作、当前平台 adapter
- 支持低置信 paused 和 Manual Checkpoint
- 结果区分任务状态和用例结果

边界：

- 始终串行执行。
- 停止仍为安全停止。
- 运行中不允许修改当前 workflow 或目标定义。

### 7.7 L7 Monitor

目标：解释运行结果、现场证据和下一步动作。

新增能力：

- 任务状态：completed、failed、paused、stopped、infrastructure_error
- 用例结果：passed、assertion_failed、inconclusive、not_evaluated
- 问题类型：device、driver、target、workflow、assertion、visual、timeout、manual
- 关联截图、日志、性能、目标解析和视觉/OCR 证据
- 本地报告导出候选能力

边界：

- Monitor 只读本地数据。
- 不上传云端。
- 不接远程分析服务。

## 8. Technical Scope

V3.0 主体技术路线：

- Flutter Desktop Mac App 继续作为主入口
- Dart Runtime 继续作为业务运行时
- Project DSL 继续作为工作流真源
- 新增跨平台 Driver Abstraction
- iOS adapter 基于 Appium / WDA / XCUITest
- Android adapter 优先基于 Appium UiAutomator2 和 ADB
- scrcpy 可作为 Android 远控增强
- Airtest / OpenCV / OCR 可作为目标解析辅助能力

辅助能力不得绕过 Runtime 直接执行用户不可见动作。

## 9. Data And Privacy

默认本地保存：

- Workflow DSL
- Target Library
- 目标截图裁剪
- 运行历史
- 运行事件
- 截图证据
- 日志摘要
- 性能采样
- 失败归因摘要

默认禁止：

- 上传云端
- 保存完整设备标识到文档或公共输出
- 保存完整 session
- 保存 WDA endpoint / ADB serial 到用户可见主界面
- 保存账号、密码、证书主体或生产环境私密数据
- 在复制摘要中包含本机绝对路径

## 10. AI Boundary

V3.0 可为 AI 辅助预留扩展点：

- 根据手动流程建议目标和动作
- 根据失败证据生成诊断摘要
- 根据截图建议等待或断言

V3.0 不包含：

- AI Agent 自动控制全部流程
- 无确认的自动修复
- 云端截图分析
- 绕过平台安全限制

## 11. Naming Boundary

V3.0 文档可以继续使用 `iOS Assist Studio` 作为当前仓库名。

但产品和代码应开始引入平台中立概念：

- Mobile Workstation
- Mobile Device
- Mobile Driver
- Target Library
- Platform Adapter
- Device Resource Lock

是否正式更名为 `Mobile Assist Studio` 或其它品牌名，另开品牌/命名决策，不阻塞 V3.0 技术文档落地。

## 12. Source Relationship

V3.0 不废弃 V2.0。

V2.0 是 iOS 本地工作站底座；V3.0 是跨平台扩展和目标驱动升级。
