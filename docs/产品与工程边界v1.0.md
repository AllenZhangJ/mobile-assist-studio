# 产品与工程边界v1.0

## 0. TL;DR

- 本文是 `iOS Assist Studio` 1-3 阶段开发的边界真源。
- 后续重构、UI 升级、视觉能力、工作流 DSL 和验证策略都必须遵守本文。
- 如果本文与阶段计划或局部需求冲突，先以本文为边界，再调整计划或实现。

## 1. Product Boundary

- 服务对象只面向本机 Mac + 一台或少量 USB iPhone。
- 不做云端、多用户、远程设备池或设备农场。
- V2.0 主入口是 Flutter Desktop Mac App。
- Web 控制台和 CLI 已归档为 Legacy，对照调试时只能通过 `legacy:*` 入口使用。
- A-F 坐标序列作为第一套基础工作流模板保留。
- 节点工作流上线后，A-F 可被导入、包装和向下兼容。

## 2. Device And Connection Boundary

- 第一版仅支持单台已连接 iPhone。
- 暂不做多设备并发、群控或多设备调度。
- Appium / WDA 是主驱动机制，不替换为其他主驱动框架。
- 可以引入 `pymobiledevice3` 作为截图、高频日志和底层诊断辅助通道。
- 点击、元素交互和核心 WebDriver 会话仍以 Appium / WDA 为主。
- Appium 状态探测必须有超时兜底，不能让 Web 状态接口或断开连接流程长期阻塞。
- CLI 点击入口和兼容辅助入口的 Appium `/status` 探测也必须有超时兜底，不能因为本地端口接受连接但不返回而卡住启动或连接流程。
- 断开连接必须能取消或限时等待正在创建中的 WebDriver 会话；晚到的连接失败不能把已断开的会话重新写成错误态。
- 证书信任、Developer Mode、WDA 构建失败等 iOS 系统限制，只做显式状态提示、错误归类和用户引导。
- 不尝试绕过 iOS 系统安全限制。

## 3. Execution Safety Boundary

- 点击任务始终串行执行。
- 多轮任务严格按顺序排队，不并行点击。
- 同一 WebDriver 会话运行任务时，不允许并发执行手动截图、手动视觉分析或其他手动设备命令，避免和 runner 抢占 Appium/WDA 命令通道。
- 非运行态 WDA 命令必须有互斥锁；手动截图、手动视觉分析和连接后自动截图进行中时，不能启动点击任务、断开/重连会话或另一个手动设备命令。
- 如果非运行态 WDA 命令对前端请求超时，但底层 WebDriver 调用尚未结算，互斥锁应继续保留到调用结算或硬上限释放，避免超时后立刻与点击任务抢占同一会话。
- 停止任务采用安全停止：当前原子动作结束后优雅停下。
- 不强杀正在执行的点击或等待动作。
- Web 服务进程退出时也必须先请求 runner 安全停止并等待其进入 idle；等待有明确超时，超时后才继续释放 WDA/HTTP 资源。
- 异常拦截器必须有最大尝试次数。
- 自动重试超过阈值后必须进入 `Paused` 或等价挂起态，等待人工确认。
- 证书信任等待属于连接期自动重试，必须有上限；看板应展示当前重试次数和上限，超过上限后进入可操作错误态，等待用户信任后手动重新连接。
- 视觉或 OCR 判断置信度不足时默认挂起，不继续盲目点击。
- 当前视觉守卫只允许向运行时返回 `continue`、`warn` 或 `pause` 决策，不允许直接执行点击。
- Tap 前置视觉守卫如果自身失败、超时或无法给出判断，必须转为挂起态；不能把守卫异常当作通过并继续点击。
- Tap 前置视觉守卫连续挂起达到默认阈值后，下一次用户点击继续视为人工确认；运行时必须记录 warning 后才允许继续该 Tap。

## 4. Visual Capability Boundary

- Phase 2 只先做基础视觉能力。
- 优先能力包括截图、红点标注、失败现场快照、基础已知弹窗识别。
- 第一版基础识别可以基于截图元数据和 WDA pageSource 的已知系统弹窗规则，不要求引入 OCR/CV 模型。
- 不在早期追求通用复杂 AI 智能。
- CV / OCR 是可选能力。
- 无模型或外部环境依赖时，系统仍必须可以运行纯坐标或表达式工作流。
- 视觉 / OCR 节点在能力不可用时应变灰或明确不可用，不影响核心底座。
- 每一次视觉状态判断都必须留下本地证据，包括截图、识别结果、命中规则和后续动作。
- 视觉证据不得保存完整 pageSource；只保留规则命中摘要、置信度、决策和截图引用。
- 视觉分支没有可靠判断结果或置信度低于运行阈值时，必须先进入挂起态，等待人工确认后才能进入后续分支。

## 5. Workflow Boundary

- 工作流采用项目内强类型自定义 DSL。
- 不强绑定第三方低代码平台。
- 表达式能力只允许声明式读取上下文字段。
- 不开放任意 JS 执行，不使用 `eval` 类能力。
- 第一批节点只包括：`Tap`、`Wait`、`Snapshot`、`Visual_Branch`、`If_Else`、`Catch`、`Sub_Workflow`。
- 工作流配置必须兼容旧 `sequence.json`。
- 系统启动或执行时应能把旧序列映射为标准节点树。
- 工作流图必须校验入口、节点引用、重复 ID、不可达节点和闭环。
- 工作流保存时必须规范化为当前 DSL 白名单字段，避免把额外字段、私密字段或未知结构长期写入配置。
- 工作流写入必须避开连接生命周期和运行生命周期；初始化、连接、等待证书信任、断开和运行期间必须由后端拒绝。
- 工作流写入与任务启动、设备初始化和重连必须互斥；保存或清除配置期间，不允许并发保存、清除、启动运行、初始化设备或重连设备。
- 离线 validator 必须接受运行时支持的合法非线性节点图；无法线性化只能作为摘要状态，不应误判为配置错误。
- 旧 `sequence` 自动包装为线性 `Tap` / `Wait` 节点树，不强制迁移配置。
- 旧 click CLI 兼容入口必须优先使用 workflow；线性 workflow 可转换执行，非线性 workflow 必须明确拒绝并引导使用 Web 控制台 runner。
- 当前执行器可执行旧序列、线性 Tap / Wait workflow，以及基础 `Snapshot`、`Visual_Branch`、`If_Else`、`Catch` 节点图。
- 当前执行器对 `Visual_Branch` 执行低置信度兜底：无判断结果或低置信度不会直接落入默认点击路径。
- `Catch` 在第一版工作流运行时中支持显式 `onError` 跳转；`Sub_Workflow` 只保留类型和 hook 边界，不做复杂调度。

## 6. Data And Privacy Boundary

- 运行历史、截图快照和诊断日志只保存在本机。
- 允许保存到项目内运行证据目录或系统临时目录，默认使用本机 evidence 目录。
- 不上传任何云端。
- 截图默认只保留最近 N 次或最近 24 小时，避免长期沉淀隐私画面。
- 运行证据目录默认不进入版本管理。
- 对外展示证据位置时使用相对路径，不暴露本机绝对路径。
- 对外展示证据位置不能包含父级逃逸片段；如果证据目录被配置到项目外，也必须降级为安全展示路径。
- 结构化事件中不重复写入截图 base64 内容，截图以本地文件引用承载。
- 截图文件名必须带唯一后缀，避免同一毫秒同一原因的截图互相覆盖。
- 文档、本地日志和前端 UI 不长期写入、不硬编码 UDID、账号、证书主体或本机绝对路径。
- 运行事件、时间轴、SSE 广播和 evidence 事件文件必须统一脱敏，不暴露本机绝对路径、完整设备标识、账号或证书主体。
- 公共文本与公共 payload 必须同时脱敏带连字符和无连字符的长设备标识。
- UI 展示设备标识时应做最小必要展示，优先使用摘要或配置变量。
- Evidence store 的异步写入必须保持运行内顺序；Web 看板以 fire-and-forget 方式写入 `runStart`、运行事件和 `runEnd` 时，事件不能错落到 session 记录中。

## 7. UI/UX Boundary

- Web 控制台采用旗舰化 Tech Noir / Cyber Grid 方向。
- UI 的核心目标是工具效率、信息密度和掌控感。
- 不做营销页，不做冗余无用页面。
- 最终布局是：左侧数字孪生设备视窗，中间工作流画布拓扑，右侧属性 / 动态参数变量池，底部全链路时间轴。
- Phase 1 只做看板升级、事件区和设备视窗 MVP。
- 完整拖拽画布在 Phase 3 落地。

## 8. Verification Boundary

- 每个 V2 阶段交付物必须通过 `fvm dart run tool/v2_verify.dart`。
- 项目级真源关系必须通过 V2 verifier；如涉及归档 Node 对照能力，还必须通过 `npm run legacy:smoke:acceptance`，包括文档入口、A-F 基础序列、脚本覆盖和公共文本隐私扫描。
- 从 Phase 1 开始补充 dry-run 和 validator 能力。
- 不连接真实 iPhone 时，也应能验证工作流 DSL 拓扑结构的闭环与合法性。
- 真机相关能力按环境分级验收：本机可用时现场验证；不可用时至少保证 dry-run、静态检查和自动化单元检查通过。
- Web 控制台生命周期必须有本地 smoke 验证。
- Web 服务关闭时必须主动释放 SSE 日志流和 HTTP 长连接，不能依赖浏览器自然断开。
- Web 服务关闭时必须先停止接收新连接，再清理空闲和活跃连接；慢请求不能阻止进程退出。
- Web 服务必须设置明确的 HTTP 请求、headers 和 keep-alive 超时，避免慢请求长期占用本地控制台资源。
- Web 服务的异步请求处理必须有最终兜底；单个坏请求、URL 解析异常或响应写入异常不能形成未处理 Promise 或导致看板进程退出。
- Web 前端的 API 请求和 SSE 实时流也必须有可见兜底；网络异常、实时流断开和结构化校验失败必须进入诊断日志或编辑区状态，不能静默失败。
- Web 前端处理 SSE 时必须同时防御非法 JSON 和合法 JSON 但结构异常的事件；坏事件只能写入诊断日志，不能中断后续页面事件处理。
- Web 前端按钮可用态必须与后端互斥 guard 保持一致；运行、连接生命周期或 workflow 写入锁定期间，不应展示会被后端拒绝的主流程操作为可用状态。
- Web 时间轴必须区分 `runEnd` 的成功、停止和错误结束；错误结束不能被展示为任务完成。
- 运行失败后的自动截图必须有超时上限；截图卡住只能降级为诊断日志，不能阻塞 `finishRun` 和运行证据收尾。
- 运行失败后的自动截图必须使用设备命令锁，避免和人工截图或视觉分析抢占同一个 Appium/WDA 命令通道。
- Web 控制台触发的 WDA 截图和 pageSource 采集都必须有超时上限；手动截图、视觉守卫、动作后证据和工作流 Snapshot 不能无限等待 WDA 响应。
- 运行中的截图、视觉守卫、动作后证据和工作流 Snapshot 超时后，runner 仍必须等待底层 WDA 命令结算或等待到硬上限后再继续后续节点，避免超时返回后立刻与下一次 Tap 抢占 WebDriver 通道。
- Web 常驻会话和 CLI 点击入口清理 WebDriver 会话、释放 pointer actions 时必须有超时上限；清理失败只能进入诊断日志或错误事件，不能让停止、断开或任务收尾无限阻塞。
- Web API 请求体必须有大小上限；malformed JSON 或超大请求必须返回结构化错误，并且不能影响后续状态接口和控制台生命周期。
- 连接生命周期必须由后端会话对象保证 single-flight；重复断开要复用同一个断开流程，断开中不能启动新的连接。
- `record:connected` 和 `pick-d:connected` 这类兼容辅助 Web 入口也必须设置请求体上限、malformed JSON 结构化 400、HTTP 超时、原子配置写入和关闭兜底，不能因为不是主入口而绕开本地服务生命周期边界。
- `record:html:connected` 这类离线 HTML 辅助入口也必须遵守日志隐私边界，不能在默认输出中暴露本机绝对路径或默认 Appium 二进制绝对路径。
- `record:connected`、`record:html:connected` 和 `pick-d:connected` 捕获截图与窗口尺寸时必须设置 WDA 命令超时；截图或尺寸采集卡住时应快速失败，而不是让辅助入口无限等待。
- Web 状态接口必须能在配置异常时降级返回结构化错误，避免看板白屏或刷新链路中断。
- Web 公共 API 的错误信息必须脱敏，不向前端暴露本机绝对路径。
- Web 状态接口里的 `session.error` 也属于公共 API payload，`message` 和 `userAction` 必须脱敏后再返回前端。
- Web 实时日志必须统一脱敏，不向前端暴露本机绝对路径或设备标识。
- Web 运行事件、时间轴和 evidence 事件写盘必须统一脱敏，不能只保护诊断日志。
- Workflow API 必须能在工作流配置异常时降级返回结构化校验结果，不回传原始无效 workflow，避免配置面板失去诊断能力或暴露未知字段。
- Workflow API 在配置文件缺失或无法读取时也必须返回结构化校验失败结果，不能让工作流编辑区只看到请求失败。
- Web 状态接口中的 USB 设备探测必须有超时降级、single-flight 保护和冷却后重试，不能让底层设备服务阻塞页面刷新、因刷新风暴堆积探针，或因单次挂起永久失去恢复能力。
- Appium 状态探测、超时兜底和连接生命周期基础能力必须有本地 smoke 验证。
- 设备初始化和 workflow 保存写入配置文件时必须使用原子写入，避免进程中断导致配置半写。
- Runner 串行执行、安全停止和事件输出必须有离线 smoke 验证。
- Runner 启动阶段的配置读取、旧序列包装或 DSL 校验失败，也必须输出结构化 `runError` 和错误态 `runEnd`，不能只停留在诊断日志。
- Tap 前视觉守卫挂起后执行停止，也必须输出对应 `stepEnd`，避免时间轴出现悬空步骤。
- 本地 evidence store 必须有离线 smoke 验证。
- Evidence store 必须覆盖 fire-and-forget 队列顺序验证，证明快速运行或快速失败时 `runStart`、事件和 `finishRun` 仍落在同一运行记录。
- 视觉分析和挂起 / 继续状态必须有离线 smoke 验证。
- 工作流 DSL、旧序列映射、闭环检测和表达式白名单必须有离线 smoke 验证。
- 新增长期维护的 V2 Dart / Flutter 源文件和测试时，必须同步纳入 `fvm dart run tool/v2_verify.dart` 覆盖范围；如确需维护归档 Node 对照脚本，必须使用 `legacy:*` 入口并同步 `legacy:check`。

## 9. Default Engineering Values

未被显式配置覆盖时，采用以下默认值：

- 运行证据保留最近 50 次或最近 24 小时。
- 同一异常阻断态最多自动处理 2 次，之后进入挂起态。
- Tap 前置视觉守卫最多连续挂起 2 次，第二次人工继续后视为明确确认。
- 视觉判断置信度低于 0.75 时挂起。
- 视觉判断置信度在 0.75 到 0.9 之间时记录 warning。
- 视觉判断置信度高于 0.9 时视为可信。
- Phase 1 截图只采关键帧：连接后、每轮开始、失败、手动刷新。
- Phase 2 再扩展到动作前后截图。

## 10. Change Rule

- 任何实现如果需要突破本文边界，必须先更新本文并获得明确确认。
- 任何阶段完成前，必须检查实现是否仍符合本文边界。
