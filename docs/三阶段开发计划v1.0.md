# 三阶段开发计划v1.0

## 0. TL;DR

- 本计划把现有文档中的产品方向、UI/UX 规范和架构路线压缩为三个可交付阶段。
- 三个阶段完成后，项目应从“Web 看板 + 坐标点击工具”升级为“视觉状态驱动的 iOS 辅助自动化编排器”。
- 每个阶段都必须可运行、可验证、可回退；不允许只完成文档或半成品框架。

## 1. Planning Principle

三阶段划分遵守以下原则：

- 开发边界以 `docs/产品与工程边界v1.0.md` 为最高阶段约束。
- 先稳定控制台，再增强运行时，最后升级编排器。
- 每一阶段都要交付用户可操作能力。
- CLI 兼容入口在第三阶段完成前继续保留。
- 当前 A-F 串行点击规则在新工作流完成前仍是执行真值。
- 视觉能力先用于证据和辅助判断，再逐步进入决策链路。

## 2. Phase 1: 旗舰控制台与运行证据底座

### 2.1 目标

把当前 Web 看板升级为稳定、清晰、可长期使用的本地控制台。

这一阶段不追求复杂工作流，而是把“连接、运行、停止、日志、序列、截图证据”做扎实，让用户可以主要通过 Web 控制台完成日常操作。

### 2.2 必做能力

- 控制台 UI/UX 升级为 Tech Noir / Cyber Grid 工作台风格
- 顶部全局状态栏：连接态、运行态、Appium、配置摘要
- 左侧设备视窗 MVP：当前截图、最近点击坐标、最近动作提示
- 中部序列视图：展示 A-F 当前序列、等待时间、当前执行步骤
- 右侧属性视图：展示当前配置、设备、运行参数
- 底部日志与事件区：实时日志、复制全部、清空全部、错误高亮
- 运行事件模型：把 A-F 每一步转换为结构化事件
- 失败现场证据：失败时保留最近截图、步骤、错误摘要
- 本地 evidence store：运行事件、关键截图、失败现场写入本机目录，并按保留策略滚动清理
- 离线配置校验：不连接真机也能校验序列、运行参数和 Appium 能力结构

### 2.3 默认落点

- `legacy/node/src/ios-assist-console.mjs`
- `legacy/node/src/ios-assist-runner.mjs`
- `legacy/node/src/ios-assist-session.mjs`
- `legacy/node/src/ios-assist-evidence.mjs`
- `config/connected-device.sequence.json`
- `legacy/node/src/ios-assist-validate.mjs`
- `docs/`

### 2.4 验收标准

- 用户可以只打开 Web 控制台完成初始化、测试 1 轮、执行 N 轮、停止任务。
- 控制台清楚区分连接状态和运行状态。
- 每个 A-F 步骤至少能在事件区看到开始、结束、结果和耗时。
- 点击失败时可以看到失败步骤、日志摘要和最近屏幕证据。
- 关键截图和事件可以在本机 evidence 目录中找到，路径不暴露本机绝对路径。
- `npm run legacy:validate` 和 `npm run legacy:validate:connected` 可用于归档 Node 离线配置校验。
- `npm run legacy:smoke:click` 可验证旧 click CLI 对线性 workflow 的兼容、非线性 workflow 的明确拒绝，以及 dry-run 不建立 Appium 会话。
- `npm run legacy:smoke:acceptance` 可验证 Legacy Web / CLI 对照入口、dry-run 和 validator 入口没有偏离归档边界，并验证 click CLI 的初始化与设备预检发生在 Appium 启动之前。
- `npm run legacy:smoke:runner` 可验证 Legacy runner 串行事件和安全停止。
- `npm run legacy:smoke:session` 可验证 Appium 状态探测、USB 探测通用超时兜底、配置原子写入和连接生命周期基础能力。
- `npm run legacy:smoke:device-discovery` 可验证状态刷新中的 USB 设备探测 single-flight、冷却和重试保护。
- `npm run legacy:smoke:public-text` 可验证公共 API 与 Web 日志文本脱敏规则。
- `npm run legacy:smoke:validate` 可验证离线 validator 接受合法非线性 workflow。
- `npm run legacy:smoke:console-guards` 可验证连接生命周期和运行生命周期中的 workflow 写入保护。
- `npm run smoke:console` 可验证 Web 控制台服务生命周期、SSE 长连接关闭、异常配置降级、公共错误脱敏和未连接兜底。
- `npm run smoke:evidence` 可验证本机证据写入、截图引用和滚动留存规则。
- `npm run smoke:visual` 可验证基础视觉规则和挂起决策。
- `npm run check` 通过。

### 2.5 不做事项

- 不做完整节点画布。
- 不做复杂条件分支。
- 不把配置格式一次性迁移为新 DSL。
- 不引入多设备并发执行。

## 3. Phase 2: 视觉辅助与异常自愈运行时

### 3.1 目标

把截图和基础视觉判断纳入执行链路，让运行时具备“看见现场、识别阻断、尝试恢复”的能力。

这一阶段的重点不是低代码画布，而是让自动化不再只依赖盲点坐标。

### 3.2 必做能力

- 截图服务统一封装：连接后可主动获取当前屏幕
- 动作前截图、动作后截图、失败截图
- 设备视窗支持截图刷新和点击标注
- 轻量视觉判断接口：当前先支持 PNG 元数据校验和 WDA pageSource 的已知系统弹窗规则
- 异常拦截器 MVP：识别常见系统弹窗或阻断态，先挂起等待人工处理，不自动点击
- 拦截器安全边界：挂起态、人工确认入口、后续再扩展最大尝试次数和自愈动作
- 自愈事件记录：拦截、点击、复检都写入时间轴事件
- 看板新增阻断态：未知画面、等待人工处理、已恢复

### 3.3 默认落点

- 新增运行时视觉/截图模块
- 新增异常拦截模块
- `legacy/node/src/ios-assist-visual.mjs`
- `legacy/node/src/ios-assist-session.mjs`
- `legacy/node/src/ios-assist-runner.mjs`
- `legacy/node/src/ios-assist-console.mjs`
- `docs/视觉状态驱动编排架构路线v1.0.md`

### 3.4 验收标准

- 用户可以在看板看到当前 iPhone 屏幕截图。
- 每次测试 1 轮后，至少能回看关键动作截图或失败截图。
- 遇到已配置的常见弹窗时，系统可以暂停主流程、处理弹窗、复检后继续。
- 遇到未知阻断态时，系统不自动处理；进入挂起态的通用策略在工作流视觉节点阶段补齐。
- 所有异常处理过程能在日志或事件区追踪。
- `npm run smoke:visual` 通过。
- `npm run check` 通过。

### 3.5 不做事项

- 不做完整拖拽画布。
- 不做开放式脚本表达式执行。
- 不要求 OCR / CV 一次覆盖所有场景。
- 不允许视觉层绕过运行时直接下发动作。

## 4. Phase 3: 视觉状态驱动工作流编排器

### 4.1 目标

完成项目最终形态：从 A-F 坐标序列工具升级为可编排、可观察、可诊断、可视觉判断的 iOS 辅助自动化工作站。

这一阶段要把前三份专题文档里的目标全部收口。

### 4.2 必做能力

- 工作流节点模型：点击、等待、截图、OCR、CV 判断、条件分支、异常处理、子流程
- 旧 A-F 序列兼容：可自动映射为基础节点组
- 工作流编译层：节点图解析、合法性校验、闭环检测
- 运行时上下文池：节点输入、节点输出、设备状态、视觉结果
- 表达式读取能力：支持从上下文读取字段用于节点参数
- 条件分支：基于节点结果或上下文状态选择下一步
- 异常分支：节点声明 `onError` 时可进入 `Catch` / 恢复节点，未声明时保持失败退出
- 三栏工作台：设备视窗、工作流画布、属性与上下文面板
- 时间轴回放：按步骤查看事件、截图、结果、耗时
- 热力图分析 MVP：聚合失败坐标或失败节点
- CLI 兼容策略：保留基础命令，或明确迁移到工作流执行入口

### 4.3 默认落点

- `legacy/node/src/core/`
- `legacy/node/src/runtime/`
- `legacy/node/src/compiler/`
- 当前过渡期落点：`legacy/node/src/ios-assist-workflow.mjs`
- `legacy/node/src/vision/`
- `legacy/node/src/console/`
- `legacy/node/src/ios-coordinate-*.mjs`
- `config/`
- `docs/`

目录拆分应在本阶段按真实代码迁移推进，不在 Phase 1 或 Phase 2 过早重排。

### 4.4 验收标准

- 用户可以在 Web 控制台配置并运行一个包含条件分支的工作流。
- 现有 A-F 点击序列仍可运行，且多轮仍严格串行。
- 工作流运行时可以读取上下文字段，并基于 `If_Else` 和 `Visual_Branch` 选择下一个节点。
- 视觉判断无结果或低置信度时默认挂起，不能直接进入可能点击的默认路径。
- 连接期证书信任自动重试有明确上限，超过上限后不再无限 Loading。
- 工作流 DSL validator 能拒绝闭环、缺失引用、不可达节点和非 `context.xxx` 表达式。
- 异常拦截器可以作为全局运行时能力参与执行。
- 控制台可以展示当前节点、上下文、设备截图、事件时间轴。
- 当前已具备工作流画布 MVP、节点边线、前端临时拖拽布局、节点属性面板、失败热力图 MVP、只读热力图分析 API 和受控 JSON 配置编辑；可视化新增 / 删除节点、拖拽改线和拖拽写回 DSL 仍需后续补齐。
- 至少一个视觉判断节点能参与实际分支。
- 失败后可以从时间轴定位到节点、截图、日志和耗时。
- `npm run check` 通过，并补充必要的 dry-run / validator 检查。
- `npm run smoke:workflow` 通过。

### 4.5 不做事项

- 不做云端多租户。
- 不做远程设备农场。
- 不做多用户权限系统。
- 不做脱离本机 Mac + iPhone 的平台化扩张。

## 5. Cross-phase Constraints

- 任何阶段都不能破坏“先初始化/连接，再执行”的基本约束。
- 点击任务必须保持串行，直到多设备调度被单独设计并验收。
- 文档中的未来能力不能在 UI 中伪装成已完成能力。
- 新增状态必须能在日志或事件中追踪。
- 新增视觉能力必须能解释输入、输出和置信边界。
- 修改配置格式时必须提供兼容策略。

## 6. Recommended Build Order

按执行优先级排序：

1. 结构化运行事件
2. 控制台布局升级
3. 截图证据链
4. 设备视窗
5. 异常拦截器 MVP
6. 时间轴事件模型
7. 工作流节点模型
8. 上下文池与表达式读取
9. 条件分支
10. 视觉判断节点
11. 工作流画布
12. 热力图与分析

## 7. Completion Definition

三个阶段全部完成时，项目应满足：

- Web 控制台是主操作入口。
- CLI 是兼容入口而不是主要体验。
- 自动化动作不再只是一串坐标，而是带状态、证据和上下文的节点。
- 屏幕截图和视觉判断参与执行决策。
- 异常阻断可以被识别、挂起或自愈。
- 时间轴可以解释每一次运行。
- 文档、配置和实现保持一致。
