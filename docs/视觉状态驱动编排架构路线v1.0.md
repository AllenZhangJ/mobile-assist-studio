# 视觉状态驱动编排架构路线v1.0

## 0. TL;DR

- 本项目未来要从“坐标序列执行器”演进为“视觉状态驱动的 iOS 辅助自动化编排器”。
- 演进路线应分层推进：核心驱动、运行时状态机、工作流编译、视觉理解、控制台表现。
- 当前代码不需要立刻重排目录；新能力进入前应先判断落在哪一层，避免继续把连接、执行、看板和视觉逻辑混在一起。

## 1. Architecture North Star

长期目标是形成五层结构：

| 层级 | 职责 | 典型能力 |
|---|---|---|
| Core Driver | 原子设备能力 | 连接、点击、滑动、输入、截图、WDA 请求 |
| Runtime | 执行状态机 | 队列、停止、重试、上下文、异常挂起 |
| Compiler | 工作流编译 | 节点图解析、条件分支、合法性校验 |
| Vision | 屏幕状态理解 | OCR、CV、模板匹配、弹窗识别 |
| Console | 本地控制台 | API、SSE、日志、时间轴、画布 |

这五层是演进方向，不代表当前必须立刻按目录完成拆分。

## 2. Current State

当前项目已经具备：

- USB 真机初始化
- Appium / WDA 连接
- WebDriver 常驻会话
- A-F 坐标点击序列
- Web 看板控制入口
- 实时日志
- CLI 兼容入口
- 本地 evidence store
- 动作前后截图证据
- 轻量视觉守卫和高置信系统弹窗挂起

当前主要约束：

- 执行配置仍以扁平序列为主
- 视觉理解刚进入主执行链路，但还只是守卫和证据层，不做自动处理动作
- 时间轴、快照和失败热力图已经具备基础数据模型：运行事件来自 `runEvents`，截图证据来自本地 evidence store，热力图从失败坐标或失败节点聚合
- 看板是控制台雏形，还不是完整工作流工作台

## 3. Workflow Model Direction

未来的执行单元不再只是坐标，而是节点。

节点应包含：

- 动作类型
- 输入参数
- 前置条件
- 成功出口
- 失败出口
- 输出映射
- 运行证据

节点之间通过执行结果流转。

最小可行路径：

- 保留现有 A-F 序列作为兼容节点组
- 在运行时内部把 A-F 步骤视为节点事件
- 先用 DSL validator 校验条件分支和变量读取
- 后续再把条件分支接入完整运行时

## 4. Context And Expression

动态上下文用于让节点之间传递信息。

上下文可以承载：

- 当前设备状态
- 当前截图摘要
- OCR 结果
- 最近点击坐标
- 运行计数
- 异常拦截结果
- 用户配置参数

表达式能力应谨慎引入：

- 第一阶段只支持读取上下文字段
- 第二阶段支持简单条件判断
- 第三阶段再考虑函数、转换和默认值

不要在早期引入过强的脚本执行能力，避免安全和调试成本失控。

当前落地状态：

- 已有 `legacy/node/src/ios-assist-workflow.mjs` 承载自定义 DSL。
- 旧 `sequence` 会自动映射为线性 `Tap` / `Wait` 节点树。
- validator 已覆盖节点类型、引用、闭环、不可达节点和 `context.xxx` 表达式白名单。
- 当前执行器已支持基础节点图运行：`Tap`、`Wait`、`Snapshot`、`Visual_Branch`、`If_Else`、`Catch`。
- `Catch` 已支持显式 `onError` 错误分支；`Sub_Workflow` 保留类型和 hook 边界，完整子流程调度和画布编辑仍属于后续工作。
- Web 控制台已具备工作流画布 MVP、节点边线、前端临时拖拽布局、节点属性查看、失败热力图 MVP、受控 JSON 编辑、校验、保存和清除自定义 workflow；拖拽改线、拖拽写回 DSL 和可视化节点新增删除仍未开放。

## 5. Global Exception Interceptor

全局异常拦截器是未来稳定性的关键能力。

它的职责：

- 在主流程动作前后轻量检查屏幕
- 识别常见弹窗和阻断态
- 暂停主流程指针
- 执行清除动作
- 再次截图确认恢复
- 将拦截过程写入时间轴

典型拦截对象：

- 系统权限确认
- 低电量提示
- 系统更新提示
- 第三方弹窗
- App 内确认弹窗
- 网络异常遮罩

设计原则：

- 拦截器不能无限点击
- 未识别的状态必须进入挂起态
- 每次自愈都要留下证据
- 拦截规则应可配置、可禁用、可回放

## 6. Visual Layer

视觉层的职责是把截图转成可用状态。

能力路线：

- 截图采集
- 区域裁剪
- OCR 文本识别
- 模板匹配
- 颜色/形状判断
- 弹窗分类
- 目标可点击性判断

视觉层输出不应直接绕过运行时下发动作。它应先产出判断结果，再由运行时根据状态机决定下一步。

## 7. Timeline And Analytics

时间轴是自动化可信度的基础。

每次动作应逐步沉淀：

- 开始时间
- 结束时间
- 动作类型
- 输入摘要
- 输出摘要
- 成功或失败
- 耗时
- 关联截图

后续分析指标：

- WDA 往返耗时
- UI 稳定等待耗时
- 单节点失败率
- 坐标失败热力
- 异常拦截次数
- 会话重连次数

## 8. Refactor Roadmap

### 8.1 Phase 1: Runtime Events

目标：

- 不大改现有行为
- 给每个 A-F 步骤生成结构化事件
- 让日志、状态和未来时间轴共用事件模型

默认落点：

- `legacy/node/src/ios-assist-runner.mjs`
- `legacy/node/src/ios-assist-console.mjs`

### 8.2 Phase 2: Screenshot Evidence

目标：

- 在关键动作前后采集截图
- 看板展示当前截图和最近动作
- 失败时保留现场

默认落点：

- `legacy/node/src/ios-assist-session.mjs`
- `legacy/node/src/ios-assist-runner.mjs`
- `legacy/node/src/ios-assist-console.mjs`

### 8.3 Phase 3: Interceptor MVP

目标：

- 加入可配置的弹窗识别和清除规则
- 拦截过程写入日志和事件
- 未识别状态进入挂起态

默认落点：

- 新增运行时拦截模块
- Runner 调用前后插入轻量检查

当前落地状态：

- 已有 `legacy/node/src/ios-assist-visual.mjs` 作为轻量视觉守卫。
- 已支持 Tap 前截图分析、Tap 后截图证据、手动分析当前画面。
- 已知 iOS 系统弹窗高置信命中时，runner 进入 `paused`，等待人工处理后从看板继续。
- 当前不自动点击处理弹窗；后续自愈动作必须通过运行时限次重试和 evidence 记录。

### 8.4 已并入 V2.0 Phase 3: Workflow Nodes

目标：

- 把扁平序列抽象为节点执行
- 支持最小条件分支
- 保持 CLI 兼容
- 在 V2.0 三阶段计划中，本能力归入完整 Workflow 画布阶段，不再作为独立第四阶段

默认落点：

- 新增工作流解析与运行时模块
- 配置层增加新旧格式兼容策略

### 8.5 已并入 V2.0 Phase 3: Visual Studio Console

目标：

- 建立三栏工作台
- 引入节点画布
- 引入时间轴回放
- 引入设备视窗
- 在 V2.0 三阶段计划中，本能力归入完整 Enterprise Workstation 阶段，不再作为独立第五阶段

默认落点：

- Flutter Desktop 工作区结构
- Dart Runtime 事件模型
- Workflow Canvas、Inspector、Timeline 和 Device View

## 9. Change Rules

- 不因未来目标一次性重构全部目录。
- 新增能力前先判断所属层级。
- 连接稳定性优先于视觉自动化。
- 视觉判断结果必须可观察、可复核。
- CLI 兼容入口在明确迁移完成前继续保留。
