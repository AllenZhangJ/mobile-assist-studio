# Web控制台UIUX规范v1.0

## 0. TL;DR

- Web 控制台的长期视觉方向是“Tech Noir / Cyber Grid”的本地真机自动化工作台。
- 当前阶段优先保证可用性、可读性和状态清晰；视觉强化必须服务排障和编排，不做装饰性堆叠。
- 本文是 UI/UX 定调，不替代实现代码和 CSS 细节。

## 1. Design Language

控制台应呈现为专业、冷静、精密的本地自动化驾驶舱。

推荐视觉关键词：

- 深色工作台
- 赛博网格
- 高对比状态光
- 低噪声信息密度
- 精准线框
- 实时系统感

避免的视觉方向：

- 营销型首页
- 大面积装饰卡片
- 过度渐变背景
- 与调试工作无关的视觉噱头
- 单一紫蓝色调

## 2. Color Semantics

颜色必须绑定状态语义，不能只作为装饰。

| 语义 | 推荐方向 | 用途 |
|---|---|---|
| Background | 黑色或极深灰 | 主工作台背景 |
| Grid / Border | 深灰网格线 | 对齐、分区、设备视窗背衬 |
| Running | 亮青、电光绿 | 正在连接、正在运行、当前节点 |
| Warning | 琥珀橙 | 等待、阻断、用户需要介入 |
| Error | 荧光红 | 失败、不可恢复错误、异常节点 |
| Neutral | 灰白层级 | 标签、次要信息、说明文字 |

状态色出现时必须有清晰的对象：

- 连接状态
- 当前执行节点
- 日志级别
- 错误区域
- 时间轴事件

## 3. Typography

推荐字体策略：

- UI 文案使用现代无衬线字体
- 日志、坐标、UDID 摘要、WDA 报文和时间戳使用等宽字体
- 标题克制，避免营销式大字
- 控制面板和日志区域优先保证扫描效率

文字原则：

- 按钮文案短而明确
- 状态文案使用用户能理解的动作词
- 错误提示先给原因，再给下一步
- 不在界面上堆解释性长文

## 4. Core Workspace Layout

旗舰版控制台的目标布局是三栏加底部时间轴：

| 区域 | 职责 | 当前阶段可落地内容 | 长期内容 |
|---|---|---|---|
| 顶部状态栏 | 全局状态 | 连接、运行、Appium、配置摘要 | 多设备、系统负载、会话守护 |
| 左栏设备视窗 | 当前 iPhone 画面 | 截图、拾点、坐标标注 | 实时推流、OCR 框选、CV 裁剪 |
| 中栏编排画布 | 执行流 | 工作流画布 MVP、A-F 模板节点、边线和前端临时拖拽布局 | 节点新增删除、拖拽改线、分支、子流程 |
| 右栏属性面板 | 当前选择项 | 坐标、等待、运行参数 | 节点属性、变量池、输出映射 |
| 底部时间轴 | 执行证据 | 实时日志、失败热力图 MVP | 快照回放、WDA 报文、失败定位 |

当前 Web 看板已经具备画布 MVP，但前端拖拽只用于整理视图，不改变 DSL 或运行真值。

## 5. Interaction States

连接状态与运行状态必须分离展示。

连接状态：

- 未连接
- 初始化中
- 连接中
- 等待信任证书
- 已连接
- 断开中
- 错误

运行状态：

- 空闲
- 运行中
- 停止中

交互规则：

- 未连接时不能执行点击任务
- 等待信任证书时显示明确介入提示
- 运行中禁用会破坏当前任务的配置操作
- 停止动作要显眼，但不要和错误态混淆
- 日志操作需要支持复制全部和清空全部
- 失败热力图只消费本地运行事件，不发设备命令，不改 workflow 配置

## 6. Digital Twin Screen

设备视窗是未来控制台的核心，不只是截图展示区。

它应承载：

- 当前屏幕快照
- 点击坐标标注
- 最近动作涟漪提示
- 拾点模式
- OCR 选区
- CV 特征裁剪
- 失败现场回放

落地顺序：

- 先支持单帧截图
- 再支持点击标注
- 再支持拾点和框选
- 最后再考虑高频推流

## 7. Workflow Canvas

工作流画布的长期目标是替代扁平序列编辑。

节点类型应围绕自动化动作和状态判断设计：

- 连接预检
- 点击
- 等待
- 截图
- OCR
- CV 判断
- 条件分支
- 异常处理
- 子流程

运行时表现：

- 当前节点应有明确高亮
- 成功路径和失败路径颜色不同
- 失败节点停留在现场
- 用户能从节点跳到对应日志和截图

当前阶段的画布边界：

- 先提供工作流画布 MVP、节点属性查看和受控 JSON 配置面板。
- 拖拽布局在当前阶段只作为前端视图状态，不写回 workflow 配置。
- 节点可点击查看类型、参数和出口。
- 当前执行节点应高亮。
- 提供加载、校验、保存和清除自定义 workflow 的入口。
- 暂不提供拖拽编辑或可视化节点新增删除。
- UI 不应把当前过渡形态包装成完整低代码编辑器。

## 8. Timeline

时间轴用于回答三个问题：

- 刚才执行到哪里
- 为什么失败
- 失败现场是什么样

事件至少应包含：

- 时间
- 节点或步骤
- 动作类型
- 结果
- 耗时
- 相关截图或日志定位

后续可以扩展：

- WDA 请求摘要
- Appium 会话事件
- 坐标热力
- 界面稳定等待耗时

## 9. UX Guardrails

- 界面要帮助用户更快定位问题，而不是制造更多信息噪声。
- 状态灯、动画和网格必须轻量，不能影响长时间调试。
- 卡片数量要克制，布局优先像工具台而不是展示页。
- 日志默认可读，错误默认可复制。
- 移动端适配可以降级为纵向分区，但不能丢失关键操作。

## 10. Implementation Notes

- 本文只定义体验方向，不要求一次性重写前端。
- 任何 UI 改动都应保持现有核心能力可用。
- 新增视觉元素前，应先确认它对应的状态语义或排障价值。
