# iOS Assist Studio V3.0 Development Plan

## 0. TL;DR

- V3.0 分阶段从 iOS-only 工作站演进为单人单设备跨平台移动自动化工作站。
- 阶段顺序是：文档和决策、跨平台抽象、目标库、Android 最小闭环、目标解析增强、结果和监控成熟化。
- 每个阶段必须保持 iOS 现有能力可用，不允许为了 Android 支持破坏 V2.0 主链路。
- 每个阶段都必须有 fake tests 和文档验收，不依赖真实设备作为唯一验证。
- V3.0 不引入多设备并发、云端平台或团队测试管理。

## 1. Planning Principles

V3.0 开发遵循：

- Embrace change：允许 Runtime、DSL、UI 和包边界为跨平台目标重构。
- Preserve safety：保留单设备、串行执行、安全停止、本地隐私和 Project DSL 真源。
- Competitor-informed：吸收 TestHub 类竞品解决过的痛点，不复制其平台形态。
- Adapter-first：先抽平台 adapter，再接 Android。
- Target-first：先引入 Target Library，再让复杂视觉和 OCR 进入执行。
- Evidence-first：每个自动化判断都要留下可解释证据。
- Fake-testable：无真实设备时仍能验证模型、状态机、validator 和 UI。

## 3. Phase 1: Platform And Driver Abstraction

### 3.1 Goal

在不改变 iOS 用户体验的前提下，把 Runtime 从 iOS-only 改造成平台中立运行时。

### 3.2 Deliverables

- `MobilePlatform` 模型
- `MobileDevice` 摘要模型
- `DeviceResourceState` / 当前设备资源锁模型
- `MobileDriverCapabilityReport`
- `MobileDeviceDriver` 接口
- iOS adapter 包装现有 Appium / WDA 能力
- Runtime snapshot 暴露 platform、capability 和 resource state
- Device / Execute / Recorder 共享资源锁判断
- fake driver test harness

### 3.3 Not Included

- 不接真实 Android。
- 不接 scrcpy。
- 不做 Target Library UI。
- 不改变现有 iOS 工作流语义。

### 3.4 Acceptance

- iOS 现有连接、截图、预览动作、录制、运行和 Monitor 测试通过。
- Runtime tests 可用 fake driver 跑通 connect / screenshot / tap / run。
- Device、Recorder、Execute 不再各自复制运行中禁用逻辑。
- 运行中 workflow 和 target 写入入口被资源锁阻断。
- V2 boundary check 继续通过。

## 4. Phase 2: Target Library Foundation

### 4.1 Goal

引入本地 Target Library，让工作流从坐标驱动逐步升级为目标驱动。

### 4.2 Deliverables

- Target model
- Target JSON store
- Target validator
- Coordinate Target
- Image Target metadata
- Region Target metadata
- Selector Target metadata
- Text / OCR Target metadata placeholder
- Target Inspector MVP
- 从 Device / Recorder 截图创建 coordinate target
- Recorder 动作可绑定 targetRef
- Workflow Inspector 可选择 targetRef
- Workflow validator 检查缺失 target

### 4.3 Not Included

- 不做完整 OCR。
- 不做真实图像匹配。
- 不做复杂 Target Library 独立导航。
- 不要求旧 workflow 全量迁移。

### 4.4 Acceptance

- 用户可从当前截图创建坐标目标。
- 用户可在 Workflow 节点选择目标。
- 缺失目标会在 Source / Validate / Inspector / Execute Preflight 提前暴露。
- 旧坐标 Tap 节点仍可运行。
- Target Store 无效文件不污染 Runtime snapshot。
- Target 删除会保护已有 workflow 引用。

## 5. Phase 3: Android Minimum Execution Loop

### 5.1 Goal

接入 Android 当前设备，实现跨平台最小执行闭环。

### 5.2 Deliverables

- Android adapter MVP
- ADB 当前设备发现
- Appium UiAutomator2 status / session
- Android screenshot
- Android tap / swipe / input
- Android app launch / stop
- Android logcat 摘要
- Android capability report
- Device 页面平台状态展示
- Execute Preflight 平台能力检查
- fake ADB / fake Appium tests

### 5.3 Not Included

- 不做 Android 多设备列表。
- 不做公网远控。
- 不做 scrcpy 流式投屏首发必选。
- 不做 Airtest 首发必选。
- 不做 APK 管理中心。

### 5.4 Acceptance

- 当前只连接一台 Android 时可建立 session。
- Android 可截图、点击、滑动、输入。
- Android 可运行基础 Tap / Wait / Swipe / Input workflow。
- Android 日志摘要可关联到 run detail。
- iOS 连接和执行不退化。
- 平台能力缺失时 UI 给出短中文提示。

## 6. Phase 4: Target Resolver, Vision And OCR

### 6.1 Goal

让目标库真正覆盖“不标准 APP”痛点，支持图像、区域和 OCR 辅助定位。

### 6.2 Deliverables

- Target Resolver contract
- Coordinate resolver
- Selector resolver
- Image template resolver
- Region resolver
- OCR text resolver
- confidence threshold
- low-confidence pause policy
- target resolution evidence
- fixture screenshot tests
- Wait For Target node
- Tap Target node
- Assert Target node
- Assert Text node

### 6.3 Not Included

- 不做云端 OCR。
- 不保存完整 OCR 原文作为长期证据。
- 不自动处理验证码或绕过安全机制。
- 不让视觉/OCR 直接绕过 Runtime 点击。

### 6.4 Acceptance

- Target Resolver 可在无设备 fixture 中测试。
- 低置信默认 paused。
- 目标找不到能生成 target issue。
- Tap Target 只在 resolver 返回可信位置后执行。
- Assert Text 可区分 assertion failed 和 infrastructure error。
- Run Detail 展示目标解析证据链。

## 7. Phase 5: Remote Control Upgrade

### 7.1 Goal

增强远控工作台，让远控、录制、目标创建和执行共享同一当前设备资源模型。

### 7.2 Deliverables

- iOS preview refresh UX polish
- Android scrcpy feasibility spike
- Android scrcpy optional remote preview
- remote control resource lock
- target capture overlay
- manual intervention from paused state
- remote action evidence summary

### 7.3 Not Included

- 不做公网远控。
- 不做多人远控。
- 不做设备共享链接。

### 7.4 Acceptance

- 远控动作和自动化执行不会并发下发。
- 运行暂停时用户可查看现场并收口。
- 从远控画面创建目标不触发设备动作。
- scrcpy 不可用时 Android 仍可通过截图刷新模式工作。

## 8. Phase 6: Result Model, Logs, Performance And Reports

### 8.1 Goal

让 Monitor 达到竞品成熟度：可解释任务状态、测试结果、问题归因、日志和性能。

### 8.2 Deliverables

- Run Status / Test Result / Issue Category 模型
- assertion result events
- Android logcat slices
- iOS syslog / device log summary candidate
- Android dumpsys performance samples
- iOS performance sampling candidate
- Run Detail target chain
- Run Detail log panel
- Run Detail performance panel
- local report export candidate

### 8.3 Not Included

- 不做 Allure 强依赖。
- 不做云端报告。
- 不做团队测试计划。
- 不做 CI 调度系统。

### 8.4 Acceptance

- 设备问题不被算作断言失败。
- 用户停止显示 stopped。
- 低置信暂停显示 paused / inconclusive。
- 断言失败显示 assertion failed。
- 运行详情能展示日志、性能和目标解析摘要。
- 复制诊断摘要不包含完整设备标识、路径、session 或 endpoint。

## 9. Verification Matrix

V3.0 常规门禁：

- `fvm dart run tool/v2_boundary_check.dart`
- `fvm dart run melos run analyze`
- `fvm dart run melos run test`
- `fvm dart run tool/macos_build_smoke.dart`
- `fvm dart run tool/v2_verify.dart`

V3.0 新增建议门禁：

- mobile driver fake tests
- Android adapter fake tests
- Target Library validator tests
- Target Resolver fixture tests
- resource lock runtime tests
- result model aggregation tests
- platform UI widget tests

命名可在实现阶段收口，但每个阶段必须有本地 fake 验证。

## 10. Documentation Triggers

以下改动必须同步 V3 文档：

- 新增或改变平台 adapter
- 新增 Target 类型
- 改变 Target Resolver 策略
- 改变 Workflow DSL 节点语义
- 改变 Run Status / Test Result / Issue Category
- 引入 scrcpy、Airtest、OCR 或 CV 依赖
- 改变隐私保存策略
- 改变单设备边界

## 11. Stop Conditions

出现以下情况时必须暂停实现并回到文档/ADR：

- Android 能力要求引入长期 Node 中间层。
- 目标解析想绕过 Runtime 直接点击。
- 需求要求多台设备并发。
- 需求要求云端上传截图或日志。
- 需求要求保存账号、密码、完整设备标识或证书主体。
- 需求要求把 Airtest / scrcpy 变成所有平台主执行链路。

## 12. Completion Criteria

V3.0 完成时应满足：

- 用户可连接一台 iOS 或 Android 设备。
- 用户可在 Device 中远控/预览并创建目标。
- 用户可在 Recorder 中生成 Target + Action 时间轴。
- 用户可在 Workflow 中编排目标驱动流程。
- 用户可在 Execute 中安全运行、暂停介入和停止。
- 用户可在 Monitor 中看懂任务状态、测试结果和问题归因。
- iOS 既有主链路稳定。
- Android 最小主链路可用。
- 所有核心数据本地保存。
- 没有多设备并发、云端依赖或团队平台功能进入主路径。
