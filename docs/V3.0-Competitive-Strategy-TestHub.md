# iOS Assist Studio V3.0 Competitive Strategy: TestHub-Inspired Local Mobile Workstation

## 0. TL;DR

- V3.0 以 TestHub APP 自动化测试模块作为主要竞品参考之一，但不复制其团队平台、设备池、SaaS 或测试管理系统定位。
- TestHub 解决的关键痛点是：设备资源冲突、不标准 APP 元素定位、远控与自动化共存、执行过程可观测、任务状态与测试结果分离。
- V3.0 要吸收这些痛点解法，并转化为本项目的单人、单台设备、本地桌面工作站能力。
- V3.0 的竞争策略是：用更强的本地体验、更安全的执行边界、更清晰的跨平台驱动抽象，覆盖移动自动化从连接到复盘的闭环。
- 竞品是痛点校准器，不是路线图复制件。

## 2. What TestHub Actually Solves

### 2.1 Device As Resource

TestHub 的关键判断是：设备不是一段配置，而是会被远控、自动化、用户操作和异常状态共同争用的资源。

它对应的真实痛点：

- 设备在线但被别人占用
- 远控正在使用，自动化又要执行
- 自动化异常退出后设备锁不释放
- 心跳失效后设备状态不可信
- 多个入口对同一设备发命令，导致状态错乱

V3.0 不做多人设备池，但必须把“当前设备资源锁”产品化。

### 2.2 Scenario As Visual Flow

TestHub 的另一个判断是：测试场景应该可以被编排出来，而不是只靠脚本写出来。

它对应的真实痛点：

- QA 能描述人工流程，但不会写 Appium / Airtest 脚本
- 业务流程需要复用、调整和回归
- 动作、元素和等待策略散落在脚本里后难维护
- 失败时无法定位到业务步骤

V3.0 已有 Project DSL 和 Workflow Canvas，应继续增强为跨平台、目标驱动的流程编排，而不是退回脚本平台。

### 2.3 Remote Control And Automation Share One System

TestHub 把投屏远控和自动化执行放进同一套设备体系。

它对应的真实痛点：

- 录制前需要人工看屏幕、点屏幕、建元素
- 自动化失败后需要人工介入验证现场
- 远控操作和自动化操作如果不共享锁，会互相打架
- 调试和正式执行如果是两套工具，用户认知成本高

V3.0 的 Device Preview、Recorder、Workflow 和 Execute 必须共用同一套设备资源锁、连接状态和运行状态。

### 2.4 Execution Status Is Not Test Result

TestHub 明确区分任务执行状态和测试结果。

它对应的真实痛点：

- 设备断线不是用例断言失败
- 驱动异常不是业务失败
- 用例失败不等于平台任务失败
- 用户需要知道下一步是修设备、修流程、还是修业务断言

V3.0 Monitor 和 Run Detail 必须把基础设施问题、流程问题、视觉问题、断言问题分开表达。

### 2.5 Non-Standard APPs Need Multiple Target Strategies

TestHub 组合图片元素、坐标元素、区域元素、OCR 和图像处理。

它对应的真实痛点：

- 很多移动 APP 没有稳定可读的控件树
- 同一页面在不同设备尺寸上布局变化
- 文本、图标、弹窗、验证码、列表结果需要不同识别方式
- 只有坐标会脆，只有 accessibility 又不够通用

V3.0 需要 Target Library，把坐标、图片、区域、文本、selector 和视觉判断统一成用户可管理的目标资产。

## 3. What We Should Adopt

V3.0 应吸收以下竞品启发：

| 竞品解法 | V3.0 转化方式 |
|---|---|
| 设备资源化 | 单人单设备资源锁、远控/录制/执行互斥、心跳和超时释放 |
| 元素管理 | Target Library：图片、坐标、区域、文本、selector、视觉目标 |
| 拖拽编排 | 继续强化 Project DSL + Canvas + Inspector |
| 远控和自动化共用设备体系 | Device Preview、Recorder、Execute 共用同一 Runtime 状态机 |
| 图像识别 + OCR + 坐标 | 作为 Target Resolver 和 Visual Evidence 的辅助能力 |
| 执行和报告分离 | Run status、test result、issue category 三层模型 |
| 性能和日志 | 本地平台适配采集，并关联到运行证据 |
| 成熟技术栈 | 优先使用 Appium、ADB、scrcpy、OCR/CV 等已验证能力，不盲目自研底层 |

## 4. What We Should Not Copy

V3.0 明确不复制：

- 不做 SaaS 平台
- 不做多用户权限系统
- 不做审计系统
- 不做团队设备池
- 不做多设备并发
- 不做云端报告中心
- 不做完整测试管理平台
- 不把 pytest / Allure 变成本项目的核心执行依赖
- 不让 Airtest 成为 iOS 主驱动链路
- 不为了 Android 能力破坏现有 iOS / WDA 安全边界

## 5. Strategic Positioning

TestHub 更像：

```text
Team Test Management Platform
  -> Device Management
  -> Element Management
  -> Case Orchestration
  -> Remote Control
  -> Scheduled Runs
  -> Reports
```

V3.0 应该成为：

```text
Local Cross-Platform Mobile Automation Workstation
  -> One User
  -> One Connected Device
  -> iOS or Android
  -> Target Library
  -> Visual Workflow
  -> Safe Execution
  -> Local Evidence
  -> Local Diagnosis
```

核心差异：

- TestHub 的优势是平台半径。
- V3.0 的优势应该是本地体验、低配置成本、设备现场感、安全边界和可诊断执行。

## 6. Competitive Gap Map

| 能力 | V2.0 当前状态 | V3.0 目标 |
|---|---|---|
| iOS 真机链路 | 强，Appium / WDA 主链路清晰 | 保持为 iOS adapter |
| Android 支持 | 缺失 | 新增 Android adapter |
| 设备资源锁 | 有连接/运行状态，但未产品化为资源模型 | 当前设备资源锁，一切入口共享 |
| 远控 | Device Preview 基线 | 跨平台远控工作台，支持创建目标 |
| 元素/目标管理 | 以动作和 DSL 节点为主 | Target Library 成为一等能力 |
| 图像识别 | Visual Branch 和轻量证据链 | 模板匹配、OCR、区域识别作为辅助定位 |
| OCR | 未成体系 | 文本目标、断言、等待和证据摘要可用 |
| App 生命周期 | iOS 侧能力散在运行链路 | iOS / Android 统一 Launch / Stop / App Profile |
| 日志 | Runtime event 和本地 evidence | Android logcat、iOS syslog 摘要、运行关联 |
| 性能 | Monitor 偏运行统计 | 设备/App 性能采样进入 Run Detail |
| 结果模型 | 有运行状态和失败分析 | 任务状态、用例结果、问题类型分离 |
| 测试管理 | 非目标 | 仍非目标，只保留本地流程与证据 |

## 7. Product Principles For V3.0

### 7.1 Embrace Change Without Losing The Core

V3.0 不能只保护现有 iOS 资产，也不能盲目推倒重来。

必须保留：

- 本地桌面工作站
- 单人单设备
- 串行执行
- 安全停止
- Project DSL 真源
- Runtime 统一状态机
- 本地隐私边界

必须改变：

- 平台从 iOS only 变为 iOS / Android
- 设备从连接对象变为当前资源
- 坐标动作从主资产变为 Target 引用的一种实现
- 视觉从分支判断扩展为目标定位和证据能力
- Monitor 从运行摘要扩展为问题归因和测试结果表达

### 7.2 Cross-Platform Means Platform Adapters, Not Lowest Common Denominator

跨平台不代表只保留 iOS 和 Android 都有的最小能力。

正确方式：

- Workflow DSL 表达用户意图
- Runtime 定义平台中立动作和目标解析
- iOS adapter 使用 Appium / WDA / XCUITest
- Android adapter 使用 Appium UiAutomator2、ADB 和可选 scrcpy / Airtest 辅助
- 平台差异进入 adapter 和 capability report

### 7.3 Target First, Coordinate Second

用户思考的是“点击打车按钮”，不是“点击横 120 纵 480”。

V3.0 应让坐标成为目标的一种实现细节：

- 坐标目标适合快速录制和兜底
- 图片目标适合不稳定控件树
- selector 目标适合结构化页面
- OCR 文本目标适合动态内容
- 区域目标适合列表、卡片和局部识别

### 7.4 Evidence Before Automation Confidence

图像识别、OCR 和远控能力会提高覆盖面，但也会带来误判风险。

V3.0 的默认策略：

- 低置信进入 paused
- 不盲点
- 不自动处理系统安全限制
- 每次视觉判断记录证据摘要
- 用户可以在现场确认后收口

## 8. Strategic Non-Goals

V3.0 不是 TestHub 的轻量克隆。

V3.0 不做：

- 团队项目管理
- 测试计划管理
- 多人协作
- 设备农场
- 公网远控平台
- 分布式调度
- 云端 AI 测试平台
- 通用低代码平台

这些能力可以作为未来平台化方向讨论，但不得进入 V3.0 主路径。

## 9. Success Criteria

V3.0 竞品吸收成功的判断标准：

- 用户能连接一台 iOS 或 Android 设备。
- 用户能在同一个工作台远控、截图、创建目标。
- 用户能把人工流程转成 Target + Action 的 Workflow。
- Workflow 可本地校验、执行、暂停、恢复和安全停止。
- 运行结果能区分任务状态、测试结果和问题类型。
- 失败详情包含目标、节点、截图、日志、性能或视觉证据。
- 所有数据保存在本机。
- 没有引入多用户、多设备并发或云端依赖。
