# iOS Assist Studio V2.0 IA: Information Architecture

## 0. TL;DR

- V2.0 采用 L1-L6 主导航：Dashboard、Device、Recorder、Workflow、Execute、Monitor。
- 全局框架由 Top Status Bar、Side Nav、Workspace Area、Bottom Console 组成。
- L7 Fleet 只做未来 IA 预留，不进入 V2.0 运行范围。
- 所有信息架构遵循 Summary First, Detail Later。
- 技术细节默认隐藏到 Drawer、Inspector、Advanced 或 Bottom Console。

## 1. IA North Star

V2.0 的信息架构要把项目从“工程调试页面”升级为“本地自动化工作站”。

核心组织方式：

- Dashboard 负责总览
- Device 负责设备
- Recorder 负责录制
- Workflow 负责编排
- Execute 负责运行
- Monitor 负责复盘

这六个模块共同覆盖用户从连接设备到复盘失败的完整闭环。

## 2. Global Shell

### 2.1 Top Status Bar

顶部状态栏固定高度为 56px。

职责：

- 展示设备状态
- 展示 WDA 状态
- 展示工作流状态
- 展示执行状态
- 承载用户菜单和全局入口

状态语义：

- Ready：绿色
- Warning：黄色
- Error：红色
- Offline：灰色

交互：

- 状态栏只展示摘要
- 点击任一状态打开详情 Drawer
- Drawer 中展示诊断、建议操作、原始摘要和高级信息

禁止：

- 在状态栏堆叠 endpoint、完整设备标识、XML、长日志
- 把错误堆成多行技术文本

### 2.2 Side Nav

左侧导航是一级模块入口。

形态：

- 收起宽度：72px
- 收起态：Icon
- 展开态：Icon + Label
- 当前页面：左侧高亮条
- Hover：背景高亮

正式导航：

- L1 Dashboard
- L2 Device
- L3 Recorder
- L4 Workflow
- L5 Execute
- L6 Monitor

辅助入口：

- Settings
- Help
- About

辅助入口不进入主流程层级。

当前 Flutter Desktop 已落地 Settings：

- Settings 位于 Side Nav 底部。
- 以右侧 Drawer 打开，不切换 L1-L6 主工作区。
- 只承载本地工作站设置摘要、运行时摘要、隐私摘要和边界摘要。
- 不承载账号、权限、多租户、云端协作或远程设备池。

### 2.3 Workspace Area

Workspace Area 是当前模块的主工作区。

原则：

- 不做营销页
- 不使用大段说明文占据主视图
- 控制台类页面优先信息密度和任务完成
- 技术细节放入 Drawer、Inspector 或 Bottom Console

### 2.4 Bottom Console

Bottom Console 是全局常驻组件。

形态：

- 默认收起高度：48px
- 展开高度：320px
- 可折叠
- 不抢占主任务区域

内容分区：

- Log
- Error
- Inspector
- Network
- Debug

原则：

- 主界面不大量铺设日志
- Console 承载细节和调试信息
- 日志支持复制全部和清除全部
- 日志与公共错误必须脱敏

## 3. L1 Dashboard

### 3.1 Goal

Dashboard 的目标是让用户打开产品后第一眼知道目前情况。

### 3.2 Content Model

Dashboard 包含：

- 欢迎与当前工作区摘要
- KPI Cards
- Recent Workflows
- Activity Trend

KPI：

- 连接设备
- 工作流数量
- 今日执行次数
- 成功率

### 3.3 Interaction

- 点击设备 KPI 跳转 Device
- 点击工作流 KPI 跳转 Workflow
- 点击执行 KPI 跳转 Execute 或 Monitor
- 点击成功率 KPI 跳转 Monitor
- 最近工作流当前支持查看只读摘要、打开 Workflow、进入 Execute
- 最近工作流当前支持收藏、复制和删除当前流程；删除会回到 A-F 基础模板
- 收藏、复制和删除只作用于本机 Project DSL / Settings，不触发设备动作

### 3.4 Boundary

Dashboard 不承载：

- 设备连接细节
- 工作流源码编辑
- 长日志
- 失败截图详情

这些信息进入对应模块或 Drawer。

## 4. L2 Device

### 4.1 Goal

Device 是设备管理中心。

### 4.2 Content Model

在单设备模式下，页面直接展示当前设备。

可展示信息：

- 设备名称摘要
- 设备标识摘要
- 电量
- 系统版本
- 连接方式
- WDA 状态
- 当前预览

### 4.3 Preview Area

实时预览占工作区主要区域。

交互能力：

- 点击
- 双击
- 拖动
- 滚动
- 缩放
- 截图刷新

当前 Flutter Desktop 已落地点击和拖动基线：在设备 connected、运行 idle 且有当前截图时，点击预览会通过 Runtime 换算为 WDA viewport tap，拖动预览会换算为 WDA viewport swipe；双击、滚动和缩放仍作为后续 Device 视窗增强。

反馈：

- 十字准星
- Ripple Effect
- 最近动作标记

### 4.4 Device List Boundary

V2.0 保持 Single Device First。

Device List 只作为 IA 预留：

- 单设备模式自动隐藏或弱化
- 不实现多设备调度
- 不实现多设备并发
- 不实现设备集群管理

## 5. L3 Recorder

### 5.1 Goal

Recorder 是录制用户行为的核心页面。

### 5.2 Content Model

页面结构：

- 录制控制区
- 手机预览区
- 动作时间轴

录制状态：

- Idle
- Recording
- Paused
- Stopped

### 5.3 Action Capture

动作以用户可理解方式展示：

- Tap 登录
- Wait 500ms
- Swipe Up
- Tap 商品

默认不展示：

- X 坐标
- Y 坐标
- WDA endpoint
- pageSource

### 5.4 Action Detail Drawer

点击动作后打开 Drawer。

Drawer 可展示：

- 坐标
- 截图
- 元素信息
- 等待时间
- 重试策略
- 高级诊断

### 5.5 Boundary

- V2.0 优先保证录制成功率
- Web 预览交互是主要录制入口
- 不承诺被动监听用户在真机上的物理触摸
- 高级信息默认隐藏

## 6. L4 Workflow

### 6.1 Goal

Workflow 是旗舰模块，用于可视化编辑流程。

### 6.2 Content Model

页面核心是全屏画布。

主要区域：

- 画布 / 源码 / 检查切换
- 节点工具区
- 工作流画布
- 右侧 Node Inspector
- 小地图
- 保存、校验、测试入口

### 6.3 Node Model

V2.0 节点方向：

- Start
- Tap
- Wait
- Swipe
- Input
- Snapshot
- Condition
- Visual Branch
- Loop
- Catch
- Sub Workflow
- End

旧 A-F 坐标序列作为基础模板保留，可导入或包装为节点树。

### 6.4 画布视图

画布视图支持：

- 拖拽
- 缩放
- 框选
- 复制
- 删除
- 自动布局
- 节点选中
- 节点状态展示

节点状态：

- 正常：蓝色
- 执行中：绿色脉冲
- 失败：红色
- 警告：黄色

### 6.5 源码视图

源码视图展示 Project DSL。

原则：

- 与画布视图映射同一真源
- 保存前必须校验和规范化
- 不允许任意脚本执行
- 无效 DSL 不应污染运行配置

## 7. L5 Execute

### 7.1 Goal

Execute 用于运行工作流。

### 7.2 Content Model

页面结构：

- 左侧执行配置
- 右侧执行摘要

执行配置：

- 执行模式
- 循环次数
- 随机延迟
- 超时时间
- 失败重试
- 执行设备

执行模式：

- 单次
- 循环
- 受控无限循环

### 7.3 Running Mode

开始执行后进入 Running Mode。

展示：

- 顶部进度条
- 当前节点
- 总节点
- 预计剩余时间
- 当前运行状态
- 安全停止入口

### 7.4 Boundary

- 执行前需要确认
- 运行期间不能写 workflow
- 运行期间不能断开或重连
- 停止采用安全停止
- 异常超过上限进入挂起态

## 8. L6 Monitor

### 8.1 Goal

Monitor 是本地运营监控和问题分析中心。

### 8.2 Content Model

第一层：指标

- 成功率
- 失败率
- 平均耗时
- 执行次数

第二层：趋势

- 7 天
- 30 天
- 90 天

第三层：执行记录

- 时间
- 工作流
- 设备摘要
- 结果
- 耗时

### 8.3 Run Detail Drawer

点击执行记录后打开详情 Drawer。

展示：

- 执行截图引用
- 执行路径
- 节点耗时
- 失败原因
- 相关日志
- 本地证据摘要

### 8.4 Boundary

- Monitor 只读取本地数据
- 不上传云端
- 不接远程分析服务
- 不实现遥测系统

## 9. Future L7 Fleet

Fleet 只作为未来 IA 预留。

可能方向：

- 设备集群
- 批量执行
- 分组执行
- 轮询执行
- A/B 执行

V2.0 不实现：

- 多设备调度
- 多设备同步
- 多设备并发
- 远程设备池

## 10. Common Interaction Containers

### 10.1 Drawer

用于：

- 编辑
- 查看详情
- 高级配置
- Inspector

### 10.2 Modal

用于：

- 删除
- 执行确认
- 危险操作确认

### 10.3 Toast

用于：

- 保存成功
- 执行成功
- 连接成功
- 非阻塞提示

### 10.4 Loading

统一使用 Skeleton。

原则：

- 避免大面积 Spinner
- 加载中仍保留页面结构
- 状态等待必须有可解释文案

## 11. IA Change Rules

- 新功能必须先归入 L1-L6 之一。
- 如果归不进去，先判断是否属于 Settings、Drawer、Console 或 Future。
- 不得为单个功能随意新增一级导航。
- 主导航只承载用户心智中的长期模块。
- 任何多设备能力进入实现前，必须先创建 Fleet 专题边界。
