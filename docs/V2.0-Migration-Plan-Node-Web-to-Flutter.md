# iOS Assist Studio V2.0 Migration Plan: Node/Web to Flutter

## 0. TL;DR

- V2.0 迁移目标是从当前 Node + Web Console 转为 Flutter Desktop Mac App。
- 迁移期间可以临时保留旧 Web 看板用于开发对照，但不得作为 V2.0 产品入口。
- Flutter App 禁止调用自建 Node 服务。
- 现有 Node 逻辑只作为 Legacy、Debug、迁移参考和短期兜底。
- 迁移必须分阶段推进，不能一次性删除旧 CLI 和旧验证脚本。

## 1. Current State

当前项目已有：

- Node CLI 初始化
- Node CLI 点击
- Node Web Console
- Appium / WDA 连接逻辑
- 常驻 session
- runner
- workflow DSL
- evidence store
- visual guard
- smoke scripts
- 中文文档真源

这些能力是 V2.0 的迁移参考，不是全部废弃。

## 2. Target State

目标状态：

- Flutter Desktop Mac App 是唯一主入口。
- Dart Runtime 管理设备连接、执行、安全停止、证据和监控。
- Dart Appium Client 直接调用 Appium / WebDriver HTTP API。
- Appium Server 由 Dart Runtime 管理生命周期。
- WDA 继续作为真机底层自动化代理。
- Node / Web / CLI 保留为 Legacy 和迁移参考。

## 3. Non-negotiable Migration Rules

- 不允许 Flutter App 调用当前 Node Web 看板 API。
- 不允许把 Node Runner 包装成 Flutter 的后台服务。
- 不允许 V2.0 新功能继续写入当前 Web Console。
- 不允许边迁移边引入第二套主执行机制。
- 不允许破坏 Appium / WDA 主驱动边界。
- 不允许突破单设备串行执行边界。

## 4. Legacy Handling

Node / Web / CLI 已物理归档到 `legacy/node/src/`。

当前状态：

- 根目录不再保留活跃 `src/`。
- 旧 Web Console、CLI、Runner、Session、Evidence、Visual 和 Node smoke 只在 `legacy/node/src/` 中保留。
- 旧 npm 入口统一加 `legacy:*` 前缀。
- 默认 `check`、`verify:all`、`smoke:v2-boundary` 和 `smoke:macos-build` 已切到 Dart / Flutter 工具。

归档语义：

- `legacy/node/src/ios-assist-console.mjs` 是 Legacy Web Console。
- `legacy/node/src/ios-coordinate-*.mjs` 是 Legacy CLI。
- V2.0 主路径不得依赖这些服务。

历史迁移已完成：

```text
src/ -> legacy/node/src/
root package.json -> 仅保留 V2 wrapper 与 legacy:* 调试入口
```

后续不得新增无前缀 Node 产品入口。

## 5. Config Migration

当前 `config/connected-device.sequence.json` 可作为迁移输入。

允许：

- Flutter 第一版读取旧 sequence。
- Flutter 将旧 A-F 序列导入为基础 Workflow 模板。
- Workflow DSL package 实现 legacy sequence import。
- Dart Runtime 读取旧 Appium endpoint、capabilities 和运行参数，用于组装 Appium client、session manager 和执行器。

不建议：

- Flutter 长期直接写入旧 sequence 作为主真源。
- Flutter 把旧配置读取逻辑扩散到 UI 层。

长期目标：

- Workflow JSON 成为 V2.0 主真源。
- Settings JSON 承载本地设置。
- Legacy sequence 只用于导入、兼容和调试。

已落地的迁移策略：

- `workflow_dsl` 负责将 legacy Tap / Wait sequence 转换为线性 Workflow DSL。
- `studio_runtime` 负责读取 legacy 配置并构造 Appium endpoint、DeviceSession capabilities 和默认 tap duration。
- `studio_runtime` 启动时优先恢复本地 Project DSL workflow；缺失或无效时才回退到 legacy sequence 或内置 A-F 模板。
- `studio_mac` 启动时优先发现项目配置，失败时才回退到内置 A-F 模板。
- Recorder Promote 写入同一个本地 Workflow Store，不再只停留在内存 snapshot。

## 6. Runtime Migration

迁移顺序：

1. Appium status detector
2. Appium process manager
3. WebDriver session client
4. Screenshot / page source
5. Tap / pointer action
6. Safe release actions
7. Session state machine
8. Runner state machine
9. Workflow DSL execution
10. Evidence store
11. Monitor repository

原则：

- 每迁移一个能力，都要有 Dart fake test。
- 不直接照搬 Node 代码结构。
- 迁移的是业务语义，不是文件形状。
- 状态机规则必须继承现有文档边界。

## 7. UI Migration

UI 迁移顺序：

1. App Shell + L1-L6 navigation
2. Top Status Bar
3. Bottom Console
4. Device status page
5. Execute MVP
6. Dashboard MVP
7. Recorder MVP
8. Workflow read-only / template view
9. Monitor MVP
10. Workflow visual canvas MVP
11. Full Workflow canvas editing

约束：

- 旧 Web 看板可在开发期用于对照，但不作为交付入口。
- Flutter UI 必须按 V2.0 IA 和 UX Spec 实现。
- 技术细节默认进入 Drawer、Inspector、Advanced 或 Bottom Console。

## 8. Workflow Canvas Migration

第一阶段不做完整 Workflow 画布。

但终极目标必须保留：

- 可视化节点编辑
- 拖拽节点
- 连线
- 缩放
- 框选
- 复制
- 删除
- 自动布局
- Visual / Source 双视图
- Node Inspector
- 运行态节点高亮
- 失败节点定位
- 时间轴和节点证据联动

阶段策略：

- Phase 1：只读模板视图，能展示 A-F 基础流程。
- Phase 2：Visual / Source / Validate 共享同一 Project DSL，并完成 Recorder Promote、本地 workflow 存储和运行事件联动。
- Phase 3：完整 Workflow 画布，支持拖拽、缩放、选中、Inspector、新增、删除、连线、Source 同步和保存校验。

## 9. Appium Installation Boundary

第一版不内置 Appium 安装器。

第一版能力：

- 检测本机 Appium 是否可用
- 显示 Appium 不可用原因
- 给出用户可理解引导
- 允许 App 启动和停止 Appium

后续能力：

- 依赖检查向导
- Appium 安装向导
- Xcode / WDA 准备检查
- 证书信任和 Developer Mode 引导
- 环境修复建议

## 10. Verification Migration

迁移阶段需要双轨验证：

Legacy 验证：

- 现有 Node smoke 在未迁移前继续可运行。
- Legacy 只验证旧能力未被误伤。

Flutter 验证：

- Dart package tests
- Flutter widget tests
- Fake Appium client tests
- Fake runtime tests
- macOS build smoke

当 Flutter Runtime 覆盖某个能力后：

- 将该能力从 Legacy smoke 迁移到 Dart / Flutter test。
- 文档更新对应真源。
- 标记 Legacy 对照脚本为可删除候选。

## 11. Completion Criteria

迁移完成时应满足：

- 用户只需要打开 Flutter Mac App。
- 设备连接、执行、停止、日志、证据都在 Flutter App 内完成。
- Appium 进程由 Dart Runtime 管理。
- Workflow DSL 由 Dart package 校验和运行。
- Monitor 使用本地 Flutter / Dart 数据源。
- 当前 Web Console 不再作为主入口。
- 当前 Node Runner 不再作为 V2.0 运行依赖。

## 12. Change Rules

- 如果临时使用旧 Web 看板做开发对照，必须在任务说明中标记为 Legacy 对照。
- 任何新功能不得只在 Web Console 中实现。
- 任何 Flutter 功能不得通过调用 Node Web API 交付。
- 迁移计划变更必须同步 V2.0 Architecture、Monorepo Plan 和 AGENTS。
