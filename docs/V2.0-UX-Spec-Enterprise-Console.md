# iOS Assist Studio V2.0 UX Spec: Enterprise Console

## 0. TL;DR

- V2.0 的文档保持视觉中立，最终实现采用 Tech Noir Enterprise Workstation。
- 体验关键词是 Dark、Industrial、Precision、Professional、Mission Control。
- 主界面遵循 Summary First, Detail Later。
- Bottom Console 全局常驻，承载 Log、Error、Inspector、Network、Debug。
- Skeleton 是默认 Loading 形态，不使用大面积 Spinner。

## 1. Design Direction

最终实现风格：

- Tech Noir Enterprise Workstation
- 深色工作台
- 工业化精密感
- 高密度但低噪声
- 状态明确
- 操作克制
- 面向长时间使用

参考气质：

- Linear 的克制和清晰
- Raycast 的命令效率
- Cursor 的工作区和 Inspector
- Arc 的空间感
- Stripe Dashboard 的信息组织

避免：

- 传统后台 Admin 风格
- 营销页式 Hero
- 大面积装饰卡片
- 无状态语义的渐变和光效
- 主界面堆长日志或技术细节

## 2. Experience Principles

### 2.1 Summary First, Detail Later

主界面只展示用户做决定所需的信息。

摘要状态：

- 就绪
- 提醒
- 错误
- 离线
- 运行中
- 暂停

内部可以继续使用英文枚举或 DSL 字段，但用户可见文案必须转换为短中文。主界面、Drawer、Inspector、Console、状态栏、Execution Timeline、Run Detail 和 Visual Evidence Chain 不直接显示 `connected`、`running`、`tap`、`INFO`、`WARNING`、`ERROR` 等底层值。

Runtime 事件源应优先产出中文短句。Flutter UI 可以保留旧英文事件的兼容转换，但不能把兼容转换当作新功能的主要文案来源。

详细信息进入：

- Drawer
- Inspector
- Advanced
- Bottom Console

### 2.2 Understandable Before Flexible

Primary User 不会 Appium / WDA。

默认交互必须避免暴露：

- 完整设备标识
- session 细节
- WDA endpoint
- source XML
- 坐标细节
- 证书主体

这些信息只在高级区域以脱敏形式展示。

### 2.3 Safe Before Fast

执行体验优先安全：

- 运行前确认
- 运行中明确停止入口
- 停止采用安全停止
- 低置信视觉判断进入挂起态
- 异常重试有上限

### 2.4 Evidence Before Guessing

失败态需要给证据：

- 当前节点
- 错误摘要
- 本地截图引用
- 相关日志
- 建议动作

不要只展示泛化错误。

## 3. Global Layout UX

### 3.1 Top Status Bar

固定在顶部，高度 56px。

展示内容：

- 设备
- 驱动
- 流程
- 运行
- 设置

状态表现：

- 绿色：就绪
- 黄色：提醒
- 红色：错误
- 灰色：离线

交互：

- 点击状态打开详情 Drawer
- Hover 显示短 tooltip
- 长错误不在状态栏展开

当前 Flutter Desktop 已落地：

- Device / Driver / Workflow / Execution 状态均可点击。
- 点击后打开全局状态详情 Drawer，优先展示当前状态摘要，再展示下一步动作和短字段。
- 状态详情只读，不触发连接、断开、运行或 workflow 写入。
- 状态详情由 `StudioRuntimeSnapshot` 派生，不展示完整设备标识、完整 session、WDA endpoint、source XML 或原始 WebDriver payload。
- 连接失败时优先展示 Runtime 的结构化诊断摘要和下一步动作；底层详情只在抽屉或控制台中以脱敏文本出现。

### 3.2 Side Navigation

默认宽度 72px。

收起态：

- Icon
- Tooltip

展开态：

- Icon + Label

当前页面：

- 左侧高亮条
- 背景轻微高亮

导航不承担说明文案。

### 3.3 Workspace Area

Workspace 是主任务区域。

要求：

- 页面第一屏必须能直接完成当前模块的核心任务
- 不放营销式介绍
- 不放大段使用说明
- 不把技术日志铺到主区域
- 空状态提供下一步动作，而不是解释项目背景
- 空状态不展示开发路线、未来目标或内部阶段；无法行动时只展示当前状态短句

### 3.4 Global Command Center

Global Command Center 是桌面工作站效率入口。

入口：

- 顶部状态栏命令按钮
- `⌘K`
- `Ctrl+K` 作为非 macOS 键盘兜底

当前 Flutter Desktop 已落地的基础形态：

- 居中命令面板
- 搜索本地命令
- 方向键选择
- 回车执行
- Esc 关闭
- 支持前往 L1-L6 页面
- 支持打开 Device / Driver / Workflow / Execution 只读状态详情
- 支持打开 Settings Drawer
- 支持运行本机环境检查
- 支持复制本机隧道命令作为高级备用

约束：

- 命令中心 MVP 不触发设备连接、断开、运行或 workflow 写入。
- 本机环境检查只刷新 Dart Runtime dependency report，不读取设备清单、不展示完整路径、不写 workflow。
- 复制本机隧道命令只作为高级备用，只写入剪贴板，不执行命令、不打开终端、不请求 sudo。
- 命令中心动作执行前先关闭面板，避免弹层层级混乱。
- 命令文案使用短中文，默认不展示 Appium、WDA、session 等底层术语。
- 后续可扩展更多安全命令，但危险操作仍必须进入确认 Modal。

### 3.5 Bottom Console

Bottom Console 全局常驻。

默认：

- 收起 48px

展开：

- 320px

分区：

- 日志
- 错误
- 检查
- 网络
- 调试

交互：

- 复制全部
- 清除全部
- 错误过滤
- 级别过滤
- 当前运行关联

约束：

- Console 中的公共文本必须脱敏
- Console 可见级别使用全部 / 信息 / 提醒 / 错误，不直接显示 `INFO`、`WARNING` 或 `ERROR`
- Runtime 英文原始消息展示和复制前必须经过安全中文化与脱敏
- 新增 Runtime 事件、视觉证据原因和运行详情摘要时，源头默认写中文短文案
- Console 不替代 Monitor
- Console 不替代状态栏
- 清除全部只清除当前 Console 视图，不删除本地运行证据、截图或历史记录

当前 Flutter Desktop 已落地的基础形态：

- 默认 48px 收起
- 展开 320px
- Log / Error / Inspector / Network / Debug 分区
- 复制全部
- 清除全部
- 级别筛选：全部 / 信息 / 提醒 / 错误
- Error 分区聚合 warning 和 error 事件
- 级别筛选只影响 Console 当前视图和复制内容，不改变 Runtime event stream、本地 evidence、run history 或截图证据
- Inspector 分区展示当前 runtime 摘要
- Network 分区先作为 Appium/WebDriver 事件预留，不作为自建 Node API 入口

### 3.6 Settings Drawer

Settings 是全局本地工作站设置入口，不是账号、权限或云端协作入口。

当前 Flutter Desktop 已落地的基础形态：

- 入口位于 Side Nav 底部。
- 以右侧 Drawer 承载 Workstation、Runtime、Privacy 和 Boundaries。
- Workstation 展示 Single Device First、Local Only Storage、Cloud Sync 等本地产品边界。
- Runtime 只展示 Flutter -> Dart Runtime -> Appium -> WDA、设备摘要、Appium 摘要、执行摘要和 Workflow 摘要。
- Privacy 展示 Hide Device Identifier、Hide Raw WebDriver Payload、Reveal Screenshots By Default、Evidence Retention 和截图预览摘要。
- Hide Device Identifier 和 Hide Raw WebDriver Payload 是只读开启状态，用于表达硬隐私边界。
- Reveal Screenshots By Default 是用户偏好；默认关闭，开启后 Run Detail 可默认展示本地 evidence 缩略图。
- Boundaries 展示 Serial Execution、Safe Stop、Legacy Node API、Arbitrary Script Execution。
- Evidence Retention 使用紧凑步进器控制本地 evidence 保留数量，变更必须通过 Runtime 持久化并应用滚动清理。
- Settings 主体不展示完整设备标识、完整 session、WDA endpoint、source XML 或原始 WebDriver payload。

## 4. Module UX

### 4.1 Dashboard UX

Dashboard 目标是快速判断系统健康度。

首页信息顺序：

1. 当前工作区摘要
2. KPI Cards
3. Recent Workflows
4. Activity Trend

KPI 行为：

- 可点击
- 跳转对应模块
- 不在卡片内展开复杂详情

当前 Flutter Desktop 已落地：

- 当前工作区摘要：Local Automation Workstation，并以 Ready、Setup Required、Manual Intervention、Action Required 等状态表达系统健康。
- KPI Cards：Connected Devices、Workflows、Today Runs、Success Rate。
- KPI 跳转：设备、工作流、执行和监控模块。
- Recent Workflows：展示当前 Project DSL workflow 摘要、最近状态、节点数量、最近运行时间和成功率，并提供 Workflow Summary Drawer、打开 Workflow、进入 Execute、收藏、复制和删除的本地入口。
- Activity Trend：展示本地 7 日运行趋势。
- 技术细节仍进入对应模块、Drawer 或 Bottom Console，不在 Dashboard 铺开。

Recent Workflows 行为：

- Summary First：默认只展示状态、节点数、最近运行时间和成功率。
- Detail Later：详情进入 Workflow Summary Drawer，展示 Project DSL、节点类型分布、最近运行和本地边界。
- 导航入口：允许打开 Workflow 或进入 Execute，但不在 Dashboard 直接编辑或执行。
- 收藏：只写本机 Settings，不触发设备动作。
- 复制：通过 Runtime 生成当前流程副本并写回 Project DSL Store，运行中不可用。
- 删除：必须确认后回到 A-F 基础模板，运行中不可用，不直接删除本地证据。
- 查看最近状态

### 4.2 Device UX

Device 目标是让用户确认“手机连上了，画面可见，WDA 可用”。

核心体验：

- 单设备模式优先显示当前设备
- 设备预览占主要空间
- 点击预览有准星和波纹反馈
- 设备状态只展示摘要
- 高级连接诊断进入 Drawer
- Device 和 Execute 主界面只用共享诊断卡展示“原因”和“下一步”，避免用户翻日志找处理方式。

状态等待：

- 初始化中显示连接阶段
- 等待证书信任时给明确用户动作
- 超过重试上限后进入可操作错误态

当前 Flutter Desktop 已落地：

- Device Center 摘要区：展示单设备模式、设备连接状态、Appium 状态、预览状态、短 session 和最近截图时间。
- Local Readiness Guide：展示 Appium Service、USB Device、Developer Trust、WDA Session、Safe Capture 和 Workflow DSL。
- Local Stack Check：展示 Appium CLI、Xcode CLI、iOS Device Tools、本机隧道和 WDA Prerequisites 的本机依赖诊断，并显示最近检查时间。
- 本机隧道在 UI 中显示为“本机隧道”，用于解释真机会话启动异常；它指 Appium XCUITest 会话所需隧道，不等同于 pymobiledevice3 辅助隧道。主界面不展示 Remote XPC、端口或进程参数。
- Local Setup Guide：作为 Advanced Drawer 展示 Appium、Xcode、USB Tools、本机隧道、WDA 和 iOS Trust 的本机准备指引；该 Drawer 只做 Summary First 指引，不自动安装、不签名、不展示完整路径、完整设备标识、WDA endpoint 或原始 WebDriver payload。
- “连接设备”是 Device 主操作：按顺序完成本机检查、必要时弹出 Mac 密码框、受控启动本机隧道、准备驱动、创建手机会话。用户不需要打开终端或理解命令。
- 如果点击前状态判断为不需要密码，但连接过程中刷新本机环境后发现仍需密码，界面必须在同一次点击流程内补弹 Mac 密码框并继续连接，不要求用户再点一次。
- 如果驱动已由外部启动，界面只提示“外部驱动可直接使用”，停止驱动动作必须禁用或给出“外部驱动未停止”，不能显示“驱动已停止”。
- 如果外部旧驱动在当前端口持续看不到已确认 USB 手机，一键连接可显示“正在重置旧驱动”并切换为受控驱动；该动作不要求用户手动停驱动，也不匹配本机隧道进程。
- 创建手机会话前必须先确认当前唯一 USB 手机；确认失败时直接展示“未找到 USB 手机”和“用数据线连接一台手机并解锁”，不进入 Appium 底层报错。
- WDA 代理瞬时断开时，界面可显示“正在恢复手机会话”，由 Runtime 自动重启受控驱动并重试一次；Xcode 构建、证书、USB 和隧道问题只显示下一步，不自动重启。
- 已有本机隧道但 registry 为空时，界面先显示“等待手机允许”；超时后若属于旧隧道残留，可在同一次密码授权内显示“正在清理旧隧道”并重启受控隧道。
- 本机驱动未识别已绑定手机时，界面展示“驱动未识别手机”和“保持解锁，点连接设备。仍失败就重插线。”，不把它误写成“未找到 USB 手机”。
- 如果只发现无线可用手机而没有 USB 手机，界面必须直接提示使用数据线连接，不展示为可继续连接状态，也不要求用户理解设备标识或底层传输类型。
- 连接诊断卡中，USB 连接问题的短标签统一显示为“USB”，下一步使用“用数据线连接一台手机并解锁”这类动作句；驱动未识别手机时短标签显示“驱动”。两类问题都不显示 `localNetwork`、`transportType`、UDID、`[device]` 脱敏占位符或 Appium 原始错误。
- Local Readiness Guide 的“手机会话”项必须复用同一连接诊断摘要和下一步动作，避免上方诊断卡和下方就绪项给出不同处理建议。
- 本机密码只用于启动本机隧道的 `sudo` stdin，不保存、不进入日志、不进入 evidence、不进入复制内容。
- 当本机隧道处于提醒或错误态时，Advanced Drawer 可显示“复制命令”动作；该动作只是排障备用，只写入剪贴板，不执行命令、不打开终端、不请求 sudo。
- 本机隧道指引必须围绕一键连接显示人工动作：点连接设备、输入 Mac 密码、解锁 iPhone 并在弹窗时允许、等待应用继续连接；不得把终端命令作为主流程。
- Local Stack Check 的控制台事件必须带出第一项可处理问题和下一步动作；不得反复只显示“发现阻断项”或“需要处理”。
- 每个 readiness 项必须包含状态、简短说明和下一步动作；主界面不展示 endpoint、完整 session、完整设备标识、source XML 或原始 WebDriver payload。
- Local Stack Check 不展示原始命令输出、不列出设备清单，不把本机路径或设备唯一标识带入主界面。
- 驱动准备是“连接设备”的内部阶段：先探测已有服务，未发现时再启动，启动后等待就绪；失败时必须转化为短中文动作提示。本机驱动不可达时提示“连接设备”或“查环境”，不得只展示连接失败原文。
- Developer Trust 等系统级限制必须转化为等待态和用户操作引导，不尝试绕过系统安全限制。
- Controls：连接设备是大号主操作；查环境、指引、停止驱动、重绑、断开、截图是辅助动作。本机检查、隧道启动、驱动准备和会话创建都是连接设备的内部阶段，不作为主界面的并列选择；连接、断开、驱动启动停止、外部驱动运行或运行处理中，停止驱动、重绑和查环境等可能打断状态机的动作必须禁用。
- Execute 页未连接时复用同一个“连接设备”主操作；运行页不重新发明连接流程，不展示第二套驱动、会话或密码说明。
- Device Preview：占据主区域，展示 iPhone 预览框、空状态、截图时间和 Gesture Ready / Gesture Locked 摘要；connected、idle 且有截图时，点击预览显示十字准星并发送受控 viewport tap，拖动预览显示轨迹线并发送受控 viewport swipe；运行中、未连接或无截图时必须锁定。
- 主界面不展示完整设备标识、完整 session、WDA endpoint、source XML 或原始 WebDriver payload。

### 4.3 Recorder UX

Recorder 目标是让用户不用理解坐标也能录制动作。

录制中：

- 显示录制中状态
- 有红点脉冲
- 动作实时进入时间轴
- 动作命名尽量贴近用户意图

动作列表默认展示：

- 动作类型
- 目标摘要
- 等待时间
- 顺序

动作详情 Drawer 展示：

- 用户可读摘要
- 坐标
- 截图
- 元素信息
- 等待时间
- 重试策略

优先级：

- 录制成功率优先于节点丰富度
- 默认隐藏高级字段

当前 Flutter Desktop 已落地的基础形态：

- L3 Recorder 已以 Recorder Workbench 表达：Recorder Controls、Live Capture、Session Summary 和 Action Timeline。
- Live Capture 可通过 Runtime 手动采集当前设备截图，用于录制现场预览；该能力只在设备 connected 且运行 idle 时启用。
- Session Summary 展示录制中 / 空闲、可截图 / 需准备、动作数量、最近预览时间、设备摘要和运行锁。
- Session Summary 在未连接时复用同一个“连接设备”主操作；录制页不展示第二套连接说明，也不要求用户回到 Device 页。
- Recorder Preflight 覆盖 Device ready、Run idle 和 Preview available。
- 开始录制 / 停止录制状态切换
- 本地插入 Tap / Wait / Swipe / Input 动作
- Input 动作主时间线只显示输入摘要，不展示明文；文本只在动作详情中编辑
- Action Timeline 展示动作类型、目标摘要、等待或耗时、顺序
- 坐标默认不在时间轴中展示
- 点击动作打开右侧 Action Detail Drawer
- Drawer 展示用户可读摘要、坐标、截图证据占位、元素摘要、等待时间和重试策略；不得展示内部动作 ID、完整设备标识、session、endpoint 或底层 payload
- Promote to Workflow 可将当前动作转换为 Runtime 中的 Project DSL
- Runtime 会拒绝校验失败的 workflow，不把无效 DSL 写成真值
- 当前动作捕获仍是 UI MVP，不代表已经接入真实设备事件流或持久化 workflow 文件

### 4.4 Workflow UX

Workflow 是旗舰工作区。

画布体验参考节点式编排工具，但保持本项目的自动化语义。

核心交互：

- 拖拽节点
- 缩放画布
- 框选
- 复制
- 删除
- 自动布局
- 双击打开 Inspector
- 画布 / 源码 / 检查切换

节点状态：

- 正常：蓝色
- 执行中：绿色脉冲
- 失败：红色
- 警告：黄色

源码视图：

- 只展示 Project DSL
- 保存前必须校验
- 错误定位到节点或字段
- 不允许任意脚本执行

当前 Flutter Desktop 已落地的基础形态：

- Visual 以 Canvas MVP 展示 workflow，包括左侧 Node Palette 节点库、Tap / Wait / Swipe / Input / Loop / Snapshot / Condition / Visual Branch / Catch / Sub Workflow 节点新增、节点卡片、网格、连线、平移、缩放、适配并居中视图、节点拖拽、Auto Layout、顶部 Add Node 可见节点新增菜单、画布级快捷键、Cmd+C / Cmd+X / Cmd+V 画布本地复制剪切粘贴、Cmd+A 全选、Esc 清空选择、框选多选、多选批量移动、Mini map 点击/拖拽导航、Node Navigator 搜索/定位、受控连接编辑、端口点选连接、端点拖拽连线、画布边选中/删除和选边插入核心节点；Loop 当前只支持 bounded Loop 骨架，复杂嵌套控制流仍归入 Phase 3 完整 Workflow 画布
- Visual 节点卡片右侧使用短中文分支摘要，只展示“后续、满足、否则、主体、通过、主线、错误”等用户可读语义，不直接展示裸节点 ID
- Visual 连线提示、端口提示和选边浮层使用节点短中文标签，不直接展示裸节点 ID；分支边在选边浮层中补充“满足、否则、主体、后续、通过、主线、错误”等短语义；标签过长时省略，不撑开画布工具条
- Catch 错误分支在 Visual 中表现为独立红色错误边，可选中、删除或插入节点；界面只说“错误”，不要求用户理解 `onError`
- Source 以格式化 JSON 展示同一 Project DSL
- 源码诊断展示解析和 validator 错误，并以节点、字段或源码级位置辅助定位；点击诊断只移动编辑器选区，不保存、不运行、不修改画布视图
- 检查视图展示离线 validator 结果，并以同一诊断模型支持点击定位；可映射到节点的错误切回画布并选中节点，源码级或字段级错误切回源码并定位编辑器选区
- Visual 节点卡片和 Node Inspector 以只读方式展示节点级 validator 问题；主画布只显示问题数量和 Summary，详细说明进入 Inspector，不在画布堆叠技术细节
- Canvas Overview 作为画布内只读概览条，短中文展示节点数、连线数、问题数和当前选区；它只消费当前 Project DSL 和编辑器选区，不保存、不运行、不修改 DSL，也不展示裸节点 ID
- Node Navigator 作为画布内轻量导航面板，默认折叠以避免遮挡节点拖拽和连线；展开后支持节点搜索，以及 Current、Failed、Issues 快速定位；它只改变视口和选中态，不保存、不运行、不修改 DSL
- Canvas Lock Banner 在画布进入只读状态时展示锁定原因，例如运行中、Source 草稿未保存或保存中；它使用 Summary First，不把技术细节铺满画布，不保存、不运行、不修改 DSL
- Workflow 页头提供“去运行”入口，只导航到 Execute 页，不直接启动任务；流程无效、源码草稿未保存、保存中或运行中时禁用，避免用户误执行旧流程
- Template Library 以右侧 Drawer 展示本地工作流模板，并支持导入到当前 Project DSL
- Inspector 展示节点统计、运行锁摘要、只读 Context Variables、单节点属性编辑和多选摘要
- Multi Selection Inspector 支持 Duplicate Selected 和 Delete Selected，默认排除 Start / End / Entry 节点，并通过 Project DSL validator 保存
- Source 可复制
- 该形态只作为完整 Workflow 画布前的桥接版本；复杂图复制、复杂图编辑和更完整的画布快捷操作仍按 Phase 3 实现

### 4.5 Execute UX

Execute 目标是让用户放心开始、看懂进度、随时安全停止。

开始前：

- 配置执行模式
- 展示执行摘要
- 点击开始后弹窗确认

运行中：

- 进入 Running Mode
- 显示进度条
- 显示当前节点
- 显示总节点
- 显示预计剩余时间
- 安全停止始终可见

挂起中：

- 状态显示 Paused / Manual Intervention
- Summary 区展示 Paused Node，不把低置信挂起直接包装为系统错误
- 主操作为 Resolve Pause，用于安全收口当前挂起态
- 不提供无条件继续执行按钮，避免低置信后继续自动点击
- 详细原因进入 Inspector、Bottom Console 或后续 Run Detail

停止中：

- 状态进入 Stopping
- 告知当前原子动作结束后停下
- 不把停止中误标为 Error

当前 Flutter Desktop 已落地：

- L5 Execute 以运行控制台表达，而不是简单按钮区。
- 左侧 Run Configuration：展示单次 / 循环 / 持续模式、当前 workflow、节点数、有限轮次和 Preflight；单次固定 1 轮，循环可调 N 轮，持续模式最多 999 轮且可随时安全停止。
- Preflight 覆盖 Device ready、Appium ready、Workflow valid 和 Run idle。
- 中间 Command Center：展示 Ready To Execute / Running Workflow / Paused For Review 等摘要状态、进度条和核心命令；执行入口是一个主开始按钮，按钮文案跟随当前有效轮次。
- Confirm Execution Modal：主开始按钮在调用 Runtime 前必须弹出确认，展示 workflow、模式、loop 数、节点数、串行执行和安全停止边界；取消不触发 Runtime，确认后才启动。
- 右侧 Execution Summary：使用 Runtime Execution Focus 展示当前循环、当前节点、完成步数、总步数、预计剩余时间、失败或暂停节点。
- Runtime State 只展示摘要状态、短 session 和最近截图时间，不展示完整设备标识、WDA endpoint、source XML 或原始 WebDriver payload。
- Last Run Evidence 在 Execute 页展示最近本地运行摘要，并提供进入 Run Detail Drawer 的入口；该入口与 Monitor 详情共用同一 Drawer 和同一 Dart Runtime 证据读取路径。
- Execution Timeline 在 Execute 页只展示最近 Runtime 事件摘要，采用 Summary First；完整日志和调试细节仍放入底部 Console，避免执行主界面被技术日志淹没。
- 单次、循环、持续、Stop / Resolve Pause 仍全部通过 Dart Runtime 调用 Appium / WDA，不经过 Legacy Node API；持续模式不是无边界无限循环，而是最多 999 轮的受控安全运行。

### 4.6 Monitor UX

Monitor 目标是解释运行结果、问题状态和用户下一步动作。

信息层级：

1. KPI
2. 趋势
3. 执行记录
4. 详情 Drawer

当前 Flutter Desktop 已落地：

- KPI：以紧凑状态条展示总数、成功率、均耗时、失败、暂停和已停；中文文案保持短、清晰、不过度技术化，避免占用趋势和记录区域
- Trend：支持 7 / 30 / 90 日本地窗口切换，展示本地运行量，并用问题段提示 failed、paused、stopped 占比
- Status Distribution：展示 completed、failed、paused、stopped 的本地状态分布
- Recent Runs Filters：提供 All、Issues、Failed、Paused、Completed，并支持按 workflow 名称、状态和 run id 本地搜索，帮助用户快速定位本地运行记录；过滤和搜索只改变当前视图，不改变证据或运行历史
- Recent Runs：保留详情入口，详情 Drawer 继续承载问题分析、执行路径摘要和证据引用
- Run Detail Visual Evidence Chain：当运行包含 Visual Branch 判断时，详情 Drawer 展示规则、截图状态、置信度、阈值、动作和原因；已知系统弹窗只展示规则命中摘要和人工处理提示；该面板只读，不展示原始 WebDriver payload、source XML、完整设备标识或本机绝对路径
- Run Detail Related Events：详情 Drawer 内展示当前运行的本地事件流，提供 All、Nodes、Issues、Screenshots 四个本地视图；事件行展示时间、状态、循环、节点摘要和脱敏错误说明，不展示截图路径、完整设备标识、session 或原始 WebDriver payload
- Run Detail Trace Filters：详情 Drawer 内的节点路径提供 All、Issues、Screenshots 三个本地视图，帮助用户先看问题节点或截图证据节点；过滤只改变 Drawer 当前视图，不改变证据或运行历史

详情 Drawer 展示：

- 执行路径
- 执行路径摘要
- 证据胶片
- 节点耗时
- 问题原因
- 问题节点
- 视觉证据链
- 截图引用
- 相关日志

证据胶片默认只展示节点、状态、循环和耗时摘要，不直接展示截图画面；用户主动 Reveal 或开启本地 Reveal Screenshots By Default 偏好后才显示本地缩略图。

状态语义：

- Completed：Ready
- Failed：Error
- Paused：Warning，表示 Manual Intervention
- Stopped：Warning，表示用户或安全停止

Paused 不应被包装为失败。低置信视觉判断、等待人工确认和安全挂起都应优先表达为可恢复的问题态。

边界：

- 只读
- 本地数据
- 不做云端遥测

## 5. Interaction Components

### 5.1 Drawer

Drawer 用于详情、编辑、高级配置。

适用：

- 状态详情
- 节点 Inspector
- 动作详情
- 执行详情
- 设备诊断

规则：

- 不用于危险确认
- 关闭不应丢失未保存编辑，需有明确提示
- 高级信息必须脱敏

### 5.2 Modal

Modal 用于打断式确认。

适用：

- 删除
- 开始长时间执行
- 停止高风险任务
- 清除历史

规则：

- 文案简短
- 明确后果
- 主按钮和取消按钮视觉区分

### 5.3 Toast

Toast 用于非阻塞反馈。

适用：

- 保存成功
- 连接成功
- 执行成功
- 复制成功

规则：

- 不承载长错误
- 错误详情进入 Drawer 或 Console

### 5.4 Skeleton

Skeleton 是默认 Loading。

适用：

- Dashboard KPI
- 设备信息
- 最近工作流
- 趋势图
- 执行记录
- 设备预览加载

规则：

- Skeleton 保留布局骨架
- 不使用大面积 Spinner
- Loading 超过预期时必须展示可理解状态

## 6. Copywriting Rules

文案面向非工程用户。

推荐：

- “设备已连接”
- “等待 iPhone 信任开发者证书”
- “工作流未保存”
- “执行已暂停，等待人工确认”
- “当前动作结束后停止”

避免：

- 直接暴露原始 endpoint
- 直接展示完整设备标识
- 把 WDA 构建日志铺在主界面
- 使用只有 Appium 工程师才理解的错误作为主文案

错误文案顺序：

1. 发生了什么
2. 为什么可能发生
3. 用户下一步做什么
4. 高级详情入口

## 7. Privacy UX

界面默认脱敏。

默认不展示：

- 完整设备标识
- 完整证书主体
- 真实账号
- 生产地址
- 敏感路径
- 完整 pageSource

如必须展示高级详情：

- 放入 Advanced
- 默认折叠
- 明确标记为诊断信息
- 仍使用脱敏摘要

## 8. Motion And Feedback

动效只用于表达状态，不做装饰。

允许：

- Recording 红点脉冲
- 当前节点绿色脉冲
- 点击 Ripple
- 状态切换轻量过渡
- Drawer 滑入滑出

避免：

- 大面积背景动效
- 影响长时间使用的闪烁
- 无状态意义的光效
- 和 Error / Warning 混淆的颜色动画

## 9. Accessibility And Density

V2.0 是高密度专业工具，但不能牺牲可读性。

要求：

- 状态不能只靠颜色表达
- 按钮和图标需要 tooltip 或文本语义
- 错误和警告有明确文案
- 关键操作有足够点击区域
- 长文本不能挤压主操作

## 10. UX Change Rules

- 新 UI 必须服务任务完成、排障或编排。
- 不增加无状态意义的装饰元素。
- 不把隐藏技术细节误认为删除高级能力。
- 所有高级能力应进入 Drawer、Inspector、Advanced 或 Console。
- 如果界面需要解释很多，优先重构信息架构，而不是继续加说明文案。
