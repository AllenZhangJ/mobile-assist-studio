# 产品定位与旗舰化方向v1.0

## 0. TL;DR

- `iOS Assist Studio` 的长期产品方向是“视觉状态驱动的 iOS 辅助自动化编排器”。
- 当前项目已经具备真机连接、Web 看板、常驻会话和坐标点击能力；下一阶段不是继续堆 CLI，而是升级为可观测、可编排、可诊断的本地工作站。
- 本文用于统一产品定调、能力边界和阶段优先级，避免后续需求在“脚本工具”和“旗舰工作台”之间来回摇摆。

## 1. Product Positioning

`iOS Assist Studio` 面向的是本机 Mac + iPhone 真机辅助自动化场景。

它不是通用云测平台，也不是 Appium 的简单 Web 包装层。它的核心价值是：

- 把真机连接、WDA 会话、点击执行、截图观察、异常处理和运行日志收束到一个本地 Web 工作台
- 让用户从频繁执行 CLI，升级为通过状态清晰的看板完成连接、测试、运行、停止和排障
- 逐步从固定坐标序列，演进到基于当前屏幕视觉状态的自动化编排

## 2. Target Experience

目标体验可以概括为四句话：

- 先连接，再编排：用户先把 iPhone 稳定接入工作台，再配置执行流
- 先看见，再点击：每个动作都应能关联屏幕状态、坐标、等待和结果
- 先诊断，再重试：失败时展示原因、现场快照和恢复路径，而不是只抛一段终端日志
- 先串行可靠，再扩展复杂：任何多轮、多节点、多设备能力都必须建立在确定的单设备串行执行之上

## 3. Product Pillars

| 支柱 | 定义 | 当前状态 | 未来方向 |
|---|---|---|---|
| 设备连接 | 管理 USB iPhone、Appium、WDA、WebDriver 会话 | 已有常驻连接 | 增强预检、重连、设备诊断 |
| 执行编排 | 管理点击序列、等待、循环、停止 | 已有 A-F 串行执行 | 升级为节点图和条件分支 |
| 视觉理解 | 理解当前屏幕状态和可交互目标 | 录制与拾点为主 | 引入 OCR、CV、截图状态判断 |
| 异常自愈 | 识别并处理弹窗、阻断和会话丢失 | 证书信任等待态已显式化 | 引入全局异常拦截器 |
| 观测分析 | 展示日志、状态、时序和瓶颈 | 已有实时日志 | 增加时间轴、快照、热力图 |

## 4. Experience Principles

- 看板是主入口，CLI 是兼容入口。
- 状态要可解释：连接态、运行态、阻断态、错误态不能混在一个文本里。
- 操作要可回溯：一次点击应能追溯到节点、坐标、等待、截图和 WDA 返回。
- 自动化要可暂停：遇到未知状态时应进入可诊断的挂起态，而不是盲目继续。
- 视觉能力先辅助判断，再接管执行：早期优先用于截图比对、异常弹窗识别和目标确认。

## 5. Non-goals

- 不把项目扩展成远程多租户云平台。
- 不默认引入复杂账号、权限、团队协作体系。
- 不为了视觉效果牺牲真机连接和执行稳定性。
- 不把所有未来 DSL、画布、分析能力一次性压进当前代码结构。

## 6. Milestone Direction

### 6.1 Foundation

继续稳定当前本地 Web 看板：

- 初始化并连接
- 连接状态
- 测试 1 轮
- 执行 N 轮
- 停止任务
- 实时日志
- 序列摘要

### 6.2 Visual Assist

把截图和视觉判断接入当前执行链路：

- 运行前截图
- 点击后截图
- 弹窗识别
- 目标区域标注
- 失败现场保留

### 6.3 Workflow Studio

把扁平序列升级为可编排工作流：

- 节点式动作
- 条件分支
- 上下文变量
- 子流程
- 运行时状态机

### 6.4 Intelligence Layer

让编排器根据屏幕状态做更细的决策：

- OCR 读取
- CV 特征判断
- 异常自愈
- 历史失败热力图
- 性能瓶颈分析

## 7. Documentation Rules

- 本文描述产品方向，不声明当前已经实现的能力。
- 当前实现真源仍以项目总文档和代码为准。
- 新增看板、视觉、编排、异常处理类需求时，先读本文再拆实现落点。
