# iOS Assist Studio V2.0 Monorepo Engineering Plan

## 0. TL;DR

- V2.0 使用 Dart / Flutter Monorepo。
- Flutter Desktop App 位于 `apps/studio_mac`。
- 可复用运行时、DSL、Appium Client 和 Design System 位于 `packages/`。
- 现有 Node / Web / CLI 已物理归档为 Legacy。
- Monorepo 管理工具采用 Melos。

## 1. Monorepo Decision

V2.0 采用 Monorepo，而不是单一 Flutter App 目录。

原因：

- UI、Runtime、Workflow DSL、Appium Client 和 Design System 的职责差异明显。
- DSL 和 Appium Client 需要独立测试。
- Runtime 不应依赖具体 UI 页面。
- Design System 不应混入业务运行时。
- Legacy Node 需要被隔离，避免继续污染 V2.0 主路径。

## 2. Target Structure

目标结构：

```text
ios-assist-studio/
  apps/
    studio_mac/
  packages/
    appium_client/
    workflow_dsl/
    studio_runtime/
    studio_design_system/
  docs/
  config/
  legacy/
    node/
      src/
  package.json
  melos.yaml
  pubspec.yaml
  analysis_options.yaml
```

说明：

- `apps/studio_mac` 是 V2.0 Flutter Desktop Mac App。
- `packages/appium_client` 是 Dart Appium / WebDriver HTTP client。
- `packages/workflow_dsl` 是 Project DSL 模型、解析、规范化和校验。
- `packages/studio_runtime` 是 Session、Runner、Evidence、Monitor 等运行时。
- `packages/studio_design_system` 是 Flutter UI 组件和主题。
- `legacy/node/src/` 保留当前 Node Legacy 实现。
- 根目录不再保留活跃 `src/`；V2 主路径不得读取或调用 `legacy/node/src/`。

## 3. Dependency Direction

允许的依赖方向：

```text
apps/studio_mac
  -> packages/studio_design_system
  -> packages/studio_runtime
  -> packages/workflow_dsl

packages/studio_runtime
  -> packages/appium_client
  -> packages/workflow_dsl

packages/studio_design_system
  -> Flutter

packages/workflow_dsl
  -> Dart only

packages/appium_client
  -> Dart IO / HTTP
```

禁止：

- `workflow_dsl` 依赖 Flutter。
- `appium_client` 依赖 Flutter UI。
- `studio_runtime` 依赖具体页面。
- `packages/*` 依赖 Legacy Node。
- `apps/studio_mac` 调用 Legacy Node Web API。
- Legacy Node 反向控制 Flutter App。

## 4. Package Responsibilities

### 4.1 apps/studio_mac

职责：

- App Shell
- L1-L6 路由
- 页面组合
- Window layout
- macOS desktop entry
- 用户操作入口

不负责：

- 原始 Appium HTTP 细节
- Workflow DSL 校验细节
- Evidence 文件写入细节
- Legacy Node 调用

当前 Flutter App UI 结构：

```text
apps/studio_mac/lib/
  main.dart                         # Flutter 启动引导
  studio_mac.dart                   # App 公开 Dart 库入口
  src/
    app/                            # Flutter App 根组件
    shell/                          # 顶部状态、导航、命令中心、设置、底部控制台
    features/
      dashboard/                    # L1 总览
      device/                       # L2 设备
      recorder/                     # L3 录制
      workflow/                     # L4 流程
      execute/                      # L5 运行
      monitor/                      # L6 记录
    shared/                         # 跨页面状态、文案、复制反馈、格式化 helper 和 App 私有兼容壳
```

当前拆分规范：

- `main.dart` 只保留 Flutter 启动引导，不承载 Shell 状态、命令中心动作或具体页面实现。
- `studio_mac.dart` 是 Flutter App 公开 Dart 库入口，测试和后续集成从这里导入 `StudioMacApp`。
- `src/app/studio_mac_app.dart` 只负责 Flutter App 根组件、主题和 Shell 注入。
- `src/shell/studio_shell.dart` 负责 Runtime 快照订阅、全局快捷键、命令中心弹窗打开、设置抽屉打开和 L1-L6 主框架组合。
- `src/shell/command_center.dart` 只负责命令中心模型、弹窗、搜索、键盘选择和结果项；`command_center_actions.dart` 只负责命令列表生成和安全剪贴板动作；`command_center_diagnostics.dart` 只负责脱敏诊断摘要文本。
- `src/shell/navigation.dart` 只负责侧栏导航和导航项元信息；Settings Drawer 与 Workspace 调度分别落在 `settings_drawer.dart` 和 `workspace.dart`。
- `features/*/*_page.dart` 作为页面入口，只负责组合主要区域和页面级状态。
- Workflow 按模板模型、模板定义、模板抽屉、模板卡片、节点库、画布主体、画布控制、Mini Map、Painter、节点行、Source、Inspector、Inspector 编辑器和 helpers 拆分；Workflow helpers 继续按页面 Tab、节点文案/图标、Source 诊断、画布历史和图结构 helper 拆分。图结构 helper 再按查询、布局、节点工厂、剪贴板和图编辑五类职责拆分，入口 helper 只保留路由说明，不重新承载算法。
- Workflow Inspector 参数表单继续按分发入口、动作节点、控制/视觉节点和子流程节点拆分；新增节点参数必须先判断归属，不得重新堆回编辑器生命周期文件或单一大表单文件。
- Device 按页面入口、设备摘要、就绪检查、操作区、本机指引和设备预览拆分。跨 Device、Execute 和 Recorder 的就绪卡片、内联步骤和三态就绪行统一放在 `src/shared/readiness_widgets.dart`，不得藏在某个 feature 文件里。
- Recorder 按页面入口、录制控制和时间轴拆分；时间线继续按列表外壳、动作摘要复制、动作详情抽屉、动作模型和 Project DSL 转换 helper 维护，不得把坐标详情、证据占位、复制摘要和 workflow 映射重新堆回单个时间线文件。
- Execute 按页面入口、执行配置和执行摘要拆分。
- Monitor 按页面入口、概览状态、趋势、状态分布、问题分类、耗时节点、运行记录、详情、证据和时间轴拆分。
- `monitor_overview.dart` 只保留概览共享状态；`monitor_trend.dart` 只承载 7 / 30 / 90 日趋势；`monitor_distribution.dart` 只承载 completed / failed / paused / stopped 分布；`monitor_issue_category.dart` 只承载本地问题分类。
- Monitor 运行记录继续按职责拆分：`monitor_history.dart` 只保留入口说明，`monitor_history_filters.dart` 只承载筛选条、搜索框和关联提示，`monitor_history_metrics.dart` 只承载 KPI 网格，`monitor_history_rows.dart` 只承载记录行和详情抽屉入口，`monitor_history_copy.dart` 只承载当前可见记录的脱敏复制文本。
- `shared/` 中状态相关 helper 按状态详情、准备度、运行状态、监控状态和格式化拆分；跨页面复用的状态色映射、普通文本复制反馈等能力必须沉到 `shared/`，不得藏在某个 feature helper 内。一级 Surface 与二级 Inset Surface 的基础视觉实现归属 `packages/studio_design_system`，App 侧私有 `_Surface` / `_InsetSurface` 只作为当前 part 架构下的兼容壳。
- 普通文本复制统一通过 `shared/clipboard_helpers.dart` 写入剪贴板和展示轻提示；业务摘要仍由各 feature 自己生成，Workflow 画布私有节点剪贴板协议仍由 Workflow 分片维护。

新增 V2.0 UI 时应优先落入对应 `features/`、`shell/` 或 `shared/`，再按需要下沉到 `packages/studio_design_system` 或 `packages/studio_runtime`。当单个页面文件继续增长时，先按页面区域和交互职责拆分，再考虑抽象成共享组件。

职责：

不负责：

当前 Appium Client 结构：

```text
packages/appium_client/lib/
  appium_client.dart                # 公共导入入口
  src/
    appium_config.dart              # 本机 Appium endpoint 和 URI 解析
    appium_error.dart               # 统一异常类型

```

`appium_client.dart` 必须保持公共入口稳定。新增 Appium/WebDriver 协议能力时，先判断是 endpoint 配置、响应模型、HTTP transport、W3C action payload 还是 Runtime 门面方法；不得把 Appium 进程管理、session 状态机、运行队列或 UI 文案写入该包。

职责：

- Project DSL models
- DSL parse
- DSL normalize
- DSL validate
- Legacy sequence import
- Expression whitelist
- Node graph validation

不负责：

- 设备连接
- Appium 调用
- UI 展示
- Evidence 存储

当前 Workflow DSL 结构：

```text
packages/workflow_dsl/lib/
  workflow_dsl.dart                 # 公共导入入口
  src/
    workflow_models.dart            # 节点类型、节点、画布位置和工作流定义
```

`workflow_dsl.dart` 必须保持公共入口稳定。新增节点类型先进入 `workflow_models.dart`，随后补 `workflow_validation.dart` 和 Runtime 执行语义；新增模板进入 `workflow_templates.dart`，不得混入模型定义；Source/Visual 的 JSON 解析 helper 归属 `workflow_json.dart`，不得在 UI 页面重复实现。

### 4.4 packages/studio_runtime

职责：

- Appium process manager
- Device session manager
- Workflow runner
- Safe stop
- Pause / resume
- Run event stream
- Evidence store
- Monitor repository
- Settings repository

不负责：

- Flutter widget
- 视觉样式
- 具体页面路由

当前 Runtime 结构：

```text
packages/studio_runtime/lib/
  studio_runtime.dart               # 公共导入入口
  src/
    runtime_models.dart             # Runtime 模型分片入口说明
    runtime_settings_models.dart    # 本机设置和隐私硬边界模型
    runtime_dependency_models.dart  # 本机依赖检查摘要模型
    runtime_run_history_models.dart # 本地运行历史和聚合指标模型
    runtime_run_detail_models.dart  # 运行详情、证据链和节点路径模型
    runtime_execution_state_models.dart # 连接、运行、快照和执行焦点模型
    runtime_execution_internal_models.dart # 执行器内部 Catch、暂停和子流程目标模型
    dependency_probe.dart           # 本机依赖探测
    appium_process.dart             # Appium 进程管理
    project_config.dart             # 项目与设备会话配置
    session_manager.dart            # WDA / WebDriver 会话管理
    evidence_store.dart             # 证据接口、Noop 和 Local store 对外委托
    evidence_store_writer.dart      # 本地 metadata、events、screenshots 和 finish 写入
    evidence_store_history.dart     # 本地运行摘要读取和 Monitor 聚合编排
    evidence_store_detail.dart      # 单次运行详情、事件解析和截图资产读取
    evidence_store_aggregations.dart # 日期趋势、问题分类、失败聚类和节点耗时聚合
    evidence_store_helpers.dart     # 证据安全路径、文件名清洗和轻量字段解析
    device_actions.dart             # Tap、Swipe、输入和截图动作
    workflow_stores.dart            # Workflow、Sub Workflow 与设置存储
    runtime_controller.dart         # Runtime 依赖、快照和广播中心
    runtime_appium_commands.dart    # 本机环境、Appium 启停和设备会话命令
    runtime_device_preview_commands.dart # 设备预览截图、手势和输入命令
    runtime_run_commands.dart       # 工作流启动、安全停止和暂停收口命令
    runtime_workflow_project_commands.dart # 当前 workflow 保存、复制和重置
    runtime_sub_workflow_project_commands.dart # 本地子流程注册、转子流程和删除
    runtime_settings_project_commands.dart # 本机设置、收藏和证据保留策略
    runtime_evidence_project_commands.dart # 运行历史、详情和截图证据读取
    runtime_project_helpers.dart    # 流程和子流程副本命名 helper
    runtime_workflow_references.dart # Sub Workflow 引用完整性校验
    runtime_workflow_execution.dart # 工作流主循环、串行推进和 Catch 路由
    runtime_workflow_node_execution.dart # 单节点调度、成功和失败焦点收口
    runtime_workflow_action_nodes.dart # Tap、Wait、Swipe、Input 和 Snapshot 动作节点
    runtime_workflow_control_nodes.dart # Condition、Visual Branch、Catch、Sub Workflow 和 Loop 控制节点
    runtime_workflow_execution_helpers.dart # 执行辅助、节点查找和短中文运行文案
```

Runtime 执行层必须保持主循环、节点调度、动作节点、控制节点、执行证据、执行规划、节点参数解析、引用校验和视觉守卫分片。新增节点语义时先判断属于设备动作节点还是控制流节点；不得把所有节点实现重新堆回 `runtime_workflow_execution.dart` 或 `runtime_workflow_node_execution.dart`。

Runtime evidence store 必须保持接口/委托、写入、历史读取、聚合算法、详情读取和 helper 分片。新增本地证据写入能力进入 `evidence_store_writer.dart`，新增 Monitor 摘要读取或聚合编排进入 `evidence_store_history.dart`，新增日期趋势、问题分类、失败聚类或节点耗时聚合进入 `evidence_store_aggregations.dart`，新增 Run Detail 解析或截图资产读取进入 `evidence_store_detail.dart`，路径清洗、文件名清洗和轻量字段解析进入 `evidence_store_helpers.dart`；不得把本地 evidence、Monitor 聚合和截图资产读取重新堆回单个 `evidence_store.dart`。

`studio_runtime.dart` 必须保持公共入口稳定。新增能力先判断是模型、设备动作、会话、证据、存储、控制器行为还是 workflow 引用校验；不要把 Runtime 逻辑写回 Flutter 页面文件。模型新增时必须进入对应 `runtime_*_models.dart` 分片，不能把设置、依赖检查、运行历史、运行详情、执行快照和执行器内部状态重新堆回 `runtime_models.dart`。控制器新增命令必须进入对应 extension 分片：本机环境、Appium 和设备会话进入 `runtime_appium_commands.dart`，设备预览截图、手势和输入进入 `runtime_device_preview_commands.dart`，工作流启动、停止和暂停收口进入 `runtime_run_commands.dart`，当前 workflow 项目命令进入 `runtime_workflow_project_commands.dart`，子流程项目命令进入 `runtime_sub_workflow_project_commands.dart`，设置和收藏进入 `runtime_settings_project_commands.dart`，本地 evidence 读取进入 `runtime_evidence_project_commands.dart`。当前 Sub Workflow Store 归属 `workflow_stores.dart`，启动恢复、注册保存、当前流程转子流程和删除保存都由 Runtime 负责，Flutter 页面只消费脱敏摘要并触发受控命令。Sub Workflow 引用完整性归属 `runtime_workflow_references.dart`，保存当前 workflow 和注册子流程前必须复用同一校验，避免缺失引用进入运行态；Flutter Workflow 诊断层也可复用该公共校验器，但只能读取 snapshot 中的脱敏子流程摘要，不得反向调用设备或运行时命令。

V2.0 Monorepo 使用：

- FVM Flutter：使用本机当前 Flutter stable 版本

- `package.json`

## 7. Legacy Boundary

当前 Node / Web / CLI 进入 Legacy 范围。

允许：

- 作为迁移参考
- 作为旧 CLI 兼容入口
- 作为调试对照
- 作为 smoke 参考

禁止：

- V2.0 Flutter App 调用 Legacy Node 服务
- V2.0 新 UI 继续写进 Web Console
- V2.0 Runtime 依赖 Node Runner
- 用 Node Web API 作为 Flutter 和 Appium 的中间层

自动化门禁：

- `fvm dart run tool/v2_boundary_check.dart` 扫描 `apps/studio_mac/lib` 与 `packages/*/lib`。
- 扫描会拦截 Legacy Web Console、Legacy 坐标 CLI、Legacy Web API、Legacy npm 入口和 `node legacy/node/src/` 中间层调用。
- 该门禁进入 `fvm dart run tool/v2_verify.dart`；`npm run check` 和 `npm run smoke:v2-boundary` 只是兼容 wrapper。
- `fvm dart run tool/macos_build_smoke.dart` 执行 `apps/studio_mac` 的 Flutter macOS debug build，用于证明 V2.0 Mac App 可构建。
- Appium / WebDriver HTTP 调用仍归属 Dart `appium_client`，不被视为 Legacy Node 调用。

## 9. Verification

Monorepo 第一阶段至少需要：

- `melos bootstrap`
- `melos analyze`
- `melos test`
- `fvm flutter test`
- `fvm flutter build macos` 或等价本机构建检查

在真实设备不可用时：

- Appium Client 单元测试使用 fake HTTP server。
- Runtime 测试使用 fake Appium client / fake device actions；设备会话和预览动作测试进入独立 `device_session_test.dart`，本地 evidence store 聚合与详情读取测试进入独立 `evidence_store_test.dart`。
- Workflow DSL 测试离线运行。

## 10. Change Rules

- 新增 Dart package 必须说明职责和依赖方向。
- 任何 package 不得为了方便直接读取 Legacy Node 实现。
- 如果包边界造成明显开发成本，应优先调整包职责，而不是绕过依赖规则。
- `packages/*/README.md` 必须说明包职责、边界、公共入口和验证命令，不得保留 Dart / Flutter 模板 TODO 或与本项目无关的示例。
- Legacy Node 已物理迁移到 `legacy/node/src/`；后续只能维护 `legacy:*` 脚本，不得恢复无前缀 Node 产品入口。
