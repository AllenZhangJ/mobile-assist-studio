# iOS Assist Studio V4.0 Development Roadmap

## 0. TL;DR

- V4.0 用终局倒推，工程正推。
- 终局是本地视觉移动自动化工作站，融合 Airtest、Pyxelator、Appium Inspector 和 appium-mcp。
- 第一版必须 iOS 真机深验证，Android 真机完成最小冒烟。
- 每批都必须保持 Node 不回流、Runtime 不旁路、Project DSL 不分裂、证据不泄露。

## 1. Backward Target

V4.0 终局能力：

```text
连接设备
  -> 检查界面
  -> 录制动作
  -> 创建目标
  -> 编排流程
  -> 视觉定位
  -> 串行执行
  -> 失败暂停
  -> 本地报告
  -> AI 建议修复
```

## 4. Batch 2: iOS Deep Validation And Android Smoke

### Goal

让 V4.0 从一开始就是双平台闭环，不让 Android 成为文档里的占位符。

### Deliverables

iOS：

- iOS 真机连接。
- 截图。
- Tap / Swipe / Input。
- 基础 workflow 执行。
- 本地证据。

Android：

- ADB 当前设备发现。
- Appium UiAutomator2 session。
- Android 截图。
- Tap / Swipe / Input 冒烟。
- 基础 workflow 冒烟。
- logcat 摘要。
- Android 短诊断。

### Acceptance

- iOS 真机可完成完整基础流程。
- Android 真机至少完成连接、截图、点击、滑动、输入和一次基础 workflow。
- 两个平台失败时都能生成脱敏诊断。
- 两个平台都能写入本地 evidence。

### Current Implementation Slice

- Runtime 已具备 Android ADB 设备发现、未授权 / 离线 / 多设备阻断、serial 脱敏和 logcat 摘要清理。
- Device 本机环境检查已接入 Android ADB 准备项：可展示无设备、未授权、离线、多设备或 ADB 不可用的短提醒和下一步，但只作为 V4 Android 准备度提示，不阻断当前 iOS 一键连接。
- Runtime 已具备 Android UiAutomator2 adapter 的 fake Appium session、截图、Tap、Swipe、Input、日志和断开测试。
- 项目已固定 Appium `xcuitest` 与 `uiautomator2` 两个平台 driver 依赖；full smoke 自动准备会先检查所需 driver 是否已被 Appium 识别，缺失时在“驱动组件”项中阻断。
- Runtime 已具备跨平台 `MobileDriverSmokeRunner`，可把 driver 能力摘要、截图、可选动作、日志和失败兜底写入现有本地 evidence。
- 本地已提供 `npm run v4:ios-smoke` 作为 iOS 真机冒烟入口；默认只建会话、截图和留档，显式加 `-- --allow-actions` 才执行真实 Tap / Swipe / Input，加 `-- --workflow-basic --allow-actions` 可执行基础 Project DSL workflow。
- 本地已提供 `npm run v4:android-smoke` 作为 Android 真机冒烟入口；默认只建会话、截图和留档，显式加 `-- --allow-actions` 才执行真实 Tap / Swipe / Input，加 `-- --workflow-basic --allow-actions` 可执行基础 Project DSL workflow。当前若驱动不可达、无 Android 手机、未授权、离线或多设备阻断，会生成 `ANDROID_SMOKE_PREFLIGHT` 脱敏 Markdown / JSON 诊断；该诊断会被 readiness 索引，但不计作真实 Android smoke run。
- 本地已提供 `npm run v4:smoke:full` 作为双平台完整冒烟编排入口，会先用 Dart Runtime 能力自动准备本机 Appium 驱动、iOS 必要隧道和 Android ADB 可见性，再顺序执行 iOS 与 Android 的真实 Tap / Swipe / Input + 基础 Project DSL workflow，并在任一平台失败后仍继续生成 full smoke Markdown / JSON、readiness / completion audit 留档；当 Android 纳入 full smoke 且前置阻断时，会同步写入同 schema 的 `ANDROID_SMOKE_PREFLIGHT` 脱敏诊断，确保 readiness / acceptance 索引到最新 Android 阻断。`npm run v4:ios-smoke:full` 和 `npm run v4:android-smoke:full` 也复用同一个 full smoke 编排器，只跳过另一个平台，单平台排障时同样具备自动准备能力。iOS 18+ 隧道仍遵守系统授权边界：可先通过 Mac App 连接设备，或用 `npm run v4:ios-smoke:full:password-prompt` / `npm run v4:smoke:full:password-prompt` 从终端隐藏输入 Mac 密码启动本次 smoke 托管隧道；`password-stdin` 入口仅作为自动化备用；Android 准备只启动 ADB server 并检查唯一已授权手机，不执行设备动作。
- Runtime 已具备完整 Project DSL workflow 冒烟 evidence 闭环；Mac App Monitor 已接入 V4 验收卡，提供当前本地线索、最新 final acceptance 脱敏摘要和 full smoke / Android smoke / acceptance audit 命令复制入口；尚未完成 Android 真机现场冒烟留档。

## 7. Batch 5: Vision Core With Pyxelator And Airtest

### Goal

让视觉能力从概念变成可运行的目标解析层。

### Deliverables

- PyxelatorProvider。
- AirtestProvider。
- Image Template Resolver。
- Visual Assert。
- Visual Branch。
- Wait For Target。
- Tap Target。
- 置信度阈值。
- 低置信暂停。
- 视觉证据链。
- 模板测试工具。
- fixture screenshot 测试。

### Acceptance

- Pyxelator 可在 fixture 中定位模板。
- Airtest 可在 fixture 中完成视觉断言。
- VisionProvider 不直接点击设备。
- 低置信不会盲点。
- Run Detail 能展示视觉判断证据。

### Current Implementation Slice

- Runtime 已具备 `CompositeTargetResolver`，统一调度 VisionProvider，provider 只解析目标，不执行设备动作。
- Runtime 已具备 `CoordinateTargetProvider`，用于把坐标目标纳入 TargetResolver 统一语义。
- Runtime 已具备 `RegionTargetProvider`，可把区域目标解析为中心点和区域证据，provider 只返回结果，不执行点击。
- Runtime 已具备 `SelectorTargetProvider`，可在 Appium Source 可用时通过 `label=`、`value=`、`text=`、`type=` 或裸文本的受控短语法解析元素中心点；它不支持 XPath、CSS、脚本或任意表达式。
- Runtime 已具备 `TextTargetProvider`，可在 Appium Source 可用时通过 label / value 精确匹配文本并解析元素中心点；显式开启 Python 视觉增强后，`PythonOcrTextProvider` 可作为后续 OCR 解析层。
- Runtime 已具备 `PyxelatorFixtureVisionProvider`，可在小尺寸 PNG fixture 中完成图片目标模板匹配，输出中心点、区域、置信度和本地证据引用；超出本地操作量上限时会返回 unsupported，避免真实大截图卡住 Dart Runtime。
- Runtime 已具备 `AirtestFixtureVisionProvider`，以 fixture 方式表达视觉断言结果；它不运行 `.air` 文件、不启动 Python、不控制设备。
- Runtime 已具备短生命周期 `PythonVisionSidecarClient`、`PythonSidecarVisionProvider` 和 `PythonOcrTextProvider`，通过 stdin/stdout JSON 调用本地 Python 视觉脚本；provider 只返回坐标、区域、置信度和证据，不拥有点击能力。Python sidecar 已支持 `pyxelator`、`airtest` 和 `builtin` 三类图片后端：Pyxelator 后端会尝试常见模板定位 API，Airtest 后端会尝试 `airtest.aircv.find_template`，不可用时返回结构化 `unsupported` 并继续下一 provider，最终可落到内置 PNG 匹配兜底。OCR 文本后端优先尝试本机 `pytesseract`，缺少包、缺少系统 OCR 能力或 API 不匹配时只返回结构化不可用。Python 视觉由本机设置里的“视觉增强”显式开启，默认关闭，避免缺失环境时拖慢主流程。
- Visual Branch 已支持 `targetRef`：配置图片、selector 或 text 目标时会通过 TargetResolver 判断目标是否匹配；低置信、未找到或不支持会进入暂停态，不会继续后续动作。
- Wait For Target 已作为 Project DSL 节点进入 Runtime：节点会串行截图、通过 TargetResolver 解析目标，匹配后继续，超时、低置信、不支持或基础设施错误都会暂停，不执行点击。
- Tap Target 已支持通过 TargetResolver 解析图片、selector 和 text `targetRef`：匹配并返回坐标后才会串行点击并释放 pointer actions；低置信、未找到、不支持或基础设施错误都会进入暂停态，不盲点。
- 图片模板资产管理已落到 Runtime：目标库 JSON 只保存项目内相对 `imageRef`，模板 PNG 保存在本地 `targets/images`，解析时由 Runtime 临时读取并交给 TargetResolver。
- 目标库已具备图片目标“试找”工具：它只使用最近截图测试目标能否命中，不刷新截图、不启动驱动、不点击设备，并通过 Runtime 事件返回短中文结果。
- Flutter Workflow 节点库、顶部新增菜单、画布展示和 Inspector 已能识别 Wait For Target；缺失目标仍由 Runtime targetRef 校验拦截，不会把坏引用静默写入流程。
- Project DSL 已对 Tap、Visual Branch 和 Wait For Target 的 `targetRef`、置信阈值和轮询参数做基础形态校验；目标存在性仍由 Runtime 项目级 Target Library 校验负责。
- 已覆盖 Pyxelator fixture 匹配、Airtest fixture 断言、Python sidecar provider JSON 映射、Pyxelator / Airtest / builtin 后端分派、Python OCR text provider 映射、Source text 未命中后转 OCR、OCR 不可用时保留 Source 未命中、Python 内置后端真实脚本冒烟、Region target 解析和区域 Tap、Selector target source 解析和 selector Tap / Wait、Text target source 解析和 text Tap、Python 视觉开关持久化和设置抽屉回归、目标库图片试找工具、操作量上限、Visual Branch targetRef 通过与低置信暂停、Wait For Target 通过与低置信暂停、Tap Target 图片目标通过与低置信暂停、本地模板资产读写与路径安全、DSL targetRef / Wait For Target 参数校验、Workflow UI 插入回归。
- Airtest 报告心智已先落到 Runtime 本地报告模型；Monitor 报告面板和脱敏 JSON 复制导出已在 Batch 7 基础切片落地，完整 Evidence 保留策略升级继续留在 Batch 7 后续切片。

## 9. Batch 7: Evidence, Report And Monitor 4.0

### Goal

吸收 Airtest 报告思路，形成本地可复盘系统。

### Deliverables

- 本地报告。
- 失败分类。
- 视觉证据链。
- 节点耗时。
- 平台差异分析。
- 日志摘要。
- 截图胶片。
- 报告导出。
- Evidence 保留策略升级。

### Acceptance

- 每次运行都能复盘。
- 每次视觉判断都有证据。
- 报告不泄露完整设备标识和本机路径。
- Android 冒烟运行也能生成报告。

### Current Implementation Slice

- Runtime 已具备 `RunLocalReport` 本地报告模型，可从 `RunDetail` 派生 overview、issue、timeline、visualChecks、screenshots、logSummary 和 platform。
- 本地报告只读取现有 evidence 真源，不新增第二套报告存储，不影响连接、执行和截图写入链路。
- 本地报告导出 JSON 会过滤不安全截图路径，并脱敏本机路径、设备号、长 session 和本机驱动地址。
- `LocalRunEvidenceStore` 已实现 `RunReportReader` 和 `RunReportExporter`，`StudioRuntimeController` 已提供 `readRunReport` 与 `exportRunReport` 入口，供后续 Monitor、导出和 AI 解释复用。
- Monitor / Execute 共用的 Run Detail 抽屉已接入 `readRunReport`，展示“本地报告”复盘面板，并提供“复制报告”和“存报告”两个脱敏 JSON 入口。
- “存报告”由 Runtime 写入当前运行目录内的 `exports/`，不让 UI 扫描 evidence 文件或拼接本机路径；该导出文件随单次运行目录一起进入本地证据生命周期。
- 报告已具备基础平台差异摘要：从 smoke / session / logs 事件提取 iOS / Android / unknown、设备短名、脱敏设备号、系统版本、连接方式、动作开关和日志数量；Monitor 本地报告面板会展示短中文平台提示，旧记录缺字段时降级为未知。
- Evidence 读取历史前会先应用同一套保留策略；App 重启后打开记录页或刷新历史也会滚动清理过期本地证据。
- 已覆盖本地报告聚合、视觉检查、截图胶片、日志摘要、平台摘要、Runtime 读取入口、Runtime 文件导出、Monitor 报告 UI、报告复制导出、报告保存、读前清理和隐私脱敏测试。
- 更深的跨平台指标型分析仍可后续增强，但 Batch 7 基础 Evidence / Report / Monitor 4.0 闭环已完成。

## 10. Batch 8: AI / MCP Core

### Goal

让产品具备 Cursor for Mobile Automation 的受控智能能力。

### Deliverables

- MCP-compatible Tool Registry。
- Tool Permission Gate。
- 流程草稿生成。
- 失败解释。
- 目标建议。
- Locator 建议。
- 模板修复建议。
- 用户确认机制。
- AI 行为日志。

### Acceptance

- AI 不直接执行危险动作。
- AI 不绕过 Runtime。
- AI 默认不上传截图。
- AI 输出可追踪。
- 没有 AI 环境时核心自动化仍可用。

### Current Implementation Slice

- Runtime 已具备 MCP-compatible `AiToolRegistry`，登记读屏摘要、流程草稿、失败解释、目标建议、Locator 建议、模板修复建议和运行流程交接工具。
- Runtime 已具备 `AiToolPermissionGate`：只读和草稿工具可自动执行，危险工具必须用户确认，未知或禁用工具直接阻断。
- `StudioRuntimeController.invokeAiTool` 是唯一 AI 工具入口；它只读取 Runtime 状态、本地报告和脱敏 Inspector 摘要，或生成草稿，不直接点击、不启动运行、不保存 workflow、不写目标库。
- Command Center 已接入“智能”抽屉，可调用读屏摘要、流程草稿、目标建议、Locator 建议、失败解释、模板修复建议和运行交接；抽屉只展示/复制脱敏结果和审计日志，不把草稿自动保存到 Project DSL，也不直接执行设备动作。
- Device Inspector 已接入“智能建议”入口，可在检查界面后调用目标建议和 Locator 建议；结果只在面板内展示草稿，不写 Target Library、不保存 workflow、不触发设备动作。
- Monitor Run Detail 已接入“智能解释”入口，可基于当前运行的本地报告调用失败解释；结果只读展示，不读取截图画面、不写报告文件、不触发重跑或修复。
- `runWorkflow` 工具即使用户确认，也只返回 `runtime.startRun` 交接结果，由 UI 和 Runtime 主按钮继续处理，AI 不拥有执行权。
- 读屏摘要只返回连接、运行、平台、能力、元素数量和是否有截图等摘要，不返回截图 base64。
- 失败解释和模板修复建议复用 Batch 7 本地报告模型，输出继续经过路径、设备号、session 和本机驱动地址脱敏。
- 目标建议和 Locator 建议只基于已脱敏元素树生成草稿，使用项目允许的 `label=`、`value=`、`type=` 短语法，不生成 XPath、CSS、脚本或任意表达式。
- Runtime snapshot 已包含最近 AI 行为审计日志；日志只记录工具、风险、状态、确认状态和短消息，最多保留最近 80 条，避免长期积累敏感上下文。
- 已覆盖权限门禁、危险工具确认、读屏摘要不泄露截图、目标 / Locator 草稿不持久化、失败解释脱敏、模板建议、流程草稿不写入当前 workflow、Command Center 智能抽屉、Device Inspector 智能建议、Run Detail 智能解释、脱敏结果复制和运行交接不直接启动的 Runtime / Flutter 测试。
- 当前 Batch 8 已具备 Core + Command Center + Device Inspector + Run Detail 可见入口闭环；后续继续增强时不得改变 AI 不直接执行动作的边界。

## 11. Stop Conditions

任一批次出现以下情况必须暂停：

- Flutter / Dart 主路径重新调用 Node。
- 第三方能力绕过 Runtime。
- iOS 基础流程退化。
- Android 首版冒烟无法闭环但仍继续堆 UI。
- 视觉低置信仍自动点击。
- 复制第三方源码但未登记许可证。
- 运行证据写入完整设备标识、本机路径或隐私信息。

- `npm run v4:smoke-readiness` 生成本地脱敏 smoke readiness Markdown 留档和同名 JSON 摘要，并索引最近 full smoke JSON 的阻断项、步骤结果和 Android smoke 前置诊断，方便人类复盘与 AI / CI 后续审计。

- `npm run v4:smoke-archive` 生成本地脱敏 smoke archive 清单，索引 readiness、full smoke、平台 run 和截图；最终验收可用 `npm run v4:smoke-archive:final` 要求完整 full smoke、截图、iOS run 和 Android run 均已留档。
- `npm run v4:acceptance-audit` 汇总 readiness、archive、completion audit 和 archive final 的本地审计报告，并在 JSON / Markdown 中嵌入本机状态摘要、Batch 0-8 验收索引、Android 前置诊断、截图留档索引、终验门禁缺口、留档数量、现场补验清单和基于结构化证据生成的命令级下一步；截图索引只展示 archive 里的脱敏相对路径和大小，不读取图片内容、不展示绝对路径；终验门禁缺口必须把 archive warnings 转成“当前问题 / 通过标准 / 建议命令”的结构化表格；最近 full smoke 若阻断在 iOS 隧道，双平台补验命令必须给出 `npm run v4:smoke:full:password-prompt`，避免普通 full smoke 再次卡密码；最终验收可用 `npm run v4:acceptance-final` 作为统一门禁。
- `npm run v4:smoke:full` 作为现场双平台完整冒烟入口，默认检查 Appium 平台 driver，自动准备本机驱动、iOS 隧道和 Android ADB 可见性，并在报告中写入“自动准备”区块；Android 纳入 full smoke 且前置阻断时，同步生成 `ANDROID_SMOKE_PREFLIGHT` JSON / Markdown，供 readiness / acceptance 展示最新 Android 阻断。需要本机隧道密码时优先用 `npm run v4:ios-smoke:full:password-prompt` 或 `npm run v4:smoke:full:password-prompt`，密码通过终端隐藏输入进入 sudo stdin，不写日志、不进报告；`password-stdin` 入口仅作为自动化备用。单平台排障可用 `npm run v4:ios-smoke:full` 或 `npm run v4:android-smoke:full`，二者同样走 full smoke 编排器并自动准备，只跳过另一个平台。
- Mac App Monitor 的 V4 验收卡只基于 Runtime 快照展示当前平台、本地运行数量、Batch 0-8 进度、Android 留档数量、问题数量和下一步提示；Runtime 启动和 Monitor 刷新时会读取最新 `FINAL_ACCEPTANCE` 脱敏摘要，并对用户可见文本二次脱敏，UI 不直接扫描文件、不读取截图、不展示报告路径或底层命令输出。它提供 full smoke、双平台密码版、iOS 密码版、Android smoke、acceptance audit、acceptance final 和“现场路线”复制入口。现场路线默认覆盖 Android 单平台 smoke、双平台 full smoke、最终验收；当 final acceptance 下一步包含 iOS 隧道补验时，会优先包含 `npm run v4:ios-smoke:full:password-prompt`。复制命令不直接启动设备动作，不读取本地 smoke 文件，不替代最终验收门禁。
- Device 本机指引提供“安卓准备”卡，面向现场用户提示开启 USB 调试、插线、点允许和复制 Android smoke 命令；该卡只复制命令，不启动 ADB、不启动驱动、不执行真机动作。Device 本机环境检查会同步显示 Android ADB 准备状态，ADB 不可用或手机未就绪只进入提醒态，不阻断 iOS 连接。
