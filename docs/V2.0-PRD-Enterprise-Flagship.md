# iOS Assist Studio V2.0 PRD: Enterprise Flagship

## 0. TL;DR

- V2.0 的正式定位是“企业级体验的本地自动化工作站”。
- 产品服务本机 Mac + 单台 USB iPhone 的真机辅助自动化场景。
- Web 控制台是主入口，CLI 仅保留为兼容调试入口。
- Appium / WDA 是唯一主驱动链路，OCR、CV、pymobiledevice3 和 accessibility source 只作为辅助能力。
- V2.0 不做 SaaS、多租户、权限系统、审计系统、云端协作、多设备并发或脚本平台。

## 1. Product Definition

`iOS Assist Studio V2.0` 是面向本地 iPhone 自动化的 Enterprise Local Workstation。

它的目标不是把 Appium 暴露给用户，而是把设备连接、WDA 会话、录制、工作流、执行、监控和诊断组织成一个可理解、可维护、可扩展的专业工作区。

产品类比：

- Cursor：IDE 级工作区、Inspector、命令中心和项目管理体验
- Figma：可视化编辑、画布、属性面板和状态反馈
- n8n：节点式工作流编排体验
- Xcode Inspector：设备和界面状态检查体验

这些类比只定义体验方向，不代表引入对应平台作为硬依赖。

## 2. Positioning Boundary

V2.0 是：

- 本地个人或小型团队使用的专业工作站
- 企业级信息架构和交互体验
- 单设备优先的真机自动化编排工具
- 面向不会 Appium / WDA / 编程用户的可视化操作环境
- 面向自动化工程师的可诊断、可验证、可扩展底座

V2.0 不是：

- SaaS 平台
- 云端自动化平台
- 企业管理系统
- 多租户平台
- 权限或审计系统
- 云端协作工具
- 多设备调度或群控系统
- 开放脚本执行平台

## 3. Target Users

### 3.1 Primary User

Primary User 是不会 Appium / WDA 的运营或 QA。

他们需要：

- 看懂当前设备是否可用
- 不理解 WDA endpoint 也能连接设备
- 不理解坐标也能录制动作
- 不写代码也能配置基础流程
- 出错时知道下一步该做什么

### 3.2 Secondary User

Secondary User 是自动化工程师。

他们需要：

- 明确的 DSL 真源
- 可追踪的运行事件
- 可解释的失败现场
- 可验证的执行安全边界
- 可扩展的视觉、诊断和工作流模块

### 3.3 Interaction Priority

所有交互优先级遵循：

1. 可理解性
2. 灵活性
3. 工程能力

任何需要用户理解坐标、session、WDA endpoint、source XML 的信息，都应默认隐藏在 Drawer、Inspector 或 Advanced 面板中。

## 4. Product Goals

V2.0 需要达成：

- 让用户通过 Web 工作站完成设备连接、录制、编排、执行和监控
- 让旧 A-F 坐标序列自然升级为基础工作流模板
- 让执行从“命令行脚本”升级为“可观察的节点运行”
- 让失败从“终端错误”升级为“状态、现场、原因和恢复路径”
- 让后续视觉识别、OCR、自愈和 AI 辅助能力有清晰扩展点

## 5. Success Criteria

V2.0 体验成功的判断标准：

- 非工程用户能完成一次设备连接、录制、执行和查看结果
- 主界面只展示 Ready、Warning、Error 等摘要状态，不堆叠技术细节
- 用户能从一次失败运行定位到节点、动作、截图、日志和原因摘要
- 旧 A-F 序列可作为基础模板导入或执行
- 所有核心运行仍保持单设备、串行、安全停止
- 无真实设备时仍可通过本地 validator、dry-run 和 smoke 验证 DSL 与控制台基础逻辑

## 6. Functional Scope

### 6.1 L1 Dashboard

目标：打开产品第一眼知道当前情况。

能力：

- 总览设备连接状态
- 总览工作流数量和最近运行状态
- 展示今日执行次数、成功率、失败趋势
- 入口跳转到 Device、Workflow、Execute 和 Monitor

原则：

- Dashboard 只做决策摘要，不承载复杂配置
- KPI 点击后进入对应模块，而不是在 Dashboard 内展开技术细节

### 6.2 L2 Device

目标：设备管理中心。

能力：

- 初始化并连接当前 USB iPhone
- 展示当前设备摘要、连接方式、电量、系统版本和 WDA 状态
- 展示实时或准实时设备预览
- 支持点击、双击、拖动、滚动、缩放等交互入口
- 点击屏幕时显示十字准星和动作反馈

边界：

- V2.0 运行时只允许一台 USB iPhone
- Device List 可以在 IA 中预留，但单设备模式下应自动弱化或隐藏列表

### 6.3 L3 Recorder

目标：录制用户行为。

能力：

- 开始、暂停、停止录制
- 在设备预览中捕获 Tap、Wait、Swipe 等动作
- 实时生成动作时间轴
- 默认隐藏坐标细节
- 点击动作后在 Drawer 中查看坐标、截图、元素信息、等待时间和重试策略

优先级：

- 优先保证录制成功率
- 工作流能力丰富度排在录制稳定性之后

### 6.4 L4 Workflow

目标：可视化编辑流程。

能力：

- 提供 Visual View 和 Source View
- Visual View 使用节点画布表达流程
- Source View 使用 Project DSL 表达同一工作流
- 支持 Start、Tap、Wait、Swipe、Input、Condition、Loop、Snapshot、Visual Branch、Catch、Sub Workflow、End 等节点方向
- 节点双击或选中后打开 Node Inspector

边界：

- Project DSL 是唯一真源
- Visual View 和 Source View 必须映射到同一 DSL
- 禁止任意 JavaScript、Python 或用户自定义脚本执行
- V2.0 不做脚本平台

### 6.5 L5 Execute

目标：运行工作流。

能力：

- 配置执行模式、循环次数、随机延迟、超时时间、失败重试和执行设备
- 支持单次、循环和受控无限循环模式
- 开始执行前进行确认
- 执行中进入 Running Mode
- 展示当前节点、总节点、进度和预计剩余时间
- 支持安全停止

边界：

- 始终串行执行
- 停止任务采用安全停止
- 无限循环必须有人工可控停止和异常上限

### 6.6 L6 Monitor

目标：运营监控中心。

能力：

- 展示本地成功率、失败率、平均耗时和执行次数
- 展示 7 天、30 天、90 天趋势
- 展示本地执行记录
- 点击记录打开详情 Drawer
- 详情中展示执行路径、节点耗时、失败原因和本地截图证据引用

边界：

- Monitor 只分析本地历史运行记录、本地日志、本地截图证据和本地失败信息
- 不上传云端
- 不接远程分析服务
- 不实现遥测系统

## 7. Technical Boundaries

主执行链路：

- Appium
- WebDriverAgent
- XCUITest

辅助能力：

- OCR
- CV
- pymobiledevice3
- accessibility source

约束：

- 辅助能力不得形成第二套执行体系
- 视觉层不得绕过运行时直接点击
- 设备诊断能力不得改变主会话状态机
- Appium / WDA 仍是点击、输入、会话和元素交互的唯一主驱动

## 8. Data And Privacy

数据只保存在本机。

允许本地保存：

- 运行历史
- 诊断日志
- 截图证据
- 工作流配置
- 失败摘要

禁止：

- 上传云端
- 写入真实完整设备标识
- 写入完整证书主体
- 写入真实账号
- 写入生产环境地址
- 在文档或默认 UI 中暴露敏感路径

所有示例必须使用脱敏数据。

## 9. AI Extension Boundary

V2.0 不包含 AI Agent 自动完成全部流程。

预留扩展点：

- Workflow Generator
- Visual Locator Generator
- Self Healing
- AI Inspector

这些能力属于 Future，不进入 V2.0 当前实现范围。

## 10. Product Principles

- Summary First, Detail Later
- Local First, No Cloud
- Single Device First
- Appium / WDA First
- Visual Assistance, Not Visual Dependency
- DSL First, No Script Platform
- Record Reliability Before Workflow Richness
- Safety Stop Before Speed
- Evidence Before Guessing

## 11. Change Rules

- 任何 V2.0 实现不得突破本文产品边界。
- 如果后续要进入多设备、云端、AI Agent 或脚本执行平台，必须先创建新的边界文档。
- 如果 UI 原型与本文冲突，以本文边界为准，再调整 IA 或 UX Spec。
