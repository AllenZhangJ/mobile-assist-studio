# iOS Assist Studio V3.0 Architecture: Cross-Platform Runtime

## 0. TL;DR

- V3.0 继续使用 Flutter Desktop Mac App + Dart Runtime，但 Runtime 要从 iOS 专属实现升级为跨平台移动设备运行时。
- V3.0 的核心新增架构是 `MobileDeviceDriver` 抽象、平台 adapter、Target Library、Target Resolver、Device Resource Lock 和分层结果模型。
- iOS adapter 继续使用 Appium / WDA / XCUITest；Android adapter 优先使用 Appium UiAutomator2 + ADB，scrcpy、Airtest、OCR/CV 作为可选辅助能力。
- Workflow DSL 表达用户意图，不直接绑定 WDA 或 ADB；平台差异由 Runtime capability report 和 adapter 处理。
- 任何辅助视觉、OCR、scrcpy 或 Airtest 能力都不得绕过 Runtime 状态机直接执行设备动作。

## 1. Architecture Decision

V2.0 主链路：

```text
Flutter Desktop Mac App
  -> Dart Runtime
  -> Dart Appium / WebDriver Client
  -> Appium Server
  -> WebDriverAgent
  -> iPhone
```

V3.0 目标链路：

```text
Flutter Desktop Mac App
  -> Dart Runtime
  -> Mobile Device Driver Abstraction
      -> iOS Adapter
          -> Appium / WDA / XCUITest
          -> iPhone
      -> Android Adapter
          -> Appium UiAutomator2 / ADB
          -> Android Device
  -> Target Resolver
      -> Selector / Coordinate / Image / Region / OCR
  -> Local Evidence / Monitor
```

V3.0 不采用：

```text
Flutter App
  -> 自建 Node API
  -> 多设备调度服务
  -> 云端设备池
```

## 2. Design Goals

跨平台 Runtime 必须满足：

- iOS 既有能力不退化。
- Android 能力不是外挂脚本，而是同一 Runtime 状态机下的 adapter。
- Workflow DSL 不写死 WDA、ADB、Airtest 或 scrcpy。
- Device Preview、Recorder、Workflow、Execute 和 Monitor 共享当前设备资源状态。
- 目标定位失败、视觉低置信和平台能力缺失都有可诊断输出。
- 单设备、串行执行、安全停止和本地隐私边界继续成立。

## 3. Core Runtime Concepts

### 3.1 Mobile Platform

平台枚举：

```text
ios
android
unknown
```

平台只描述当前设备类别，不描述驱动实现。驱动实现由 capability report 暴露。

### 3.2 Mobile Device

`MobileDevice` 是当前设备摘要模型。

字段方向：

- platform
- displayName
- maskedIdentifier
- osVersion
- connectionKind
- batteryLevel
- currentApp
- screenSize
- driverStatus
- resourceState

禁止：

- 在主 UI 展示完整 UDID、ADB serial、session、endpoint 或本机路径。

### 3.3 Device Resource Lock

当前设备资源锁负责协调：

- remoteControl
- recording
- workflowExecution
- targetCapture
- diagnostics

锁状态方向：

```text
idle
remoteControl
recording
running
paused
stopping
diagnosing
error
```

规则：

- 运行中不能写 workflow。
- 运行中不能写 target definition。
- 远控动作和自动化动作不能并发下发。
- 录制可以捕获本地动作，但是否执行设备动作必须由用户明确触发。
- paused 是人工介入态，不是失败态。

### 3.4 Mobile Device Driver

`MobileDeviceDriver` 是 Runtime 使用的平台中立能力接口。

能力方向：

- discoverCurrentDevice
- connect
- disconnect
- heartbeat
- captureScreenshot
- getPageSource
- tap
- swipe
- inputText
- launchApp
- stopApp
- pressHome
- collectLogs
- samplePerformance
- releaseActions

设计要求：

- 所有动作先经过 Runtime 前置校验。
- 所有动作返回结构化结果。
- Tap / gesture 失败时仍尽力 release actions 或平台等价清理。
- 平台不支持的能力必须返回 capability error，而不是静默降级。

### 3.5 Platform Adapter

iOS adapter：

- Appium / WebDriver session
- WDA screenshot
- WDA page source
- W3C pointer actions
- XCUITest app lifecycle
- iOS syslog / device log 摘要作为后续能力

Android adapter：

- Appium UiAutomator2 session
- ADB device discovery
- ADB shell / logcat / dumpsys
- Screenshot
- W3C actions 或 ADB input 兜底
- App package / activity lifecycle
- scrcpy remote control optional

适配原则：

- adapter 实现平台差异。
- Runtime 暴露统一状态和事件。
- UI 不直接调用 adapter。

### 3.6 Capability Report

每个 adapter 必须提供 capability report。

字段方向：

- screenshot
- tap
- swipe
- input
- pageSource
- selectorTargets
- imageTargets
- ocrTargets
- appLifecycle
- logs
- performance
- remoteControl

用途：

- Device Readiness
- Execute Preflight
- Workflow validator 的平台相关诊断
- Target Library 的可用性提示

## 4. Target Library Architecture

### 4.1 Target Definition

Target 是用户可管理的移动 UI 目标。

目标类型：

```text
coordinate
image
region
text
selector
app
composite
```

基础字段方向：

- targetId
- name
- platformScope
- type
- sourceScreenshotRef
- strategy
- threshold
- fallback
- privacyLevel
- createdAt
- updatedAt

Target Library 是本地项目数据，不属于运行证据；运行证据只引用 targetId、目标摘要和解析结果。

### 4.2 Target Resolver

Target Resolver 把 Target Definition 解析为可执行动作目标。

输入：

- current screenshot
- optional page source
- current platform
- target definition
- runtime context

输出：

- resolved point
- resolved region
- confidence
- evidence summary
- failure reason

规则：

- 低置信不执行点击。
- 找不到目标默认进入 paused 或 failed，取决于节点策略。
- OCR / CV 只产出解析结果，不直接执行动作。
- selector 和 image 可组合，优先使用更稳定策略。

### 4.3 Target Strategy Priority

默认推荐顺序：

1. 平台 selector 可用时优先 selector。
2. selector 不可用时尝试 image / region。
3. 动态文本使用 OCR。
4. coordinate 只作为快速录制和显式兜底。

用户可在 Target Inspector 中调整策略顺序。

## 5. Workflow DSL Changes

V3.0 DSL 应保持 V2.0 的安全模型：

- JSON Project DSL 仍是真源。
- Condition 只允许读取 `context.xxx`。
- 不开放 eval。
- 不开放任意脚本。
- 保存和执行前必须 validator 通过。

新增模型方向：

- `targetRef`
- `appRef`
- `platformRequirements`
- `assertion`
- `resultPolicy`

新增节点方向：

- Launch App
- Stop App
- Tap Target
- Wait For Target
- Assert Target
- Assert Text
- Collect Logs
- Collect Performance
- Manual Checkpoint

兼容规则：

- 旧 Tap 坐标节点仍可运行。
- Recorder 新生成的节点优先使用 targetRef。
- 旧 Workflow 可通过迁移工具把常用坐标封装成 coordinate target。

## 6. Execution Model

### 6.1 Run Status

任务状态描述执行是否完成：

```text
idle
running
paused
stopping
completed
failed
stopped
infrastructureError
```

### 6.2 Test Result

测试结果描述业务断言是否通过：

```text
notEvaluated
passed
assertionFailed
inconclusive
```

### 6.3 Issue Category

问题类型描述用户下一步应该修哪里：

```text
device
driver
target
workflow
assertion
visual
timeout
manual
unknown
```

规则：

- 设备断线不等于断言失败。
- 目标找不到不等于业务失败，除非节点声明为 assertion。
- 低置信视觉判断默认是 paused 或 inconclusive。
- 用户手动停止是 stopped，不是 failed。

## 7. Evidence Architecture

V3.0 evidence store 在 V2.0 基础上扩展：

- target resolution events
- OCR summaries
- image match summaries
- platform log slices
- performance samples
- app lifecycle events
- assertion results

证据引用原则：

- 保存截图引用和安全摘要。
- 不保存完整 page source 到长期证据，除非用户显式导出且脱敏。
- 不保存输入明文，默认只记录长度和字段摘要。
- 日志必须裁剪、脱敏和按运行关联。

## 8. Package Direction

V3.0 可在现有 Monorepo 中逐步演进。

可能的 package 方向：

```text
packages/mobile_driver        # 平台中立驱动接口和 capability 模型
packages/target_library       # Target models, JSON, validator, resolver contracts
packages/appium_client        # 继续承载 Appium / WebDriver HTTP client
packages/studio_runtime       # 编排 driver、targets、workflow、evidence
packages/workflow_dsl         # DSL 模型和校验
packages/studio_design_system # UI 基础组件
```

也可以先在 `packages/studio_runtime` 内以分片落地，再在边界稳定后抽包。

禁止：

- `target_library` 依赖 Flutter UI。
- `mobile_driver` 依赖具体页面。
- Android adapter 调用 Legacy Node。
- Flutter 页面直接调用 ADB、WDA、scrcpy 或 Airtest。

## 9. Android Technology Boundary

Android 第一优先级：

- Appium UiAutomator2
- ADB discovery
- ADB logcat
- ADB dumpsys
- screenshot
- app package lifecycle

Android 第二优先级：

- scrcpy 投屏增强
- Airtest 图像识别辅助
- OpenCV 模板匹配
- OCR 引擎

约束：

- scrcpy 只负责远控或视频流，不负责绕过 Runtime 动作锁。
- Airtest 可作为 Target Resolver 或 Android 辅助动作适配，但不能取代 Project DSL 和 Runtime runner。
- OCR/CV 必须通过目标解析和证据链进入系统。

## 10. iOS Preservation Boundary

V3.0 不得破坏 V2.0 iOS 边界：

- Appium / WDA / XCUITest 仍是 iOS 主驱动。
- 不绕过 Developer Mode、证书信任或 WDA 签名。
- 不把 pymobiledevice3 隧道作为 WDA session 就绪依据。
- 不把视觉能力作为 iOS 点击主路径。
- 不长期暴露完整设备标识、session、endpoint 或 source XML。

## 11. Migration Strategy

迁移顺序：

1. 引入平台和 capability 模型，但 iOS 行为不变。
2. 抽出 `MobileDeviceDriver` 接口，让 iOS adapter 包住现有 Appium/WDA 能力。
3. 引入 Device Resource Lock，统一 Device / Recorder / Execute 前置判断。
4. 引入 Target Definition 模型和本地 Target Store。
5. 让 Workflow 支持 targetRef，但保留坐标节点。
6. 接入 Android 最小 adapter。
7. 扩展 Target Resolver 到 image / OCR。
8. 扩展 evidence 和 Monitor 结果模型。

## 12. Verification Strategy

每个阶段必须支持无真实设备验证：

- driver interface fake tests
- iOS adapter fake Appium tests
- Android adapter fake Appium / fake ADB tests
- Target Library JSON / validator tests
- Target Resolver unit tests with fixture screenshots
- Workflow DSL migration tests
- Runtime resource lock tests
- Evidence aggregation tests
- Flutter widget tests for platform and target states

真实设备验证只作为补充，不得成为唯一门禁。

## 13. Open Questions

- Android 第一版是否必须接 scrcpy，还是先用 screenshot refresh 达成闭环？
- OCR 引擎优先本地 Tesseract，还是先接可替换接口和 fixture 测试？
- Target Library 是否独立 package，还是先在 Runtime 内稳定模型？
- V3.0 是否正式更名为 Mobile Assist Studio？
- 本地报告是否需要导出 HTML，还是先在 Monitor 内展示？

这些问题不阻塞 V3.0 真源落地，但实现前必须在对应 ADR 或专题文档中收口。
