# iOS Assist Studio V2.0 Flutter Desktop Architecture

## 0. TL;DR

- V2.0 主入口正式迁移为 Flutter Desktop Mac App。
- V2.0 不再以当前 Web 看板作为产品入口。
- V2.0 不新增自建 Node 中间 API 层。
- Flutter / Dart Runtime 直接管理 Appium 进程，并通过 WebDriver / Appium HTTP 协议调用 Appium。
- Appium / WDA 仍是唯一主驱动链路。
- 现有 Node / Web / CLI 只作为 Legacy、Debug、迁移参考和短期兜底，不允许成为 V2.0 产品路径依赖。

## 1. Architecture Decision

V2.0 采用：

```text
Flutter Desktop Mac App
  -> Dart Runtime
  -> Dart Appium / WebDriver Client
  -> Appium Server
  -> WebDriverAgent
  -> iPhone
```

不采用：

```text
Flutter Desktop Mac App
  -> 自建 Node Web API / Node Runner
  -> Appium Server
  -> WebDriverAgent
  -> iPhone
```

核心原因：

- V2.0 是 Mac App，不是浏览器控制台。
- 用户操作应表现为调用本地 App 方法，而不是调用自建 Web API。
- 当前 Node/Web 层继续作为主路径会造成架构漂移。
- Appium 本身已经是标准自动化服务进程，不需要再包一层自建 Node 服务。

## 2. Version Boundary

V2.0 第一版使用当前本机 FVM Flutter 版本：

- Flutter stable：3.35.7
- Dart：3.9.2
- Target：macOS Desktop
- Minimum macOS：macOS 13+

原则：

- 第一阶段不主动升级 Flutter SDK。
- 先稳定 Mac App 架构、Monorepo、Runtime 和 Appium Client。
- SDK 升级应作为独立任务评估，不与架构迁移混做。

## 3. Product Entry Boundary

V2.0 的唯一主产品入口：

- `apps/studio_mac`

允许存在但不作为产品入口：

- Legacy Node CLI
- Legacy Web Console
- Legacy smoke scripts
- 旧配置和旧序列迁移工具

禁止：

- Flutter App 调用自建 Node Web API。
- Flutter App 依赖旧 Web Console 状态接口。
- V2.0 新能力继续围绕当前 Node Web 看板增长。
- 用 Node 服务作为 Dart Runtime 的长期中间层。

## 4. Runtime Ownership

Dart Runtime 承担 V2.0 核心业务运行时。

职责包括：

- 设备连接状态
- 结构化连接诊断
- Appium 进程生命周期
- WebDriver session 生命周期
- 工作流运行
- 串行执行
- 安全停止
- 挂起与继续
- 运行事件
- 本地证据
- 本地监控数据

Appium 负责：

- 接收 WebDriver / Appium 命令
- 驱动 XCUITest Driver
- 管理 WDA 通信
- 执行 iPhone 上的自动化动作

WDA 负责：

- 真机上的底层自动化代理
- 截图、点击、输入、页面源等设备能力

## 5. Flutter App Responsibilities

Flutter Desktop App 负责：

- L1 Dashboard
- L2 Device
- L3 Recorder
- L4 Workflow
- L5 Execute
- L6 Monitor
- Top Status Bar
- Side Nav
- Workspace Area
- Global Command Center
- Bottom Console
- Drawer / Modal / Toast / Skeleton
- 本地设置
- 本地文件选择
- 本地运行可视化

Flutter UI 不直接包含设备业务细节。

UI 应通过 ViewModel / Controller 调用 Runtime 层，由 Runtime 层统一处理 Appium、Workflow、Evidence 和状态机。

当前 Flutter App 代码组织遵循低耦合高内聚：

- `main.dart` 只保留 Flutter 启动引导，不再承载工作区依赖或 `part` 路由。
- `studio_mac.dart` 是 Flutter App 的公开 Dart 库入口，启动脚本、测试和后续集成应统一从这里导入 `StudioMacApp`。
- `src/studio_mac_workspace.dart` 是当前阶段的工作区库边界，集中承载仍需共享私有上下文的 UI 分片；后续 Feature Library 继续从这里向外拆。
- `src/app/` 承载 Flutter App 根组件。
- `src/shell/` 承载 Shell 生命周期、命令中心、导航、Settings Drawer、Workspace 调度和 Bottom Console。
- `src/shell/top_status_bar.dart` 只承载顶部摘要状态栏。
- `src/shell/command_center.dart` 只承载 Global Command Center 的命令模型、弹窗、搜索和键盘选择。
- `src/shell/command_center_results.dart` 只承载命令中心结果行、结果图标、结果文本和搜索空态；结果 UI 不生成命令、不执行命令、不读取 Runtime。
- `src/shell/command_center_actions.dart` 只承载命令列表生成、状态详情打开、本机检查和剪贴板动作入口；导航命令必须调用 Shell 自身的导航方法，不得在 actions 分片直接修改 Shell 状态；新增命令先判断是否是导航、只读详情、本机检查或安全复制，不得重新堆回 Shell 主文件。
- `src/shell/command_center_diagnostics.dart` 只承载命令中心可复制的脱敏诊断摘要；摘要只读取 Runtime snapshot，不连接设备、不启动 Appium、不写 workflow、不展示截图、路径、端点或完整标识。
- `src/shell/status_detail_drawer.dart` 只承载状态胶囊按钮、状态详情抽屉和状态详情卡片；连接、驱动和运行状态的短文案、色调、图标和下一步建议不得写在抽屉里，统一复用 `src/shared/status_presenters.dart`。
- `src/shell/bottom_console.dart` 只承载 Bottom Console 外壳状态、展开收起、当前标签、级别筛选、复制和清空动作。
- `src/shell/bottom_console_controls.dart` 只承载 Bottom Console 标签按钮、级别筛选按钮和对应枚举。
- `src/shell/bottom_console_views.dart` 只承载 Bottom Console 日志、错误、检查、网络和调试内容视图。
- `src/shell/bottom_console_text.dart` 只承载 Bottom Console 事件级别、脱敏短中文、折叠摘要和复制文本 helper。
- `src/features/` 按 L1-L6 页面归属拆分，页面入口只组合页面级状态和主要区域。
- `src/shared/` 承载跨页面状态 presenter、状态色、Workflow 状态、准备度、Runtime 节点短中文名、格式化、普通文本复制反馈、文案 helper 和 App 侧私有兼容壳；一级 Surface 与二级 Inset Surface 的真实视觉实现归属 `packages/studio_design_system`，带状态色边框的本地紧凑卡片统一使用 App 侧 `_ToneBorderSurface` 兼容壳。
- `src/shared/status_presenters.dart` 只承载连接、驱动和运行状态到短中文文案、状态色、图标和下一步建议的映射；Top Status、状态详情、设备摘要、设置、控制台和执行摘要不得各自维护重复 switch。
- `src/shared/connection_diagnostic_card.dart` 只承载连接失败后的共享短诊断卡；Device 和 Execute 可复用该卡展示“原因”和“下一步”，并提供“复制诊断”的脱敏摘要入口，不得在页面内各自拼接 Appium / WDA 长错误。
- 共享连接诊断卡必须把 USB 连接问题压缩为“USB”短标签和数据线连接动作；当本机驱动未识别已绑定手机时，短标签使用“驱动”，下一步使用“保持解锁，点连接设备。仍失败就重插线。”；不展示 `localNetwork`、`transportType`、完整设备标识、`[device]` 脱敏占位符或 Appium 原始错误；复制摘要也不得包含本机路径、本地端点、完整设备标识、完整 session 或原始 WebDriver payload；脱敏占位统一使用中文短词，如“[标识]”“[本机路径]”“[本机地址]”，避免用户误解为真实配置值；Device 和 Execute 必须共用这套展示，不各自维护连接问题标签。
- Shared readiness helper 按路由说明、依赖状态、驱动、设备、会话、流程和图标拆分：`readiness_helpers.dart` 与 `readiness_entry_helpers.dart` 只保留拆分路由说明，`dependency_status_helpers.dart` 只承载 dependency report 的检查时间、短状态、色调和图标映射，`readiness_driver_entries.dart` 只承载驱动服务准备项，`readiness_device_entries.dart` 只承载 USB 手机和开发者信任准备项，`readiness_session_entries.dart` 只承载 WDA 手机会话和安全截图准备项，`readiness_workflow_entries.dart` 只承载流程校验准备项，`readiness_icon_helpers.dart` 只承载准备度图标映射。新增准备度文案先判断属于依赖状态、驱动、设备、会话、流程还是图标，不得重新堆回单个 helper 文件。会话准备项在错误态必须优先消费 `lastConnectionDiagnostic` 的摘要、下一步、短标签、图标和色调，不得与连接诊断卡各自维护两套错误提示。
- Monitor 共享 helper 已按职责拆分：`monitor_status_helpers.dart` 只保留路由说明，`monitor_analysis_helpers.dart` 只承载问题分类、原因文案和色调，`monitor_history_filter_helpers.dart` 只承载运行记录筛选、搜索、关联记录和筛选按钮元信息，`monitor_event_trace_helpers.dart` 只承载 Run Detail 事件 / 节点轨迹筛选、摘要和状态色。后续新增 Monitor 共享逻辑必须先判断这三类归属，不得重新堆回综合 helper。
- `src/shared/clipboard_helpers.dart` 只承载普通文本写入剪贴板和轻提示；各 feature 继续生成自己的脱敏摘要，Workflow 画布私有节点剪贴板协议继续归属 `workflow_page_system_clipboard_actions.dart`。

当前 Dart Runtime 代码组织遵循同样的低耦合边界：

- `studio_runtime.dart` 是 Runtime 公共入口，Flutter App 只依赖该入口。

- Runtime 模型按职责拆分为 settings、dependency、run history、run issue、run duration、run failure、run event、run trace、run analysis、run detail、execution state 和 execution internal 分片；`runtime_run_history_models.dart` 只承载运行历史基础摘要、单日聚合和总汇总，`runtime_run_issue_models.dart` 只承载问题分类和关联运行摘要，`runtime_run_duration_models.dart` 只承载节点耗时统计、趋势和关联运行摘要，`runtime_run_failure_models.dart` 只承载失败聚类和关联运行摘要，`runtime_run_event_models.dart` 只承载运行证据事件和视觉证据链，`runtime_run_trace_models.dart` 只承载节点路径和截图证据引用，`runtime_run_analysis_models.dart` 只承载失败分析、详情指标和问题分类，`runtime_run_detail_models.dart` 只承载完整运行详情聚合；`runtime_models.dart` 只保留模型入口说明，禁止重新堆叠所有模型。

- `runtime_controller.dart` 只保留 Runtime 依赖、快照和广播中心；本机环境、驱动和设备会话命令归属 `runtime_appium_commands.dart`；设备预览命令按截图、点击、手势和输入拆到 `runtime_device_preview_capture_commands.dart`、`runtime_device_preview_tap_commands.dart`、`runtime_device_preview_gesture_commands.dart` 和 `runtime_device_preview_input_commands.dart`，`runtime_device_preview_commands.dart` 只保留路由说明；设备预览共用的 connected + idle + session 校验、坐标校验、时长校验、pinch 坐标构造和 actions 释放归属 `runtime_device_preview_helpers.dart`；工作流启动、安全停止和暂停收口归属 `runtime_run_commands.dart`。
- 本地项目命令按业务边界拆分：`runtime_workflow_project_commands.dart` 承载当前 workflow 保存、复制和重置，`runtime_sub_workflow_project_commands.dart` 承载本地子流程注册、当前流程转子流程和删除，`runtime_settings_project_commands.dart` 承载本机设置、收藏和证据保留策略，`runtime_evidence_project_commands.dart` 承载运行历史、运行详情、本地报告和截图证据读取，`runtime_project_helpers.dart` 承载流程和子流程副本命名 helper。这些命令不直接调用设备、不启动 Appium、不执行 workflow。

- “连接设备”由 Dart Runtime 受控串行执行当前 USB 手机对齐、本机检查、必要时通过密码弹窗启动 Appium XCUITest 隧道、等待目标手机出现在 tunnel registry、准备 Appium 驱动和创建 WebDriver session。密码只写入 `sudo` stdin，不进入 Runtime event、snapshot、evidence 或日志；如果点击前快照判断本机隧道已就绪，但 Runtime 刷新后发现仍需要密码，Flutter 连接入口必须在同一次点击流程内补弹 Mac 密码框并继续连接，不要求用户再点一次；连接流程会使用 `xcrun devicectl` 只发现当前 USB 手机并过滤 localNetwork 设备，若唯一 USB 手机与配置不一致则自动写回本地项目配置并同步 Runtime session / tunnel 配置；若只发现无线可用手机而没有 USB 手机，连接流程必须停止并提示使用数据线连接，不能继续使用旧配置创建 session。创建 WebDriver session 前必须再走一次当前 USB 手机确认，覆盖一键连接内部重连、直接重连和未来命令中心入口，确保旧配置或无线设备不会被送入 `/session`。隧道进程存在但 registry 为空时必须先进入“等待手机允许”中间态，等待已有 registry 变可用，不继续创建 session；若等待超时且该隧道不是当前 Runtime 持有的受控进程，连接流程可在同一次密码授权内清理旧 tunnel-creation 进程并启动新的受控隧道。若 registry 或 session 显示 Appium 看不到目标手机，连接流程可在唯一 USB 手机明确时自动重绑并重跑一次；若绑定手机已确认但驱动仍不可见该设备，连接流程可重置当前端口旧驱动并重试一次：受控 Appium 正常停止，外部旧 Appium 只按当前 host/port 精准清理主服务，不匹配 tunnel-creation；最终失败时展示“驱动未识别手机”和“保持解锁，点连接设备。仍失败就重插线。”；若受控 Appium 创建会话时出现 `socket hang up`、`could not proxy command` 或 `port 8100` 等 WDA 代理瞬时断开，连接流程可自动停止受控驱动、重新准备并重试一次；Xcode 构建、证书信任和 USB 问题不得自动重启。无法唯一确认时，主界面只提示“未找到 USB 手机”和“用数据线连接一台手机并解锁，再点连接设备”。Device Controls 保留“重绑”短入口，作为手动兜底，不需要重启应用。
- `DeviceSessionManager` 是创建 `/session` 前的最后防线：如果 session 配置缺少有效当前手机绑定，或仍包含脱敏 / 示例设备占位符，必须直接抛出可诊断的设备绑定错误，不允许去掉 UDID 后继续请求 Appium，也不允许把占位符交给 Appium 再等待底层报错。
- Runtime 必须区分受控驱动和外部驱动：受控驱动是本应用通过 `AppiumProcessManager` 启动并持有进程句柄的 Appium，外部驱动是 `/status` 可用但本应用没有进程句柄的 Appium。外部驱动可用于连接设备；如果它在当前 host/port 上持续报“驱动未识别手机”，一键连接可按端口清理旧 Appium 主服务并切换为受控驱动。“停止驱动”仍不能声称已停止未接管的外部进程；UI 应禁用该动作或提示“外部驱动未停止”，并允许用户继续走“连接设备”。
- Local Setup Guide Advanced Drawer 由 Flutter UI 基于 `StudioRuntimeSnapshot.dependencyReport` 派生，只读展示 Appium、Xcode、USB Tools、本机隧道、WDA 和 iOS Trust 的本机准备指引；本机隧道处于 Warning / Error 时，Drawer 可提供复制启动命令动作作为高级备用。复制动作只写入剪贴板，不执行命令、不请求权限、不调用 Runtime；该 Drawer 不自动安装、不签名、不绕过 iOS 系统限制。
- Readiness Guide 只消费 `StudioRuntimeSnapshot`，不直接暴露完整设备标识、完整 session、WDA endpoint、原始 WebDriver payload 或底层构建日志。
- Appium / Xcode / WDA 的深层细节进入 Drawer、Console 或 Advanced 面板，主界面只保留 Ready / Warning / Error 级别摘要与下一步动作。
- Controls 仍调用 Dart Runtime 的 Appium / WebDriver 方法，不调用 Legacy Node API。

- Runtime 执行事件当前已收口为短中文动作文案：Tap、Wait、Swipe、Input 和 Snapshot 直接输出“第 N/M 轮：点击/等待/滑动/输入/截图 ...”，Flutter UI 只消费 Runtime event stream，不再把英文运行句作为主要展示真源。
- Runtime 已支持 Swipe、Input 和 Loop 的基础执行语义：Swipe 通过 Appium / WDA W3C pointer action 执行并释放 actions，Input 向当前焦点输入文本且证据默认只记录文本长度，Loop 是 bounded control-flow 节点，只按有限 `count` 在 body 和 after 两条边之间串行切换。
- Runtime 已支持 Sub Workflow 的基础执行语义：节点通过 `workflowId` 引用 Runtime 本地注册的 Project DSL 子流程，执行前先运行 `WorkflowValidator`，然后在同一 WebDriver session 和同一串行 runner 中执行子流程，完成后回到父流程后继节点。
- Runtime 已支持受限 Sub Workflow 参数传递：节点通过 `inputMap` 把父流程 `context.xxx` 映射为命名输入，子流程通过 `context.inputs.xxx` 读取；DSL validator 和 Runtime 都会拒绝脚本、函数调用或任意表达式，运行证据只记录参数名和数量。
- Runtime snapshot 已暴露只读 Sub Workflow summaries，Flutter Inspector 可展示本机已注册子流程并把所选 `workflowId` 写入节点草稿；摘要包含 workflowId、名称、节点数、合法性和直接引用的子流程 ID，不包含设备、session、路径或底层 payload；保存仍走 Runtime `updateWorkflow` 和 validator。
- Runtime 提供受控本地子流程注册、当前流程转子流程和删除方法，Inspector 可添加一个只包含 Wait 的安全示例子流程，可把当前 Project DSL 主流程存为本地子流程，也可删除未被当前流程或其它子流程引用的本地子流程；这些动作写入本地 Sub Workflow Store，项目启动时只恢复 validator 通过的子流程，不连接设备、不启动 Appium、不执行子流程。
- Runtime 保存 workflow 或注册子流程前必须校验 Sub Workflow 直接引用和嵌套引用是否都存在，并拒绝直接自引用和间接循环引用；缺失引用、嵌套缺失引用或循环引用不能进入当前 Runtime workflow，也不能写入 Sub Workflow Store。
- Runtime 启动运行前必须再次执行项目级 workflow 校验；即使 workflow 来自启动恢复、测试预览或外部构造，缺失 Sub Workflow 引用、嵌套缺失引用和循环引用也必须在设备动作、运行证据和执行主体开始前被拒绝。

## 6. Appium Process Management

V2.0 第一版由 Flutter / Dart Runtime 管理 Appium 进程和本机隧道进程。

职责：

- 提供“连接设备”主入口：本机检查、必要时启动隧道、准备驱动、创建会话
- 检测本机 Appium 是否可用
- 通过受控密码弹窗启动 Appium XCUITest 本机隧道
- 启动 Appium
- 启动后轮询 Appium `/status`，只有 ready 后才允许进入设备连接
- 健康检查 Appium
- 停止 Appium
- 记录 Appium 生命周期事件
- 将失败归类为用户可理解状态

Appium 状态判断必须区分：

- 进程已启动：本机 Appium 进程已经被拉起，但不代表服务可接收 session。
- 服务已 ready：Appium `/status` 返回 ready 后，才允许创建 WebDriver session。
- 手机会话创建不是普通健康检查，可能触发 WDA 构建、安装和启动；Dart Runtime 必须使用项目配置中的长请求超时等待 `/session`，不得沿用 `/status` 的短超时。
- Appium `/status` 健康检查仍保持短外层超时，避免错误端口或无响应服务拖慢“连接设备”。

真机发现边界：

- Dart Runtime 启动 Appium 进程时默认让 XCUITest driver 优先使用 `devicectl` 发现真机，避免新系统下旧 usbmux 发现列表为空。
- iOS 18 及以上真机配置必须携带平台版本信息，确保 XCUITest driver 进入 RemoteXPC 路径。
- RemoteXPC 隧道缺失时，连接诊断必须提示“本机隧道未就绪”，并引导用户复制 Appium XCUITest 隧道命令；不得继续提示 pymobiledevice3 隧道作为 WDA 会话解决方案。

若 Appium 在超时时间内未 ready，Runtime 应进入可操作错误态，并在 Bottom Console 记录最后一次状态摘要。

第一版不做：

- 自动安装 Appium
- 自动安装 Xcode 或系统依赖
- 绕过 Developer Mode、证书信任或 WDA 签名限制

后续版本应补：

- Appium 安装向导
- 依赖检测向导
- Xcode / WDA 准备检查
- 可修复项和不可修复项分组

## 7. Appium Client Boundary

V2.0 使用 Dart 封装 Appium / WebDriver Client。

当前只封装项目需要的能力：

- status
- create session
- delete session
- screenshot
- viewport size
- page source
- tap
- swipe
- input text
- release actions

原则：

- 不追求覆盖完整 Appium 客户端生态。
- 不暴露原始 WebDriver 细节给 UI。
- 不把 Appium 原始错误直接铺到主界面。
- Runtime 将 Appium 错误转换为连接态、运行态、挂起态或可操作错误。

当前已落地的最小协议面：

- Appium `/status`
- WebDriver `create session`
- WebDriver `delete session`
- WebDriver screenshot
- WebDriver window rect / viewport size
- WebDriver page source
- W3C pointer tap
- W3C pointer swipe
- W3C release actions
- WebDriver focused text input

当前 `appium_client` 内部按职责拆分为 endpoint 配置、统一异常、status/session/viewport 模型、JSON HTTP transport、W3C action payload 和最小 client 门面；`appium_client.dart` 只作为公共入口，不承载具体实现。

Tap 必须通过 Runtime 执行，Runtime 负责在 Tap 后尽力释放 actions。UI 不直接构造 WebDriver payload。

Screenshot 在 Phase 2 先作为设备预览状态进入 Runtime snapshot：

- Runtime 负责调用 WebDriver screenshot。
- UI 只读取 snapshot 中最近一次截图进行预览。
- 手动截图仅在设备已连接且运行空闲时触发。
- Phase 2 不长期保存截图；证据存储和滚动清理在 Evidence 阶段落地。

## 8. State Model

连接状态和运行状态必须分离。

连接状态：

- disconnected
- initializing
- connecting
- waitingForDeveloperTrust
- connected
- disconnecting
- error

运行状态：

- idle
- running
- paused
- stopping

约束：

- 只有 connected 且 idle 时可以启动任务。
- 运行中不能断开、重连或写 workflow。
- workflow 写入中不能启动任务。
- waitingForDeveloperTrust 必须有重试上限。
- 停止任务采用安全停止。
- 连接失败必须先进入 Runtime 诊断分类，再映射到 UI 状态和短中文文案；当前分类至少覆盖手机未信任证书、手机锁定、驱动未就绪、USB 设备不可用、WDA 构建失败、WDA 会话未启动和未知错误。
- Runtime 连接诊断必须由 `lastConnectionDiagnostic` 作为结构化快照字段对外暴露；Device、Execute、Top Status、Command Center 和 Bottom Console 只消费摘要、下一步和已脱敏详情，不得从原始异常文本自行推断。
- Runtime 诊断详情必须脱敏和裁剪，不得把完整设备标识、本机路径、本地端点、session 或原始 WebDriver payload 直接展示在 Device、Execute 或 Bottom Console 中。

## 10. Data Storage Boundary

第一版本地数据采用文件优先：

- Workflow：JSON
- Run History：JSONL
- Evidence：本地文件目录
- Settings：JSON
- Legacy sequence：只读迁移来源或兼容读取

长期可演进为：

- SQLite / Drift：历史运行、Monitor 聚合、筛选和搜索
- Indexed evidence catalog：证据索引
- Workspace profile：工作区级配置

隐私原则：

- 只保存在本机。
- 不上传云端。
- 不在默认 UI 和文档中暴露完整设备标识、账号、证书主体或敏感路径。
- 截图证据遵守滚动清理策略。

当前已落地的本地运行证据：

- Runtime 可在本机项目目录下写入运行记录。
- 每次运行包含 `metadata.json`、`events.jsonl`、`finished.json`。
- `events.jsonl` 记录 run start/end 和 step start/end。
- 本地证据存储有最大保留次数和最大保留天数；清理时先按最新证据时间删除过期运行目录，再按最大条数兜底删除旧运行。
- 证据写入失败不应中断正在执行的 Appium 自动化任务，只进入 warning 日志。
- Runtime 可读取本地运行记录并形成 Monitor 基础 KPI 和最近运行列表。
- Runtime 可从本地运行记录形成 7 / 30 / 90 日 daily runs 聚合，作为 Monitor 趋势图唯一数据来源。
- Runtime 可从本地运行记录形成平均耗时聚合；仅统计具备开始和结束时间且耗时有效的本地运行。
- Runtime 可读取单次运行详情，组合 metadata、events 和 finish summary 为 `RunDetail`。
- `RunDetail` 暴露节点事件时间线、节点耗时聚合、失败节点、暂停节点、问题原因、运行耗时和执行路径统计。
- `RunDetail` 可派生 Issue Analysis MVP，包括问题分类、问题节点、问题循环、问题节点耗时和关联截图证据数量。
- `RunHistorySummary` 可从最近本地 run detail 聚合问题分类、耗时节点、耗时趋势和失败聚类，展示平均耗时、峰值、样本数、问题数、问题节点、次数、影响流程、关联运行摘要、最近发生时间和 7 日节点耗时变化；问题分类、耗时节点、耗时趋势和失败聚类的关联运行摘要只保留 run id、流程名、状态、必要统计值和发生时间；该聚合不读取截图画面、不展示本机路径或原始 WebDriver payload。
- Snapshot 节点在 `saveEvidence` 开启时会把截图以本地文件形式写入当前 run evidence 目录。
- Snapshot 截图 evidence 在事件中只记录相对引用，不记录本机绝对路径。
- Monitor KPI 当前展示总数、成功率、均耗时、失败、暂停和已停，并区分 completed、failed、paused 和 stopped；`paused` 使用 warning 语义，表达等待人工介入，而不是系统失败。
- Monitor 当前已展示 7 / 30 / 90 日本地 Trend、失败趋势、常见问题、Status Distribution、Issue Categories、耗时节点和耗时趋势；趋势表达本地运行量与问题量，失败趋势表达 failed、paused 和 stopped 的本地日趋势，常见问题表达本地失败聚类和关联运行入口，状态分布表达 completed、failed、paused、stopped，问题类型聚合来自最近本地 run detail 的 Issue Analysis，并提供本地“看记录”入口；耗时节点和耗时趋势来自最近本地 run detail 的 node trace 聚合。
- 常见问题“看记录”会优先使用 Runtime 聚合的关联 run id 筛选 Recent Runs；旧摘要缺失关联列表时按问题分类做保守降级筛选；该交互只改变当前 Monitor 列表，不读取截图、不打开详情、不修改本地 evidence。
- 问题分类“看记录”优先使用 Runtime 聚合的关联 run id 筛选 Recent Runs；旧快照或手工预览数据缺少关联 run id 时按分类对应状态保守降级；该交互只改变当前 Monitor 列表，不读取详情、不读取截图、不修改本地 evidence。
- 耗时节点和耗时趋势“看记录”只使用 Runtime 聚合的关联 run id 筛选 Recent Runs；该交互只改变当前 Monitor 列表，不读取截图、不打开详情、不修改本地 evidence。耗时趋势进入关联记录后可展示本地跨时间深挖摘要 MVP，包括峰值日、峰值耗时、问题日、最近样本日和每日样本块；该摘要只消费 Runtime 已聚合的趋势点，不读取单次运行详情或截图。
- 关联运行深挖摘要已落地 MVP，展示影响流程、问题数、完成数、最近发生时间和关联流程名；跨运行对比摘要已落地 MVP，展示完成/失败/暂停/已停分布、最近连续问题、最近变化和脱敏状态时间线；两者都只消费当前关联运行列表，不读取运行详情、不读取截图、不写回 Runtime 或 evidence。
- Monitor Recent Runs 当前支持 All、Issues、Failed、Paused、Completed 本地视图筛选，并支持按 workflow 名称、状态和 run id 本地搜索；筛选和搜索由 Flutter UI 基于 Runtime snapshot 完成，不写回本地 evidence 或 run history。当前可见记录可复制为脱敏摘要，用于本地沟通和排查；复制只读取当前 UI 列表，不读取详情、不读取截图、不写回本地文件。
- Monitor 可从最近运行打开本地详情 Drawer，展示运行摘要、问题分析、Execution Path Summary、Evidence Filmstrip、节点路径、节点耗时和问题原因。
- Run Detail 可展示本地处理建议，建议来自问题类型、问题节点、截图证据数量、最慢节点和视觉证据摘要，帮助用户判断先看画面、补截图、查等待、重连手机或人工确认；该建议不读取截图画面、不触发设备动作、不进行 AI 自动修复。
- Run Detail 可复制本地脱敏诊断摘要，摘要只包含流程、状态、轮次、问题类型、问题节点、原因、路径计数、截图数量和视觉检查次数；不得包含截图画面、evidence 路径、设备标识、WDA endpoint、session 或原始 WebDriver payload。
- Run Detail Related Events 基于 `RunDetail.events` 只读展示本地运行事件，支持 All、Nodes、Issues、Screenshots 本地筛选；该视图不读取截图、不展示 evidence 相对路径、不写回 Runtime 或 evidence 文件；Sub Workflow 事件可显示“传参 N 项”和字段名摘要，但不得展示参数值。
- Run Detail 节点路径可在 All、Issues、Screenshots 三个本地视图之间切换；该筛选由 Flutter UI 基于 `RunDetail.nodeTraces` 完成，不写回 Runtime、不改变 evidence 文件。
- `RunDetail` 可从节点 trace 派生脱敏截图证据引用列表；Flutter 只在用户主动 Reveal 时通过 Runtime 读取本地相对 evidence 缩略图，不在主界面展示路径或默认渲染隐私画面。
- Monitor 详情会标记节点是否有关联截图 evidence；默认不展示私密截图画面，用户主动 Reveal 或开启本地 Reveal Screenshots By Default 偏好后才通过 Runtime 读取本地相对 evidence 引用并显示缩略图。
- Monitor Run Detail 已提供截图索引与回放 MVP：Evidence Filmstrip 负责列出截图证据索引，Reveal 后的回放区显示“第 N/M 张”以及上一张/下一张控制；切换回放只按需读取当前本地 evidence 缩略图，不展示相对路径或本机绝对路径；截图预览框由 Monitor 证据分片复用，不再绑定在时间轴文件中。
- Monitor 截图预览不展示坐标、session、原始 WebDriver payload 或本机绝对路径；底层原始 evidence 仍只保存在本机。

当前未落地：

- 更完整的跨运行关联深挖分析，例如跨问题根因对比、节点级跨运行路径差异和多次失败解释；当前已有本地深挖摘要和跨运行状态对比 MVP。
- 更完整的跨时间节点耗时深挖分析，例如根因对比、节点路径差异和多轮趋势解释；当前仅有趋势点摘要 MVP。

## 10.1 Legacy Config Import

V2.0 第一版允许 Flutter / Dart Runtime 直接读取旧 `sequence` 配置文件作为迁移输入。

读取内容包括：

- Appium host / port / path
- Appium executable override
- Appium server log level
- Appium / XCUITest capabilities
- Tap duration
- Legacy Tap / Wait sequence

边界：

- Flutter 只能读取配置文件，不调用旧 Node Web API。
- 旧 `sequence` 会在 Dart 内映射为 V2 Workflow DSL。
- capabilities 只进入 Runtime 和 Appium session，不在主界面展示完整敏感值。
- 设备标识必须在 Runtime 会话前通过当前唯一 USB 手机自动对齐；缺少有效设备绑定时不得创建 Appium session，脱敏或示例占位符不得进入 Appium session 请求。
- 启动 Appium 时优先使用配置声明的 executable；未声明时可发现项目本地 Appium 二进制，最后才回退到系统 PATH。
- Flutter Shell 创建 Runtime 时必须从当前目录、应用可执行文件目录、脚本目录、常见环境变量和开发期工作区兜底目录向上查找项目配置，避免打包或双击启动时误退回系统 PATH。
- 开发期 Debug / Profile Mac App 允许关闭 App Sandbox，以便本地工作站读取项目配置、写入本地证据并启动本机驱动进程；正式 Release 若保持沙盒，必须通过用户选择项目目录和安全授权恢复同等能力。
- 找不到项目配置时，Local Stack Check 必须提示“项目配置未找到”，不得误报为驱动工具不可用。
- 项目配置存在但被沙盒或文件权限拦截时，Local Stack Check 必须提示“应用无法读取项目配置”，不得继续引导用户反复安装驱动工具。
- Workflow 与 Settings 的长期真源分别是 Project DSL JSON 与 `StudioSettings` JSON。
- `fvm dart run tool/v2_boundary_check.dart` 会扫描 Flutter App 和 Dart packages 的产品源码，确保该边界不依赖人工记忆；`npm run smoke:v2-boundary` 仅是兼容 wrapper。

## 11. Design Boundary

V2.0 最终实现采用：

- Tech Noir Enterprise Workstation
- Dark
- Industrial
- Precision
- Professional
- Mission Control

- macOS 专业工具

用户可见文案规则：

- Mac App 默认使用中文界面。
- 主界面文案优先短词和短句。
- 能用 2 个字表达时不拉长到 4 个字；能用 4 个字表达时不拉长到 6 个字。
- 主界面避免直接展示 Appium、WDA、WebDriver、session 等专业词。
- 用户侧统一表达为“驱动”“手机会话”“连接”“流程”“记录”等。
- 专业细节只进入控制台、抽屉、高级面板或源码视图。
- 中文文案变更必须关注按钮、状态胶囊和窄宽度布局，不得撑开界面。

## 12. Change Rules

- 新的 V2.0 UI 能力必须落在 Flutter App。
- 新的 V2.0 Runtime 能力必须落在 Dart Runtime 或 Dart packages。
- 新的 V2.0 设备调用必须通过 Dart Appium / WebDriver Client。
- 如确需保留 Node 行为，必须标记为 Legacy / Debug / Migration，不得成为主路径依赖。
- 如果未来需要重新引入后台服务层，必须先创建新的架构决策文档并明确原因。
